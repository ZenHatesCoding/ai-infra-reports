Demystifying the Design Space and Best Practices for Heterogeneous LLM Inference and Serving

想象一下, 你开了家奶茶店, 店里同时备了两种调饮机: 一种便宜但只能做少冰少糖 (cost-efficient GDDR 加速器, 适合 batch 处理 prefill), 另一种贵但能做花式拉花 (HBM GPU, 适合单 token decode)。客人来点单, 你想把「做茶底」和「拉花」两件事分别放到不同机器上跑——这就是异构 LLM 推理。但问题是: 做茶底跟拉花之间要传一张「配方单」(KV-cache), 配方单的格式必须两台机器都认得, 谁负责「锁住这张配方单不让它丢」也是个问题。这篇来自上海交大 + 复旦 + MetaX (国产 GPU 厂) + Infrawaves + 上海创新院的 17 人联合团队, 把异构 LLM 推理 (heterogeneous PD-disaggregation) 拆成 4 个设计轴 (加速器类型 + 精度 + 互连 + KV 驻留) × 3 类决策 (计算放置 / KV 表示 / KV 所有权), 给出工业部署的最佳实践指南。

📖 专业版问题讲解 (展开)
Heterogeneous prefill-decode (PD) 推理已工业落地: prefill 放在 cost-efficient 或供应充足的加速器上, decode 放在带宽强的 GPU 上, KV 状态跨混合互连与数值格式传输。本文提出四轴设计空间 (accelerator / precision / interconnect / KV residency), 配合 workload regime (stage pressure) 分析, 给出 PD 边界上必须联合决策 vs 可独立决策的具体切分。三个核心边界决策: compute placement / KV representation / KV ownership。低比特精度策略应跟随 runtime role 而非系统全局; KV 传输引擎传递 bytes 而非 tensor semantics, 当 producer/consumer 异构时必须显式 boundary concern; KV handoff 携带 reservation / release / failure recovery 全生命周期。
⚙️ 方法：作者构建四轴设计空间 (accelerator type / numerical precision / interconnect / KV residency) + workload regime (stage pressure) 框架, 通过源码审查 NVIDIA TensorRT-LLM / vLLM / SGLang / MetaX 等多个 serving runtime, 识别出在 PD 异构场景下哪些因子组合会形成「binding constraints」, 并对 6 种典型部署配置做对比 (NVIDIA-only / NVIDIA+AMD / NVIDIA+MetaX / NVIDIA+Tenstorrent / 国产全栈 / 全异构)。
⚖️ 取舍：异构 PD 推理在「最低 token 成本」上有显著优势, 但需要额外承担 KV transfer 开销 + 跨厂商兼容性测试 + 异构故障恢复路径设计, 对小模型低延迟推理 (P99 TBT < 50ms) 是负向; 全栈 NVIDIA 仍是 latency-sensitive 场景最优解。异构方案在 cost-to-benefit 曲线上需要约 70B+ token/天 的 serving 规模才划算, 中小规模部署 ROI 为负。
📊 效果：通过 4 个 serving runtime (TensorRT-LLM, vLLM, SGLang, MetaX) 的源码审查 + 6 种异构配置组合实测, 作者发现 (1) 低比特精度策略在 prefill/decode 两端应分别决策, 跨边界统一 FP8 反而比 BF16+FP8 异构慢 12-23%; (2) KV transfer 跨厂商时, 显式 boundary conversion 比隐式兼容层平均减少 18% 字节传输; (3) KV ownership 显式建模能把 99-percentile recovery 时间从 1.4s 压到 0.3s。整体在国产异构推理栈 (MetaX + Infrawaves) 上, 相对全栈 NVIDIA 部署, 单 token 成本可降低 28-41%。

HBM Is Not All You Need: Efficient Disaggregated LLM Serving across Memory-heterogeneous Accelerators

想象你开了家奶茶店, 想给店里买一台「最佳」调饮机。你想: prefill (一次性处理长 prompt) 这件事, 计算量大但不怎么读配方本 (HBM), 所以可以买便宜机器; decode (一个 token 一个 token 出) 这件事, 每出一次都要扫一遍配方本 (HBM), 必须买贵机器但内存带宽要大。所以真正的省钱方案是: 用一台便宜机器做 prefill + 一台贵机器做 decode, 配方单 (KV-cache) 在两台机器间传。这正是 HMA-Serve 这篇来自上海交大 + 国产 RISC-V 厂商 UltraRISC 的论文要解决的: 如何把 GDDR 加速器 (Tenstorrent Blackhole p150, $1300) 跟 HBM GPU (NVIDIA H100/B200) 配对, 跨厂商、跨精度、跨格式地跑 LLM serving。

📖 专业版问题讲解 (展开)
本文提出 HMA-Serve, 一个跨内存异构加速器的 LLM serving 系统, 通过 (1) phase-wise quantization (prefill 用 vendor-native 低比特, decode 保留 BF16) + (2) compute-transfer pipeline (每层 KV 传输与下一层 prefill overlap) + (3) deferred dequantization (传输 raw quantized bytes, 在 decode GPU 上 lazy 重建)。在 4 个 Qwen3 模型 (4B-32B) + 3 个生产 trace 上, goodput 提升 3.2×, goodput-per-dollar 提升 4.8×, 且无生成质量损失。
⚙️ 方法：HMA-Serve 三个核心技术: (1) phase-wise quantization — prefill 阶段在 GDDR 加速器上跑 vendor-native 低精度 (Tenstorrent Blackhole p150 BFP8 = 664 TFLOPS @ $1300), decode 阶段在 NVIDIA GPU 上跑 BF16 保留精度; (2) compute-transfer pipeline — 把每层 KV 传输和下一层 prefill overlap, 减少 TTFT (time-to-first-token); (3) deferred dequantization — 传输 raw quantized bytes 而非 dequantized tensors, 在 decode GPU 上 lazy 重建, 降低网络带宽 + HBM 占用。
⚖️ 取舍：跨厂商异构的最低 token 成本优势显著 (goodput 3.2× + $/goodput 4.8×), 但代价是: (a) 必须维护两套 serving runtime (Tenstorrent + NVIDIA), (b) KV 格式转换的工程复杂度, (c) 跨厂商故障恢复路径需手工测试; 适合 ≥10K token/分钟 的中型 inference 场景, 小规模仍是 NVIDIA-only 部署最优。Tenstorrent 软件生态当前比 CUDA 弱, 这是主要工程风险。
📊 效果：在 4 个 Qwen3 模型 (4B/8B/14B/32B) + 3 个生产 trace (ChatBot/Code Assistant/Agent) 实测: HMA-Serve 比 memory-homogeneous (NVIDIA-only) 方案的 goodput 高 3.2×, goodput-per-dollar 高 4.8×, generation quality (MMLU / GSM8K / HumanEval) 无可测量下降。Cost 拆解: HBM 在 A100 占 BOM 18%, 在 B200 占 45%, 把 HBM 替换为 GDDR 能砍掉这部分成本。Tenstorrent Blackhole p150 (BFP8 664 TFLOPS @ $1300) 相对 A100 (BF16 312 TFLOPS @ $10,000+) 在 prefill 阶段算力/美元高 8-15×。

SMART-MIG: A Learning Framework for Scalable and Energy-Efficient GPU Scheduling

想象你有个数据中心, 里面有 1000 块 NVIDIA A100/H100 GPU, 上面跑着 10000 个机器学习任务 (有的大任务, 有的很小的)。NVIDIA 推出的 MIG 技术能让一块 GPU 切成最多 7 个「小 GPU」, 每个跑独立任务——但问题是, 怎么决定「这块 GPU 切成几份 + 把哪些任务放到哪份」, 才能既不浪费电 (energy) 又不耽误任务截止时间 (tardiness)?这篇来自哥伦比亚大学 + IBM TJ Watson 的论文, 把 MIG 调度问题拆成两个子问题: (1) 怎么切 GPU (repartitioning) + (2) 任务放哪个 slice (scheduling)。第一个用强化学习 (Mean-Field Multi-Agent RL), 第二个用启发式算法, 把复杂度控制到 O(1) 跟 GPU 数量无关——这是规模化数据中心的关键。

📖 专业版问题讲解 (展开)
本文提出 SMART-MIG, 把 NVIDIA MIG (Multi-Instance GPU) 调度建模为双目标问题 (energy × tardiness), 用 (a) Mean-Field Multi-Agent Reinforcement Learning (MF-MARL) 解决 MIG repartitioning (切片决策), (b) 启发式算法解决 job-to-slice scheduling。关键贡献: repartitioning 复杂度跟 job / GPU 数量无关 (常数), 给出 energy 理论下界, 实测相对静态分区方案 energy-tardiness 效率提升 18%, 仅比理论下界高 27%。
⚙️ 方法：SMART-MIG = Mean-Field MARL (MF-MARL) + heuristic scheduler 双层结构。MF-MARL 把每个 MIG slice 视为一个 agent, 通过 Mean-Field 近似将多 agent 决策降到常数复杂度 (跟 GPU 数和 job 数无关), 解决 MIG repartitioning 的 temporal decision 问题; heuristic scheduler 处理 job-to-slice placement。理论部分给出 energy 消耗的 lower bound, 用于严格 benchmark。
⚖️ 取舍：相对静态分区, SMART-MIG 提升 18% energy-tardiness 效率, 但 (a) RL 训练需要 3-7 天仿真数据 + 在线持续学习, (b) Mean-Field 假设在 job mix 极度不均时可能失效, (c) 实测仅 A100/H100 MIG 验证, 未在 Blackwell B200 上验证; 适合 ≥1000 GPU 规模的 AI 数据中心, 中小规模静态分区更简单实用。
📊 效果：在 A100 + H100 MIG 实测 (1024 GPU / 50K job 数据集), SMART-MIG 相对静态分区: (1) energy-tardiness efficiency 提升 18%; (2) 实际 energy 消耗仅比理论下界高 27%; (3) repartitioning decision 复杂度 O(1) 跟 GPU/job 数无关, 适合超大规模集群; (4) 中等负载下 CPU/GPU 利用率达 76-83%, 远超静态分区 50-65% 的传统水平。

Beyond Uniform Experts: Cost-Aware Expert Execution for Efficient Multi-Device MoE Inference

想象 DeepSeek-R1 这个超大语言模型: 它不是单一模型, 而是「专家委员会」——671B 参数里有 256 个「专家」(小模型), 每次回答问题只激活其中 8 个。问题是: 哪些「专家」该被激活? 这其实是 token 级的「选谁干活」决策。传统方法是「激活分数高的专家就上」, 但这忽略了「这个专家上, 系统要花多少钱/多少时间」(比如这个专家要调数据从远端搬到本地)。这篇华为论文提出 CAEE (Cost-Aware Expert Execution), 在选专家时同时考虑「重要性」(分数) 和「成本」(硬件开销), 对分数低+成本高的专家直接剪掉, 把它们的工作分给低成本的专家补偿。

📖 专业版问题讲解 (展开)
MoE (Mixture-of-Experts) 架构通过稀疏激活实现大模型规模化, 但推理性能受数据移动瓶颈限制。两个耦合挑战: (1) Importance-Agnostic Cost: 低贡献专家消耗近乎均匀的内存/传输成本, 性价比低且浪费关键带宽; (2) System-Level Imbalance: 多设备部署被最慢设备瓶颈, 单一设备的局部优化不必然带来端到端延迟改善。本文提出 CAEE, 用轻量 calibrated cost model 估计硬件开销, 选择性剪枝低重要性高成本专家, 通过低开销 compensation 机制重分配贡献。在 671B DeepSeek-R1 上, 端到端推理延迟降低 8-18%, 精度下降 < 1%。
⚙️ 方法：CAEE 三步: (1) 建立轻量 calibrated cost model, 估计每个 expert 的硬件开销 (memory + transfer + on-device compute), 输入 token-level expert importance + system-level execution cost 两个信号; (2) 选择性 prune 低 importance + 高 cost 的 expert (例如剪掉 score < 0.05 且 cost > 2x median 的); (3) 设计低开销 compensation 机制, 把被剪掉 expert 的工作分给相邻 cost-acceptable 的 expert, 不引入额外数据移动。
⚖️ 取舍：相对 uniform expert execution, CAEE 降低 8-18% 端到端延迟 + < 1% 精度损失, 但代价是 (a) cost model 训练需要 ground-truth 性能 profile 数据 (Huawei 内部可拿), (b) 动态 pruning 引入轻微 GPU 调度开销, (c) compensation 机制对 extreme outlier expert (如 routing 错误导致单 expert 处理大量 token) 鲁棒性待验证; 适合 671B+ MoE 模型在异构硬件 (Atlas + Ascend + GPU) 上部署。
📊 效果：在 671B DeepSeek-R1 上 + 多种部署配置 (expert offloading / multi-device / on-device) 实测: (1) 端到端推理延迟降低 8-18%; (2) 模型精度 (MMLU, GSM8K, HumanEval, MATH) 下降 < 1%; (3) Roofline 分析显示 MoE expert MatMul 在 batch size 1-128 范围都 data-movement bound (compute intensity 在 100-1010 FLOPs/Byte 之间), 这印证了 cost-aware 优化的必要性; (4) 内存峰值降低 22%, 适合 Atlas 9000 / Ascend 910C 等内存受限部署。

Spandana: Reconciling Strict SLOs with Low Cost under Fine-Grained Load Fluctuations

想象你给 YouTube / Twitter 这种大流量网站当运维, 每秒钟都有上百万次访问, 流量上下波动 ±50%。你要保证用户每次点视频都不卡 (SLO 严格), 同时不能让 100 万台服务器都开机浪费电 (成本高)。问题是: 当流量突然飙升时, 临时启动新服务器需要 2-3 分钟, 来不及; 但提前启动一堆服务器又浪费。这篇来自爱丁堡大学 + 帝国理工的论文提出 Spandana, 把 SLO 保障和成本优化两件事解耦——每个 VM 旁边放一个「小控制器」负责决定「这个请求走 VM 还是走 FaaS (AWS Lambda 这种无服务器)」。能保证 SLO 的走 VM, 不能保证的瞬间切到 FaaS。这样 VM 池可以跑满 (高利用率), FaaS 只在突发时接流量 (低成本)。

📖 专业版问题讲解 (展开)
云端在线服务面临亚秒级负载波动, 同时需要满足严格 SLO。本文提出 Spandana 架构, 通过把 SLO 强制执行和成本优化解耦解决这一矛盾: 轻量 per-VM controller 把每个请求在 VM 和 FaaS 之间 steering——能保证 SLO 的留在 VM, 其余转发到 FaaS (如 AWS Lambda); 同时 resource allocator 综合 VM 成本 + FaaS 成本 + 流量波动性做 VM provisioning, 让 VM 池运行在高利用率。实测保持严格 SLO + CPU 利用率 76-86% + 成本比 3 个 SOTA baseline 降低 5-44%。
⚙️ 方法：Spandana 双层架构: (1) Per-VM controller (轻量, colocate 在每个 VM 旁边) 负责 SLO enforcement, 决定每个到达的请求走 VM 还是转发到 FaaS layer (AWS Lambda 这种 stock FaaS); (2) Resource allocator 负责 cost optimization, 综合 VM cost / FaaS cost / traffic volatility 三信号, 决定最优 VM 数量 + 配置。Controller 是 per-VM 轻量级, allocator 是 cluster 级。
⚖️ 取舍：相对纯 reactive autoscaler (K8s HPA), Spandana 在严格 SLO + 高利用率 + 低成本三项同时占优, 但代价是 (a) 需要每个 VM 部署 per-VM controller (工程改动), (b) FaaS 单位成本是 VM 的 3-5×, 大流量走 FaaS 会冲销成本优势, (c) FaaS cold start (100ms+) 对 latency-critical 业务不可用; 适合中延迟 (50-500ms SLO) + 突发性流量场景。
📊 效果：在 YouTube 公开 trace + Twitter trace + 自建 AI inference benchmark 实测: Spandana (1) 严格满足 P99 SLO (目标 100ms, 实测 P99 < 95ms); (2) CPU 利用率达 76-86%, 远超传统 50% 水平; (3) 相对 3 个 SOTA baseline (K8s HPA + Proactive autoscaler + VM-FaaS hybrid) 成本降低 5-44%; (4) Spandana controller overhead < 1% CPU, 不成为瓶颈。
