Less Experts, Faster Decoding: Cost-Aware Speculative Decoding for Mixture-of-Experts

奶茶店有 30 个调饮师 (MoE 专家), 每次客人下单系统只叫其中 2-3 个, 推理速度才快. 问题是现在流行的「推测解码」挑草稿 token 时只看哪个调饮师被叫概率高, 结果挑出来的 token 分散在 30 个调饮师里, 调饮师换来换去浪费时间. 本文 ECOSPEC 提出「成本感知」选 token: 先用一个轻量预测器估算每个 token 会激活哪些调饮师, 然后优先挑那些「调饮师已被叫过」的 token, 让一次验证里调饮师尽量不换. 在 DeepSeek-V3.1 (671B) + Qwen3-235B-A22B + GPT-OSS-120B 三大 MoE 模型上, 推理加速最高 1.62 倍.

📖 专业版问题讲解 (展开)
本文 ECOSPEC 框架瞄准的是「大规模稀疏 MoE 推理里专家被频繁换来换去」这个老大难. MoE (混合专家) 架构本身靠稀疏激活来扩展模型容量, 但「推测解码」(一种先用小草稿模型猜多个 token 再由大模型统一验证的加速技术) 在选草稿时只看置信度, 结果挑出来的 token 会路由到一批互不相干的专家上, 把显存 (HBM, 高带宽内存) 流量搞炸. ECOSPEC 的关键改动是在草稿评分函数里额外加入「这条 token 会激活哪些专家、这次激活要付多大显存代价」的成本项, 同时维护一个「当前验证步已激活专家」的缓存, 让选出的 token 优先复用已激活专家. 论文在 DeepSeek-V3.1 这种 671B 级别 MoE 上做到 1.62 倍端到端加速, 显存流量同步下降.
⚙️ 方法：第一步先离线训练一个轻量「专家预测器」 (小模型), 输入是 token 特征 + 当前上下文, 输出是「这个 token 大概率激活哪几个 MoE 专家」的概率分布. 第二步在「推测解码」的草稿树搜索阶段, 把原本只看置信度的评分函数改为「置信度乘专家复用率」, 草稿候选如果能复用当前验证步已经在 HBM 里的专家权重, 就优先选中. 第三步维护一个动态「专家缓冲」 (动态 expert buffer), 记录当前 verification 步激活过的专家集合, 缓冲随验证推进而更新. 整个框架不改目标模型的验证规则, 直接接到现有 vLLM 或 SGLang 或 TensorRT-LLM 推理栈上. 测评在 DeepSeek-V3.1 (671B) + Qwen3-235B-A22B + GPT-OSS-120B 三个 MoE 上跑, 评测维度包括推理 (MATH + HumanEval) + 编码 (MBPP) + 问答 (Natural Questions) + 对话 (MT-Bench).
⚖️ 取舍：优势方面, ECOSPEC 不改目标模型的验证规则, 可以无缝接到现成的 vLLM + SGLang + TensorRT-LLM 推理栈上, 工业落地门槛低; 对 671B 级别的大规模 MoE 模型加速效果最明显. 劣势方面, 轻量专家预测器需要额外训练成本, 单 token 推理延迟略有增加 (毫秒级); 动态专家缓冲会占一部分显存, 在边缘端极致显存受限场景不适用; 当 MoE 路由器本身就只用 top-2 激活时, 专家散射空间本身很小, ECOSPEC 收益有限.
📊 效果：ECOSPEC 在三个大规模 MoE 模型上一致拿到端到端推理加速, 最高 1.62 倍. 「活跃专家足迹」(active expert footprint, 即每次验证步实际从 HBM 加载的专家权重量) 同步降低, 显存带宽压力变小. 草稿接受率比纯置信度基线只损失不到 2%, 但每次验证步里被切来切去的专家数量大幅减少. 在 dense Transformer 上常规 SD (推测解码) 的收益相对有限, 因为 dense 模型权重复用天然充分; 在 MoE 上 ECOSPEC 的收益最显著. 整体 serving 成本下降, 单 token 平均显存读取次数减少.

EMO: Energy Efficiency Modeling and Optimization for AI Workloads

工厂车间有 10 台机器同时工作, 关键机器满速跑, 非关键机器 (搬运 + 包装 + 质检) 其实可以慢一点省电, 不影响整体交付. EMO 发现 GPU 加速 AI 训练也有这种「忙闲不均」: CPU/GPU 协同 + 多 GPU 通信等待 + 多 stream 并发, 都产生 slack, 非关键 kernel 可以降频跑. EMO 给出一个轻量建模框架, 自动找到哪些 kernel 可以慢一点, 不用高保真建模 (那种方法 overhead 太大).

📖 专业版问题讲解 (展开)
EMO (Energy Modeling and Optimization) 这篇工作, 抓住 GPU 加速 AI 工作负载里一个被忽略的事实: 训练或推理过程中, 关键的 GPU 计算 kernel 满速跑的时候, CPU-GPU 协同 + 多 GPU 之间的通信握手 + 多个 CUDA stream (CUDA 流, GPU 上多任务并行通道) 的并发等待, 都产生了「空闲时间片」(slack, 即本可以降频但没人管的时段). 在这些 slack 时段, 非关键 kernel 本可以降频 (DVFS, 动态电压频率调节) 跑, 既省电又对端到端延迟零影响. 现有方法要么只能用静态调频策略, 要么需要高保真建模 (per-kernel profiler) 才能找到 slack, 但开销大到不可用. EMO 提供一个轻量级方案, 训练一个小模型直接预测 slack 在哪 + 该降多少频, 在多种 AI workload 上省电 15-30%.
⚙️ 方法：第一步在 GPU kernel 真实执行时收集细粒度 profile 数据, 包括每个 kernel 的实际耗时 + 上下游依赖关系 + slack 窗口大小. 第二步训练一个轻量 slack 预测模型, 模型输入是 kernel 类型 + 上下游依赖 + 当前 GPU 频率, 输出是这个 kernel 还能再降多少频而不影响端到端延迟. 第三步 DVFS 控制器根据预测结果给非关键 kernel 动态降频, 关键 kernel 保持高频. 第四步在多 stream 并发 + 多 GPU 通信握手等待期间, 主动把等待方 kernel 降频, 把功耗省下来. 第五步用 pre-flight calibration (起飞前校准) 保证端到端延迟 SLA (服务等级协议) 不破. 整套框架部署在标准 PyTorch + CUDA 栈上, 不需要改模型代码.
⚖️ 取舍：优势方面, EMO 的轻量级 profiling 开销不到 1%, 容易集成到生产推理服务里; 不需要改模型代码, 跨 PyTorch + JAX + TensorFlow 多个框架都适用; 对延迟敏感的 workload 友好 (slack 预测精准, 不破延迟 SLA). 劣势方面, slack 预测模型需要 per-workload 校准 (每个 workload 单独跑一次训练), 跨 workload 迁移成本中等; 极短 kernel (小于 100 微秒) 的 slack 窗口太小, EMO 收益有限; 对 GPU 频率切换延迟敏感的平台 (某些移动端 GPU 频率切换慢) 效果打折.
📊 效果：EMO 在多类 AI workload 上验证: CNN 训练 + Transformer 训练 + 大模型推理 + 视觉模型推理, GPU 能耗降 15-30% (具体数字随 workload 变化), 端到端训练或推理延迟保持不变 (SLA 守住). 比静态调频 baseline 显著省电, 比高保真建模方法 profiling 开销低 10 倍. 验证平台覆盖 NVIDIA H100 + A100, AMD MI300X 上也能跑 (走 Radeon 软件栈). 对 latency 敏感的在线推理 workload 友好.

Realizable N:M Sparse Transformer Inference via Search-Kernel Co-Design

Vision Transformer (ViT) 推理慢, 因为全是矩阵乘法. 2:4 这种 N:M 稀疏化可以砍掉一半算力, 但实际跑起来速度并没快一半, 因为 GPU 调度在稀疏情况下会乱掉 (execution regularity 差). 本文给 ViT 做「软件找稀疏模式 + 硬件写配套 CUDA kernel」协同设计: 软件按层找最优 N:M 稀疏模式 (按 latency 预算搜索), 硬件写专用 CUDA kernel MD-SpMM 把稀疏矩阵乘法转成「微稠密 Tensor Core 对齐」数据流. 在 ViT/Swin 上拿到 2.2 倍延迟加速.

📖 专业版问题讲解 (展开)
Vision Transformer (ViT, 视觉 Transformer) 推理时核心算子是矩阵乘法, 理论上 2:4 这种 N:M 半结构化稀疏 (每 4 个连续权重里裁掉 2 个) 可以砍一半算力, 但实际 GPU 跑起来速度并没快一倍. 根本原因是稀疏数据流让 GPU 调度器「看不懂」 (执行规律性差), Tensor Core 利用率掉. 本文给 ViT 做软硬协同设计: 软件层按端到端延迟预算逐层搜索最优 N:M 稀疏配置, 硬件层写专用 CUDA kernel (作者命名 MD-SpMM) 把稀疏矩阵乘法重新组织成「微稠密 + Tensor Core 对齐」的数据流. 在 ViT 和 Swin Transformer 上拿到 2.2 倍延迟加速, 精度不降.
⚙️ 方法：硬件层设计 MD-SpMM, 一个 N:M 稀疏矩阵乘法的 CUDA kernel. 关键是把稀疏 GEMM 重新组织为微稠密 Tensor Core 对齐的数据流, 同时用推理感知的自适应并行度维持 GPU 流处理器 (SM) 利用率. 软件层做按延迟预算的逐层稀疏率搜索, 用三阶段启发式 (coarse-to-fine, 从粗到细) + 约束松弛 (constraint relaxation) 避免早熟收敛, 实现部署感知的稀疏配置. 搜索目标函数是端到端推理延迟, 不是单纯的浮点运算次数. 评测覆盖 ViT 和 Swin 全模型 + A100 + H100 GPU 平台, 对照 dense baseline + 静态 2:4 稀疏 + 单纯搜索稀疏三种.
⚖️ 取舍：优势方面, 2.2 倍延迟加速加上精度可比, 是 ViT 和 Swin 实际部署的重要提升; 软硬协同解决了 2:4 稀疏「理论节省不等于实际加速」的老大难; 开源代码可直接用. 劣势方面, 搜索阶段耗时较长 (单个模型数小时 GPU 时), 不适合动态模型场景; 当前只支持 ViT + Swin 架构, 扩展到大语言模型需要重新设计 MD-SpMM kernel (因为 LLM 注意力矩阵稀疏模式不同); N:M 稀疏率固定, 不支持动态调整.
📊 效果：MD-SpMM 加上搜索稀疏, 在 ViT 和 Swin 上拿到端到端延迟 2.2 倍加速 (对比 dense baseline), 模型精度可比. 同延迟预算下, 精度比单纯稀疏 baseline 显著优. CUDA kernel 在 A100 + H100 上稳定 80% 以上 Tensor Core 利用率, 解决了 2:4 稀疏执行不规律的痛点. 源代码公开 (GitHub liuganhuo/realizable-nm-sparse-transformer), 可以直接接到 PyTorch + TensorRT 推理栈.

FlashDiff: Efficient Regional Execution and Scheduling for Diffusion Model Serving

扩散模型 (画图 + 视频 + 音频) 推理时, 每一轮去噪都要更新整张图的每个区域. 问题是背景 (天空 + 平滑区) 其实早早就画好了, 反复更新是浪费; 而且不同区域去噪速度不一样, 整体要等最慢那块. FlashDiff 抓两个洞见: 不同 latent 区域稳定速度不同; 相邻去噪步 latent 强相关. 它把 latent 拆成语义一致的「patch 块」, 只更新仍需细化的区域, 并把省下来的算力重新分配给同一 batch 其他请求. 实测端到端 latency 降 30-97%, throughput 涨 1.2-2.2 倍.

📖 专业版问题讲解 (展开)
FlashDiff 这篇面向的是扩散模型 (Stable Diffusion + Sora 类视频 + AudioLDM 音频) 在线推理服务, 解决「全区域反复去噪」的浪费问题. 关键观察有两个: 第一, 同一张图里不同 latent (潜在表示, 模型内部的图像压缩表征) 区域去噪速度天然不一致 (天空几轮就稳定了, 细节物体要几十轮); 第二, 相邻去噪步的 latent 状态高度相关. FlashDiff 利用这两个性质, 把 latent 切成语义一致的 patch 块, 只更新还没稳定的区域, 把省下来的算力再分配给同 batch 其他请求. 系统三层: 早期注意力信号做语义 patch 切分; 轻量运行时控制器估计区域活跃度并跳过低价值更新; affinity-aware 在线调度器把依赖区域共置 GPU 并复用回收算力.
⚙️ 方法：第一步做 latent 分解, 用早期去噪步的 self-attention (自注意力) 信号做语义聚类, 把整张 latent 切成 coherent execution regions (图像用空间 patch, 音频用时间 segment), 既保留语义结构又暴露细粒度并行. 第二步做运行时区域活跃度预测, 训练一个轻量预测器, 跳过那些进一步细化也不影响输出质量的低价值区域, 评估指标是步级输出质量损失. 第三步做 affinity-aware 在线调度, 把相互依赖的区域 (同一生成目标或相邻去噪步) 共置在同一 GPU 上减少通信, 同时平衡残留负载, 把回收的算力复用给同 batch 其他请求. 三机制联动形成「语义 patch 并行」 (semantic patch parallelism), 不依赖传统的 sequence parallelism 或 tensor parallelism, 因此没有它们的高通信开销.
⚖️ 取舍：优势方面, 端到端延迟显著降低 (30-97%), 不改模型架构, 是 serving 侧优化, 部署门槛低; 吞吐量涨 1.2-2.2 倍直接降低单图像 GPU 成本; 对 production 级别的扩散推理服务实际意义大. 劣势方面, 区域切分需要早期注意力信号, 增加少量 overhead; 活跃度预测器需要 per-model 训练; 不改模型架构意味着不能彻底解决 compute-bound (算力受限) 瓶颈, 极致超低延迟场景需要配合蒸馏一起用.
📊 效果：FlashDiff 在 Stable Diffusion 图像 + Sora 类视频 + AudioLDM 音频三个 workload 上验证: 端到端推理 latency 降 30-97%, 吞吐量涨 1.2-2.2 倍, 同时消除了 24-66% 不必要的计算, 输出质量不降. 跟 sequence parallelism + tensor parallelism 基线比, 通信开销大幅降低, 算力利用率提升. 对 Adobe Firefly 这种日均百万级图像生成的 production 场景收益最大. 比 FlashAttention 这种 transformer 层面的优化更直接解决扩散推理瓶颈.

HeteroMosaic: Exposing and Exploiting Heterogeneous Execution Opportunities for Energy-Efficient Edge LLM Inference

现在手机 / 笔记本芯片是「CPU + 集显 + NPU」三种加速器一起干活的, 但现有 LLM 推理框架只会粗粒度选一种 (要么全 GPU 要么全 NPU), 浪费了一半算力. HeteroMosaic 把 LLM 推理当成「跨加速器调度问题」: 用异构 roofline 模型判断哪种 workload 跑在哪种加速器最划算, 然后把 LLM 拆成 dependency-preserving micro-batch, 让 CPU + iGPU + NPU 三种同时跑. 实测 AMD Ryzen AI 平台上, 推理速度比 llama.cpp 高 2 倍, 省电 45%.

📖 专业版问题讲解 (展开)
HeteroMosaic 瞄准的是「边缘 SoC 异构资源被浪费」这个长期痛点. 现代手机和笔记本芯片里同时有 CPU + 集成显卡 (iGPU) + 神经网络处理器 (NPU) 三种加速器, 但现有 LLM 推理框架只会粗粒度选一种后端 (要么全 GPU 要么全 NPU), 等于把另外一半算力浪费掉. 本文把边缘 LLM 推理重新定义成「跨加速器调度问题」, 提出 HeteroMosaic 框架: 用异构 roofline 模型 (一个分析不同硬件上限的工具, 类比传统 roofline 用于预测性能上限) 判断哪种 workload 适合放哪种加速器, 再把 LLM 任务图拆成 dependency-preserving micro-batch (保留依赖关系的微批次), 让三种加速器同时跑. 在 AMD Ryzen AI 平台上, 推理速度比 llama.cpp 高 2 倍, 省电 45%, 论文被 MICRO 2026 接收.
⚙️ 方法：第一步建异构 roofline 模型, 刻画不同 iGPU 与 NPU 相对能力系统的潜在异构收益, 帮系统决定哪种 workload 适合放哪种加速器. 第二步做 LLM 任务图分解, 把 LLM 执行切成 dependency-preserving micro-batches, 暴露跨加速器重叠的机会. 第三步做 trace-guided (基于真实执行 trace 的) 联合优化, 联合调度加设备分配一起优化, 同时考虑 device variation (设备差异) + 统一内存争用 + DVFS + NPU runtime 行为. 整套系统用 PyTorch C++ 实现, 跨 AMD Ryzen AI 多种平台 (NPU 重型 + balanced + iGPU 重型) 验证, 对照 llama.cpp + HeteroInfer 等基线.
⚖️ 取舍：优势方面, 异构优先调度填补了边缘 LLM 推理的核心空白, AMD 加 UIUC 工业学术合作背书; 省电 45% 加加速 2 倍双重收益, 直接降低边缘 AI 部署成本; 跨多种 SoC 平台适用. 劣势方面, 当前只验证 AMD Ryzen AI 平台, Apple Core ML 或 Qualcomm QNN 移植需要重写 runtime 部分; roofline 模型构建需要 per-SoC 校准, 工程成本中等; 对极致低功耗场景 (亚瓦级) NPU 算力可能不足, 异构收益边际化.
