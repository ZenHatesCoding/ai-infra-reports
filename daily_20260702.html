KernelFlume: Elastic Core-Attention Scaling for Agentic Long-Context Decoding

想象一下: 你家开了 100 个外卖取餐台, 每个取餐台都跑一个 AI 助手。AI 助手工作时需要把「之前说过的话」记在小本本 (KV cache, 也就是对话上下文缓存) 上, 但每个小本本都越写越长, 写多了就把桌子占满, 没法接新单了。KernelFlume 干的事情就是: 让 100 个取餐台之间可以互相「借」小本本——取餐台 A 写满了, 它的小本本可以临时存到取餐台 B 那里, 等要用了再搬回来, 类似水库大坝之间互相放水。团队用 NVIDIA A6000 48GB 显卡做实验, 在 64K 长上下文场景下, 跟 SGLang v0.3 比, 吞吐 (tokens/sec) 提升 1.7×-2.3×, 显存峰值下降 35%-50%, 长尾延迟 (P99) 下降 28%。

📖 专业版问题讲解 (展开)
KernelFlume 提出一种基于「细粒度核心注意力 (fine-grained core attention)」的弹性 KV cache 共享机制, 用于 agentic long-context 解码场景。系统将传统 decoder 的 attention 拆分为「核心层 (core layer, 高频访问)」和「外围层 (peripheral layer, 低频访问)」, 并通过 PCIe + NVLink 混合互联实现跨 GPU 的核心层 KV cache 共享。实验在 64K-1M token 上下文窗口下, 长文本 agent 任务 (SWE-bench、Multi-SWE-bench) 相对 SGLang v0.3 / vLLM v0.6 / TensorRT-LLM 0.11 实现 1.7-2.3× 吞吐提升, 显存峰值降低 35-50%, P99 尾延迟降低 28%。
⚙️ 方法：KernelFlume 的核心是「核心-外围注意力分层 (Core-Peripheral Attention Decoupling)」, 把传统 multi-head attention 切成高频核心 (Q 投影的 top-50% 头) 和低频外围 (剩余头) 两部分。核心层 KV cache 通过 NVLink 共享到「核心 GPU 池」, 外围层 KV cache 留在本地 DRAM。引入 PCIe + RDMA 异步预取机制, 解决跨 GPU 池访问的延迟。实现基于 PyTorch 2.5 + FlashAttention-3 + Megatron-LM 0.12 改造, 支持 elastic 扩缩容 (核心池 GPU 数量可动态调整 4-32)。
⚖️ 取舍：优势: 长上下文 agent 场景吞吐高、显存省、弹性扩缩容; 缺点: 核心-外围分层引入额外 Q 投影计算 (~8% overhead), 对短上下文 (<8K) 反而比 SGLang 慢 5-10%, 仅适合 32K+ 长上下文 agent 场景。跨 GPU 池 NVLink 带宽需求 80-100 GB/s, NVL72 拓扑专属, 传统 8-GPU 服务器需用 InfiniBand 400G 替代, 实际部署成本不低。
📊 效果：在 64K token 上下文 + SWE-bench Verified 任务上, 8×H100 + NVLink 拓扑相对 SGLang v0.3 吞吐 1.83× / 显存峰值降 41% / P99 降 28%; 128K token 上下文 + Multi-SWE-bench 多步 agent 任务相对 vLLM v0.6 吞吐 2.14× / 显存峰值降 47%; 1M token 上下文极端测试 (NIAH 检索 + 1M 文档问答) 仍保持 1.5× 吞吐, 但跨 NVLink 域 RPC 调用成为瓶颈。

SHIFT: Dynamic Compute Relocation Framework for Communication-Aware Chiplet-Based Systems

想象一下: 你有一台「积木拼成的电脑」, 每个积木是一块小芯片 (chiplet), 比如计算积木 (CPU 核心)、存储积木 (HBM)、通信积木 (PCIe 通道)。问题来了——计算积木工作时, 经常要等「数据从存储积木搬过来」, 搬运时间很长, 计算积木就只能空转, 浪费电。SHIFT 干的事情: 让计算积木自己「搬家」, 跑到距离存储积木最近的位置去干活, 这样搬运时间几乎为零。团队用 UCI 校园 8-积木原型机做实验, 跑 ResNet-50 训练, 跟传统「积木固定位置」比, 训练速度提升 22%, 能效 (TOPS/W) 提升 18%。

📖 专业版问题讲解 (展开)
SHIFT 是一种面向 chiplet 异构系统的「动态算力迁移 (Dynamic Compute Relocation)」运行时框架。系统通过统一的片间互联抽象 (UCI-Link) 暴露 chiplet 位置 + 拓扑信息, 调度器根据当前算子 (operator) 的数据访问模式, 动态把对应计算 chiplet 迁移到数据 chiplet 附近, 最小化 inter-chiplet 数据搬运延迟。在 UCI 8-chiplet 原型 (4× compute + 2× memory + 2× I/O chiplet) 上, ResNet-50 训练相对静态布局提升 22% 速度, TOPS/W 提升 18%。
⚙️ 方法：SHIFT 核心是「通信感知调度器 (Communication-Aware Scheduler)」, 在算子图 (DAG) 编译期静态分析数据流, 运行时动态选择 chiplet 布局策略。引入「虚拟 chiplet 抽象 (Virtual Chiplet Abstraction, VCA)」, 让调度器对应用层隐藏物理位置, 通过「移动域 (migration domain)」把 4-8 个 chiplet 分组, 算子内迁移 latency 控制在 200ns 内。实现基于 UCI-Link (UCI 自研片间互联, 带宽 1 TB/s, 延迟 5ns) + 开源编译框架 MLIR, 兼容 PyTorch / TensorFlow / JAX。
⚖️ 取舍：优势: chiplet 异构系统上计算-通信 overlap 几乎最优, 能效比显著; 缺点: 动态迁移引入 200ns-2μs 切换开销, 对 latency < 5μs 的高频算子 (e.g. FlashAttention) 反而成负向。VCA 抽象让框架失去直接控制物理 chiplet 的能力, debug 复杂度上升, 实际生产部署需要专门的诊断工具。UCI-Link 是 UCI 自研互联, 商业 chiplet 系统 (AMD EPYC / Intel Ponte Vecchio) 用 UCI 标准 UCI-Express 不兼容, 需重写。
📊 效果：在 8-chiplet UCI 原型上跑 ResNet-50 (90 epochs, ImageNet), 训练时间从 47.3 小时缩短到 38.7 小时 (-22% 加速), 能效从 28.4 TOPS/W 提升到 33.5 TOPS/W (+18%); 跑 BERT-Large pretraining, 训练时间 -17%, TOPS/W +15%。对 latency 敏感的小算子 (< 5μs) 实际损失 3-7%, 整体被大算子收益覆盖。
🏢 团队：Daryoosh Delavari, Boris Vaisband, Laleh Popryho, Rachel Vaisband → University of California, Irvine; University of Illinois Chicago

LASER: Load-Aware Serving with Early-Exit for Reasoning LLMs at the Edge

想象一下: 你手机/智能音箱上跑一个 AI 助手, 简单问题「今天几号」AI 答得快, 复杂问题「写一首七言绝句」AI 要想半天。LASER 干的事情: 让 AI「想不下去就早交卷」——它内部有一个「难度计」, 觉得这道题太难, 就会提前 2-3 步结束思考, 直接给一个「够用」的回答, 省下来的算力可以同时服务更多用户。团队在 4 卡 Jetson Orin 64GB 边缘服务器 + 1.5B 参数推理模型上做实验, 跟不早退的 vLLM 边缘版比, 吞吐 (req/sec) 提升 2.8×, 平均延迟降低 41%, 用户满意度 (人工标注) 持平甚至略升 (+3%)。

📖 专业版问题讲解 (展开)
LASER 提出一种「负载感知的早退出 (Load-Aware Early-Exit)」边缘推理调度框架, 专门针对 reasoning LLM 在边缘 GPU (Jetson Orin / RTX 4090) 上的部署。框架包含三层: (1) 难度估计器 (Difficulty Estimator), 基于输入 token 嵌入 + 上下文长度预测是否需要完整 reasoning chain; (2) 动态早退策略 (Dynamic Early-Exit Policy), 在 1/3、2/3、3/3 reasoning step 处选择性中断并触发回退到非 reasoning 路径; (3) 边缘资源调度器 (Edge Resource Scheduler), 在多请求并发时根据难度估计结果动态分配算力。
⚙️ 方法：LASER 核心是「轻量级难度估计器」, 用 50M 参数的 TinyBERT 蒸馏模型, 实时判断每个请求「是否需要完整 reasoning」。如果估计为「简单」, 在第 1/3 reasoning step 早退; 估计为「中等」, 在 2/3 处早退; 估计为「复杂」, 才走完整 chain。引入「边缘异构调度」, 4 卡 Jetson Orin 上根据早退概率动态分配 batch size, 简单问题 batch=16 并发, 复杂问题 batch=4 串行。实现基于 vLLM v0.5 + HuggingFace Transformers 4.55, 兼容 Llama-3.1-8B、Qwen2.5-7B、DeepSeek-R1-Distill-1.5B。
⚖️ 取舍：优势: 边缘设备上吞吐大幅提升, 推理成本 (TOPS) 显著降低, 用户满意度持平; 缺点: 难度估计器本身有 5-8% 误判率, 误判为「简单」的复杂问题被早退后, 答案质量下降 12-15%, 需要后置答案验证兜底。边缘调度器动态 batch 调整引入 5-15% 调度开销, 对单请求延迟反而比 vLLM 高 8-12%。TinyBERT 蒸馏模型需要每 3 个月重新训练一次, 运营成本不容忽视。
📊 效果：在 4× NVIDIA Jetson Orin 64GB + DeepSeek-R1-Distill-1.5B 上, Math-500 + GSM8K + HumanEval 三个混合 benchmark: 相对 vLLM v0.5 边缘版, 吞吐从 4.2 req/sec 提升到 11.8 req/sec (+181%), 平均 token 延迟从 1.84s 降到 1.08s (-41%), 答案准确率从 71.4% 提升到 73.6% (+2.2pp, 因为部分问题早退触发了 base model 兜底); 在 100 个并发用户混合负载下, P99 延迟从 8.7s 降到 5.1s (-41%)。

Omni-Flow: A Unified Workflow Orchestration and Distributed KV Cache Sharing Framework for Multimodal Inference

想象一下: 你开的奶茶店现在不只卖奶茶, 还卖咖啡、果汁、冰沙——每种饮品的「配方本」(权重) 都堆在调饮台上 (GPU 显存), 还有每个客人点的「加料清单」(KV cache) 也越积越多。Omni-Flow 干的事情: 让你把所有饮品配方本放到一个「共享书柜」里, 所有调饮台都能用, 加料清单也搬到云端, 调饮台空了随时能搬到新地方去。系统支持文字+图片+视频混合菜单, 一个调饮台能同时处理 50+ 客人的不同饮品组合。在 32 张 H100 集群上, 跑 Qwen2.5-VL-72B 多模态模型, 跟 vLLM 比, 吞吐提升 1.9×, 显存节省 38%。

📖 专业版问题讲解 (展开)

Energy-Aware Scheduling for Serverless LLM Serving on Shared GPUs

想象一下: 你公司有 10 张 GPU 显卡, 部门 A 在跑 AI 翻译, 部门 B 在跑 AI 客服, 部门 C 在跑 AI 写代码。每张卡都很贵 (一张 H100 大约 30 万人民币), 但大部分时间 5 张卡都没人用, 另外 5 张卡累死累活。Energy-Aware Scheduling 干的事情: 让所有部门「按需」用卡, 谁忙谁多拿, 谁闲谁少拿, 没用上的卡直接断电省电。系统还会在电价便宜时多算一点, 电价贵时少算一点。团队用 University of Pittsburgh + HPE Labs 的混合 serverless 平台做实验, 跑 Llama-3.1-8B + Mixtral-8x7B, 跟传统「卡分给部门固定占用」比, GPU 利用率从 47% 提升到 78%, 数据中心电费下降 24%, 用户平均响应延迟持平。

📖 专业版问题讲解 (展开)
Energy-Aware Scheduling (EAS) 提出一种「能耗感知的 serverless LLM serving」框架, 用于共享 GPU 集群的多租户推理场景。框架分三层: (1) 功耗预测器 (Power Predictor), 基于 LSTM 预测每个 serverless function 的瞬时功耗; (2) 调度器 (Scheduler), 根据实时电价 + GPU 利用率动态调整 function 部署密度, 闲置 GPU 断电进入低功耗状态; (3) QoS 保障器 (QoS Guardian), 保证每个租户的 SLO (p99 延迟 + 最小可用 GPU 数量)。实现基于 Knative 1.15 + Volcano 1.10 + NVIDIA MIG (Multi-Instance GPU), 支持 Llama-3.1-8B、Mixtral-8x7B、Qwen2.5-7B 三个模型。
⚙️ 方法：EAS 核心是「电价 + 功耗双约束调度」。调度器每 60 秒拉取一次实时电价 (PJM/CAISO 公开 API), 配合 GPU 实际功耗 (通过 NVIDIA NVML 读取) 动态决策: 电价高 + 负载低时, 缩容 GPU 实例 + 闲置卡断电; 电价低 + 负载高时, 扩容 + 唤醒备用卡。引入「QoS 预算」(每个租户最小可用 GPU 比例), 即使缩容也保证底限。LSTM 功耗预测器训练数据来自 HPE Labs 内部 8 个月生产 trace, MAE 6.2%。
⚖️ 取舍：优势: 共享 GPU 集群利用率从 47% 提升到 78%, 数据中心电费下降 24%, 同时不破坏租户 SLO; 缺点: 调度器复杂度上升, 决策延迟 200-400ms 比传统 Knative 调度慢 5-10×, 对 latency 极敏感 (P99 < 100ms) 的请求反而不友好。MIG 切片粒度有限 (1 卡最多 7 个 MIG 实例), 不能完全替代卡级调度。LSTM 模型在负载突变 (e.g. 秒杀 / 大促) 时预测误差上升 15-25%, 需要 fallback 策略。
📊 效果：在 University of Pittsburgh 校园 8×H100 集群 + HPE Labs 生产 trace 重放下, 跑混合 12 个 serverless function (10 个 Llama-3.1-8B + 2 个 Mixtral-8x7B): GPU 利用率从 47% 提升到 78% (+31pp), 24 小时数据中心电费从 $487 降到 $371 (-24%), P99 延迟从 1.84s 略升到 1.92s (+4.3%, 在 SLO 2.0s 内); 模拟 4 倍负载峰值 (秒杀场景), QoS Guardian 触发率 8.3%, 没有出现 SLO 突破。
