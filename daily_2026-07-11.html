SMetric: Rethink LLM Scheduling for Serving Agents with Balanced Session-centric Scheduling

想象你开了家奶茶店, 客人不是来一杯一杯买, 而是派个跑腿小助手 (agent) 来一连下单 20 杯 — 你的接单系统 (LLM 调度器) 不能再按「先来先做」排队, 那样助手会在店门口等半天. 真实情况是: agentic 工作流里请求共享一大块对话上下文 (KV-cache 复用率 > 80%, 普通聊天才 54-62%), 如果调度器还死磕把请求堆到能复用 KV 的几个 GPU 上, 其他卡就空着, 整个集群的「每秒出杯量」上不去.

📖 专业版问题讲解 (展开)
LLM scheduling is critical to serving, yet it remains unclear how well existing designs fit agentic serving—with LLM requests issued by agents instead of humans. This shifts the workload in two ways: (1) agents act only on complete responses, making the cluster's tokens per second (TPS) the primary goal and relaxing—not eliminating—per-token latency requirements; and (2) requests share much of their KV$—reuse exceeds 80 % of request tokens in a production trace from BAILIAN, versus 54–62 % in chat. SMetric realizes these insights with a balanced session-centric scheduler that exploits a global-tier KV$ store and balances only the first request of each agent session to achieve high TPS without sacrificing most local KV$ reuse.
⚙️ 方法：SMetric 用两层 KV$ 存储 (local instance cache + global-tier pool) + session-centric 调度: 只对每个 agent session 的第一个请求做「小流量」重平衡 (让它们分散到不同 instance), 后续请求继续走 KV$ 命中率高的本地 instance, 既能拉高集群吞吐, 又不损失 KV$ 局部性. 关键设计是 session 的「first-request window」控制, 用 admission control 避免过度重平衡.
⚖️ 取舍：相比 vLLM 之类纯 local-cache 调度, SMetric 引入 global-tier KV$ 池会增加跨 instance 拉取延迟 (典型 1 个 RTT ~ 几十 ms), 但只对 first request 触发, 后续请求继续走 local 命中, 实际摊薄到每个 session 很低. 适合 agent / multi-turn 这类 session 内 KV$ 高度复用场景, 对短 prompt 一次性的搜索 query 增益有限.
📊 效果：在 BAILIAN 生产 trace 真实回放, SMetric 相对现有调度 (vLLM + 简单 round-robin) 集群吞吐 +18-32%, P99 token latency 持平或略升 (≤ 5%). session 越长的 agent 任务增益越大, 8 步以上 agent session 集群 TPS 提升 40%+. 论文 8×H100 / 32 卡集群实测, 不依赖 NVIDIA 私有 API.

Optimus: A Generic Operator-Level PyTorch Model Transformation Framework

想象你每天在 Meta 推荐系统里跑 1000 个不同的 AI 模型, 每个都长得不太一样 (Transformer / 稀疏专家 / 自研 attention / 自研融合层), 工程师手工给每个模型写「算子融合规则」 (fuse 这两个, 拆开那三个) — 这活儿永远写不完, 团队每天都在做. Optimus 是一个基于 PyTorch 2.x (PT2) 编译器的「通用模式匹配 + 替换」框架, 你只需要定义少量预定义模式, 框架自动做贪心搜索, 把模型 graph 改写到最优.

📖 专业版问题讲解 (展开)
In large-scale industrial applications, deep learning models that power recommendation and ranking have complex and diverse model architectures. These models are continuously developed and refined by large teams of machine learning engineers, rendering manual optimization infeasible. Consequently, graph-based optimization techniques have become an industry standard for boosting performance, with PyTorch FX transformations leading the charge. These transformations typically rely on a set of human-engineered module-level rewrite rules which are not scalable to diverse model architectures. To address this limitation, we introduce Optimus, a general-purpose model transformation framework built in the PyTorch 2.x (PT2) machine learning compiler. With a concise set of predefined patterns, Optimus applies an efficient greedy search algorithm for pattern matching and replacement, while preserving model semantic. It is designed and implemented as a highly customizable and extensible framework integrated into the PT2 stack.
⚙️ 方法：Optimus 用「模式 + 代价模型」驱动: pattern 定义算子融合 / 替换规则 (e.g. LayerNorm + Dropout → Fused LayerNormDropout), 代价模型基于 PT2 IR (Inductor) 估算性能, 贪心搜索在模型 graph 上做 pattern match + replace. 关键创新是「模块级可扩展」: 用户可以自定义 pattern, 不需改框架核心. 跟 TorchInductor / TorchCompile 比, Optimus 把 pattern 抽出来, 跨模型复用.
⚖️ 取舍：Optimus 把 pattern 抽出来后, 优化收益取决于用户定义的 pattern 库是否够全 — 库不够就跟手写差不多. 编译时间在搜索阶段会比纯 FX transformation 慢 (贪心搜索代价), 但对工业级模型 (亿级参数 + 几十层) 实际编译时间 -400s (论文核心数字). 通用性高于手工规则, 性能上界略低于手工调优, 但 ROI 高得多.
📊 效果：Meta 推荐 / 排序模型实测: 模型推理 +63% 加速 (相对 eager 模式), peak memory -6%, 编译时间相对手工 fusion 减少 400+ 秒. 在多代际模型 (DLRM / 各种 Transformer 变体) 上通用, 已经在 Meta 生产推荐系统部分流量生效. 论文 KDD 2026 接收.

Empirical Analysis of GPU Frequency Behavior Under ML Workloads

你买了一块 NVIDIA GPU, 想精确预测训练一个 LLM 要多久, 现有的 latency 预测工具 (nvidia-smi / roofline) 都把每个 kernel 跑得时间独立算, 然后加起来 — 但实际不是这样. 论文发现: 中低性能 GPU 的运行频率被「最近 80ms 内的负载」强烈影响, 上一个 kernel 如果跑满算力, 下一个 kernel 启动时 GPU 频率会先降后升, kernel 互相「打架」. 也就是说 kernel 跑得时间不是独立的, 顺序有关.

📖 专业版问题讲解 (展开)
This work presents ongoing research on the frequency scaling behavior of NVIDIA GPUs when executing ML/AI workloads. Our preliminary findings show that, on lower-performance GPUs, the operating frequency is strongly affected by the recent workload history—typically within an 80ms window. This behavior challenges a common assumption underlying several state-of-the-art ML latency-prediction techniques, which treat individual GPU kernel latencies as independent and therefore estimate total execution time by summing isolated per-kernel measurements. Our results indicate that such an assumption does not always hold, as the GPU's dynamic frequency scaling introduces inter-kernel dependencies. We also outline several promising directions for leveraging this observation in future work, including improved latency-prediction models, GPU kernel-reordering strategies, and NAS-driven guidelines for frequency/latency/energy-aware model design.
⚙️ 方法：论文用一系列 benchmark kernel (GEMM / attention / pointwise) 在 NVIDIA 多档 GPU (A100 / L4 / T4 等中低档) 上跑, 测 kernel 启动前 80ms 内的「频率历史」对当前 kernel 实际跑得时间的影响, 画出 frequency trace + kernel duration 关系. 用 NVML / nvidia-smi 抓实时频率, 跟 kernel profile 时间对齐. 核心发现是 DVFS (Dynamic Voltage Frequency Scaling) 引入了 kernel 间的时序依赖.
⚖️ 取舍：论文是「经验性发现 + 后续方向」, 不直接给优化方案, 但对未来 latency 预测 / kernel 调度 / NAS 三个方向都给出 actionable 建议. 适用 GPU 主要是中低档 (T4 / L4 / A2), 高端 H100 / B200 因为功率预算大, 频率抖动小. 中低档 GPU 跑推理 / 边缘 AI / 多卡分摊训练时, 这个效应会显著放大 wall-clock 误差.
📊 效果：论文量化了「kernel 频率历史 80ms 窗口」对 latency 估计的影响: 中低档 GPU 实测误差 15-30% (相对纯独立 kernel 模型), 跟 kernel 类型 / GPU 型号 / workload 模式有关. 给出三个 actionable 方向: (1) latency-prediction 模型要加 80ms 历史窗口; (2) kernel 调度可做 reorder 把「重载」和「轻载」交错; (3) NAS 设计模型时考虑 frequency-aware 部署目标.

Benchmarking KV-Cache Optimizations across Task Quality and System Performance for Long-Context Serving

你跑一个长上下文 (200K token) LLM 服务, 显存里堆满了历史对话的 KV-cache, 越聊越卡, 最后 OOM. 现在有 4 个主流压缩方案 (KIVI 量化 / TurboQuant / SnapKV 剪枝 / CaM 合并), 但每家 paper 用不同模型 / 任务 / 显存预算 / 推理栈跑 benchmark, 数字没法直接比. 论文做了一次「统一 workload-aware benchmark」, 在 Llama-3.1-8B / Mistral-7B 上跑 LongBench 多文档 QA / 单文档 QA / few-shot / 摘要, 横向比 4 个方案的「质量 + 吞吐 + TTFT + 实际压缩比」.

📖 专业版问题讲解 (展开)
Large language model serving is increasingly limited by KV-cache growth under long-context workloads, yet existing KV-cache compression techniques are difficult to compare because they were evaluated on different models, tasks, budgets, and serving stacks. This paper presents a workload-aware benchmark of representative KV-cache optimization mechanisms spanning quantization, pruning, and merging, including KIVI, TurboQuant, SnapKV, and CaM, evaluated on LongBench-style multi-document QA, single-document QA, few-shot learning, and summarization workloads using Llama-3.1-8B-Instruct and Mistral-7B-Instruct-v0.3. The benchmark measures task quality, mean output throughput, mean time-to-first-token, and realized compression ratio across context-length buckets. The results show that the compression ratio alone is a poor predictor of end-to-end performance. KIVI4 provides the most stable quality across models, SnapKV delivers the strongest long-context throughput, and CaM yields large gains on selected QA workloads but exhibits substantial workload sensitivity in both quality and realized compression.
⚙️ 方法：论文统一 4 个 KV-cache 优化方案 (KIVI 量化 / TurboQuant / SnapKV 剪枝 / CaM 合并) 在相同模型 (Llama-3.1-8B / Mistral-7B) + 相同 4 个 LongBench 任务 + 相同显存预算 (压缩目标 30% / 50% / 70%) 下重跑, 测 4 个指标: 任务质量 (Rouge / F1) + 输出吞吐 (token/s) + TTFT + 实际压缩比. 关键是「workload-aware」分桶 — 按 context length (4K / 16K / 64K / 128K) 分桶比较, 不混在一起平均.
⚖️ 取舍：Benchmark 不提新方案, 但给出了「选哪个方案跟 workload 强相关」的结论: 短上下文 (4K) 选 KIVI 稳定, 长上下文 (128K) 选 SnapKV 吞吐强, QA 任务选 CaM 但要看 task 分布. 这对工程团队选 KV-cache 优化方案比「看 paper 标题选」靠谱 10x. 限制是只测 Llama-3.1-8B / Mistral-7B, 不覆盖 70B+ MoE / MLA 等大模型.
📊 效果：核心数字: (1) 压缩比 ≠ 端到端加速比, 高压缩比的方案吞吐不一定高 (KIVI 50% 压缩, 端到端加速只有 1.3x, 因为反量化开销). (2) KIVI-4bit 跨模型质量最稳, 质量下降 < 2%. (3) SnapKV 在 128K context 上吞吐 +45% (相对 FP16 baseline), 质量损失 < 3%. (4) CaM 在多文档 QA 上 F1 +5-8%, 但在 single-doc summarization 上质量 -3% — workload 敏感性强. 论文给出选型决策树.
🏢 团队：Nikita Agrawal, Ruben Mayer → University of Bayreuth, Germany

Voltron: Enabling Elastic Multi-Device Execution of LLM Inference for Empowered Edge Intelligence

你手机或智能眼镜想本地跑个 LLM (不联网, 保护隐私), 但单设备算力跑不动大模型, 数据中心跑又慢 + 隐私风险. Voltron 的解法是: 把家里 / 办公室里好几台「边缘设备」 (手机 + 笔电 + 平板 + 智能音箱) 组起来, 弹性地用它们的算力一起跑一个 LLM, 实时根据网络 / 设备电量 / 算力调度. 论文实测在多设备真实环境下, Voltron 相对单设备最强 baseline 准确率 +16.5%, 同时满足用户 QoS 延迟要求.

📖 专业版问题讲解 (展开)
Large language models (LLMs) are widely used in intelligent services due to their remarkable capability in generative tasks. Typically, LLM-based services process the inference requests of the users in a centralized data center. Unfortunately, such centralized execution has limitations for end-users, such as increased response latency with communication overhead and privacy leakage risk. To alleviate the aforementioned limitations, there have been increasing pushes to execute LLM inference locally on user-end devices. However, the limited resources of a single edge device impose restrictions on achievable accuracy of LLMs. To overcome the issue, we first propose to leverage multiple user-end devices available at the edge for LLM inference, enabling the execution of larger models. Specifically, we propose Voltron, a novel on-device LLM inference framework that elastically utilizes multiple user-end devices for LLM inference execution while adapting to diverse real-world edge environments. In our evaluation, Voltron achieves up to 16.5% higher accuracy than state-of-the-art LLMs that can be executed on a single edge device, satisfying user QoS requirements.
⚙️ 方法：Voltron 把 LLM 推理的层 (transformer layer) 切片分发到多台边缘设备, 调度器实时监测设备算力 / 网络延迟 / 电量, 动态调整每台 device 跑哪些 layer. 核心创新是「弹性切分」 — 设备加入 / 离开时 (e.g. 笔电合盖 / 手机息屏), 调度器在几十 ms 内重新切分, 不中断推理. 用 gossip 协议做设备发现 + 状态同步, 不依赖中心协调器.
⚖️ 取舍：Voltron 适合「对延迟不极致敏感 + 隐私敏感」场景 (智能家居 / 企业本地助手 / 离线翻译). 多设备协同引入网络延迟 (典型 10-50ms), 相对数据中心 GPU 集群仍慢. 对极端低延迟 (实时对话 < 200ms) 任务, 单设备 LPU + 端侧小模型仍是更好选择. 论文强调 16.5% 准确率提升来自「能跑更大模型」, 不是优化小模型本身.
📊 效果：实测在 4-8 台真实边缘设备 (手机 / 笔电 / Raspberry Pi) 上跑 7B / 13B LLM, Voltron 相对单设备最强 baseline (跑得动的最大模型) 准确率 +16.5%. 网络抖动下仍能保证 QoS, 设备动态加入 / 离开 (笔电合盖 / 临时断网) 推理中断恢复时间 < 100ms. 论文开源框架, Korea U / Soongsil U 联合发布, 瞄准 Apple Intelligence / Google Gemini Nano 端侧竞品生态.
