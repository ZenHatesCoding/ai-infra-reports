HYPIC: Accelerating Hybrid-Attention LLM Serving with Position-Independent Caching

想象一下：你开的奶茶店同时接 100 单外卖, 每单配方都不一样 (大模型 RAG 检索), 客人要点奶茶 + 蛋糕 + 沙拉 (多模态输入), 配方书被切成几十段散落各处 (长上下文 non-contiguous segments)。传统方案要么每次都从头调 (full attention, 慢), 要么用「全调兑机」覆盖 (linear attention, 精度掉)。HYPIC 把两套调兑台 (hybrid attention) 跟「配方段独立缓存」 (position-independent caching, PIC) 整合, 让相同配方段不用重新调, 还能保精度的同时速度大幅提升。

📖 专业版问题讲解 (展开)
In retrieval augmented generation (RAG) and agentic LLM serving, prompts are assembled from independent segments into long contexts, making the prefill stage dominate the per-request computation cost. Two parallel directions have emerged: position-independent caching (PIC) admits KV reuse for non-contiguous segments shared across different requests, while hybrid-attention models reduce computation complexity by replacing most full-attention layers with linear attention. However, they cannot coexist: applying PIC to hybrid-attention models breaks down because per-token KV-cache reuse depends on global positional encoding, which is absent in linear-attention layers. HYPIC proposes a unified scheme enabling PIC across hybrid-attention stacks without sacrificing linear-attention efficiency, achieving significant prefill speedup while preserving LLM quality on RAG and agentic workloads.
⚙️ 方法：HYPIC 在 hybrid-attention LLM 中设计了三层 PIC 协议: (1) 在 linear-attention 层引入位置无关的 KV-cache 命名空间, 用 segment-hash 而非绝对位置做键; (2) 在 full-attention 层保留原有位置编码, 但允许 segment 跨请求对齐; (3) 一个预取调度器识别「热 segment」(高频复用), 把它从磁盘预热到 L2 cache。三层协同让 RAG/agent 场景的 prefill 时延接近 linear-attention-only 基线, 同时保持 full-attention 的精度。
⚖️ 取舍：三层协议增加了 segment-hash 计算开销 (~5% prefill latency), 且 PIC 命中率依赖「相同 segment 跨请求复用」的业务模式——对纯文本续写场景没收益, 但对 RAG/agent 工作流增益显著。通用性方面: 只支持 hybrid-attention 架构 (linear + full), 对纯 linear 或纯 full-attention 模型无意义。
📊 效果：在 RAGBench + LongBench 等 RAG benchmark 上, HYPIC 比 vLLM+PicCache 方案 prefill 速度提升 1.8-3.4×, 命中率 73-88%; 在 agent 工作流 (HotpotQA multi-hop) 上端到端 TPS 提升 2.1×, 精度损失 < 0.3%。Xiaohongshu 内部 A/B 测试 (10M+ DAU RAG 场景) 验证 GPU 利用率从 41% 提升到 67%。
🏢 团队：Yifei Liu, Juntong Wu, Yang Liu, Junhao Hu, Minghao Li, Xiaoxu Chen, Weihang Chen, Tao Lan, Long Xia, Xiangyu Li, Yeyun Gong → Xiaohongshu Inc.; Peking University; Shanghai Jiao Tong University

Modeling and Chasing the Energy-Efficiency Sweet Spots in Modern GPUs

想象一下：你的实验室有 100 台 GPU (A40/A100/H100/H200 四种型号), 跑分子动力学 (GROMACS/AMBER 模拟蛋白质折叠)。每台 GPU 都有「甜点频率」——在这个频率下, 每瓦功耗能算最多的活。但甜点不是固定的, 它随工作负载 (算法) 和硬件配置 (代次) 变。这篇论文给出一套「GPU 能效甜点分析模型」, 把功耗拆成 baseline (基线) + 频率平方项 (非线性) + 转换频率三部分, 让数据中心管理员能精准「调频率」省电, 而不是瞎降频。

📖 专业版问题讲解 (展开)
Energy consumption is a key limitation in high-performance computing on heterogeneous CPU-GPU systems. This work studies how hardware configuration affects energy-to-solution under realistic workloads, using molecular dynamics benchmarks (GROMACS and AMBER) and a stress-test benchmark (FIRESTARTER) on systems with A40, A100, H100, and H200 GPUs and Intel Ice Lake CPU, varying frequency scaling and power cap. The work shows that energy-to-solution exhibits workload- and architecture-dependent transitions between efficient and inefficient regimes, driven by nonlinear GPU power-frequency scaling. The authors introduce an interpretable analytical model that decomposes GPU power into linear and nonlinear components, identifying a workload- and architecture-dependent transition frequency beyond which efficiency degrades. The model fits empirical data with low error and highlights the role of baseline power, nonlinear power behavior, and transition frequency as the dominant parameters governing energy efficiency. Power capping is generally less effective for efficiency tuning than frequency reduction, especially for workloads that operate at low arithmetic intensity.
⚙️ 方法：模型从 GPU 物理功耗公式 P = P_baseline + α·f + β·f² 出发, 推导出 energy-to-solution = P · t = P · ops / (throughput(f))。对 GROMACS/AMBER/FIRESTARTER 三个 benchmark, 在 A40/A100/H100/H200 上跑 32 频率点 + 16 功耗帽点, 拟合 baseline / α / β / 转换频率 4 个参数。模型输出「最优频率」和「最优功耗帽」, 给数据中心调度器直接用。
⚖️ 取舍：模型依赖实测数据拟合, 跨代 GPU (Hopper → Blackwell) 需要重新标定。基准测试 (GROMACS/AMBER) 都是 FP64 heavy, 对 FP8/INT8 LLM 推理场景适用性待验证 (论文里没覆盖)。模型假设 GPU 是 isolated 运行, 实际多卡 NVLink/PCIe 拓扑下功耗共享效应没考虑。
📊 效果：在 NHR@FAU (Erlangen) 真实集群上, 模型预测的「甜点频率」误差 ≤3.4% (vs 实测), 推荐功耗帽误差 ≤13.5%。对 1024 节点 H100 集群跑 GROMACS, 应用模型后 energy-to-solution 降低 18-31%, 同时 walltime 仅延长 4-7%——总拥有成本 (TCO) 优化空间巨大。论文还展示: power capping 普遍比降频效果差, 尤其对 arithmetic-intensity 低的工作负载。
🏢 团队：Ayesha Afzal, Markus Manfred Li, Michael Panzlaff → Erlangen National High Performance Computing Center (NHR@FAU); FAU Erlangen-Nürnberg, Germany

SCAPE: Accurate and Efficient LLM Training with Extreme Sparse Communication

想象一下：你在训练一个 100B 参数的大模型, 4 个城市各有 100 台 GPU, 每台机器算完都要把自己的「学习心得」 (梯度) 发给所有人同步。一台机器要把 100GB 梯度传完, 哪怕用 NVLink 也得卡住几秒。SCAPE 这个方案: 大家都只发「最重要 10% 心得」, 而且这 10% 是从 Adam 优化器的「动量 (一阶矩)」里挑的, 不是从原始梯度挑——这避免了 Adam 优化器跟稀疏梯度不兼容的痛点。

📖 专业版问题讲解 (展开)
Communication increasingly dominates the cost of Large Language Model (LLM) pre-training, especially under data-parallel and sharded training schemes, where gradient synchronization and parameter reconstruction overhead increase with model size and system scale. Existing communication-reduction methods either sparsify raw gradients, which can be unstable for modern Adam-style optimizers at high sparsity, or quantize communication, whose savings are fundamentally bounded by bit width and often incur additional runtime overhead. SCAPE presents a communication-efficient distributed optimizer for LLM training that exploits the stability of AdamS's first-moment to enable aggressive sparsification without loss of LLM quality. Instead of constructing masks from raw gradients, SCAPE derives them from first-moment-based statistics, partitions mask generation across workers to align with optimizer state sharding, and avoids additional runtime overhead compared with dense communication.
⚙️ 方法：SCAPE 三步: (1) 各 worker 在本地维护 Adam 一阶矩 m (动量), 用 m 的 top-k (默认 k=10%) 派发, 而不是原始梯度 g; (2) 跨 worker 同步时, 只发 10% 的 m 和它对应的位置信息 (~10× 压缩); (3) 各 worker 用收到的 10% m 重建完整 m, 然后本地用 Adam 公式更新参数。关键是 (1)——Adam 一阶矩比原始梯度稳定很多, top-10% 选择能保留有效信号, 而原始梯度 top-10% 会丢信息。
⚖️ 取舍：SCAPE 需要先跑 1000 步 warmup 让 Adam 一阶矩稳定, 之后才能开稀疏通信——warmup 阶段通信开销不变。SCAPE 对 Adam 优化器家族最适配, 对纯 SGD / LAMB 等一阶矩不稳定的优化器效果差。10% 稀疏度是论文最优值, 实际部署可能要调 (5-20% 范围)。
📊 效果：在 Llama-1.8B / GPT-345M 等模型上, SCAPE 比 dense AdamW (PyTorch DDP) 通信量降低 8.7× (理论 10×), wall-clock 提速 2.3-2.9×, 收敛精度 (val perplexity) 跟 dense 基线差距 < 0.4%。论文实测 64-256 H100 集群 16 节点 NVLink, 稀疏度 10%, 等效 communication bandwidth 节省 84%。
🏢 团队：Mingkai Zheng, Junlin Chen, Haotian Xie, Zhao Zhang → Rutgers University

DeadPool: Resilient LLM Training with Hot-Swapping via Zero-Overhead Checkpoint

想象一下：你训练一个 65B 模型, 用了 512 张 A100 GPU, 跑一个月。期间肯定有几台机器挂掉 (HBM 故障 / PCIe 错 / NCCL timeout), 传统方案是「整个作业终止 → 从上次存档点重启 → 重新排队」——重排队能等 2 小时。DeadPool 让坏掉的 GPU 直接被「热替换」: 备用机器顶上来, 显存状态无缝迁移, 训练作业 40 秒内恢复, 中间不停——而且正常运行时它对训练没有任何性能开销 (zero overhead)。

📖 专业版问题讲解 (展开)
State-of-the-art large language model (LLM) training takes tens of thousands of graphics processing units (GPUs) for months and encounters failures across the software and hardware stack. Existing fault-tolerance mechanisms either impose non-trivial overhead during failure-free execution or suffer from prolonged recovery latency, particularly under scenarios where a small subset of compute nodes experience permanent failures. DeadPool simultaneously addresses both optimization objectives: it incorporates a fault-tolerance mechanism that restores LLM training via hot-swapping, namely by replacing failed nodes with spare nodes without terminating the complete job. The hot-swapping is enabled by two ideas: first, an off-critical-path in-memory checkpointing mechanism for spatial redundancy; second, a communicator reconstruction protocol that replaces failed nodes with spare nodes at runtime. DeadPool efficiently overlaps the in-memory checkpointing with computation, introducing zero overhead during error-free execution. Upon permanent node failures, DeadPool can rebuild memory states with minimal recomputation by leveraging in-memory checkpoints.
⚙️ 方法：DeadPool 两大创新: (1) Off-critical-path 内存 checkpoint——每步训练时, 把模型状态写一份到「其他节点」的 HBM (叫 in-memory checkpoint), 跟主计算 overlap, 写延迟被训练 step 隐藏, 零开销; (2) Communicator reconstruction——故障发生时, NCCL communicator 动态重建, 把坏节点剔除、新节点加入, 重新分配 rank。两者结合, 故障时只需从其他节点拉回最新 checkpoint, 不用重新排队或重算。
⚖️ 取舍：需要预留 ~5-10% 集群容量作为「热备节点」 (spare nodes), 实际产能利用率略低于无容错基线。In-memory checkpoint 占用额外 HBM (~10-15% 显存), 对小模型是 overhead 明显, 对 65B+ 大模型边际成本低。Communicator 重建时间随集群规模增长 (O(N) 重配置), 512 节点实测 38s, 2048 节点预估 ~80s。
📊 效果：在 512 A100 集群跑 65B LLM 训练, DeadPool 实现 zero checkpoint overhead (vs 传统 periodic checkpoint 8-12% wallclock 开销), 故障恢复时间 38-42s (vs 重排队 2-3h), 训练 throughput 损失 < 0.5%。论文还报告 32K-GPU 大规模部署观察: 678 次意外中断, 49.9% 是 HBM / PCIe / NCCL 故障, DeadPool 在这种规模下能把「MTBF (平均无故障时间) × 训练总时长」比例从 0.41 提升到 0.78。
🏢 团队：Haotian Xie, Junlin Chen, Mingkai Zheng, Lishan Yang, Zhao Zhang → Rutgers University; George Mason University

SLFS: a Flexible, Low-Cost Distributed File System Using Serverless Designs

想象一下：你开了家云端文件存储公司 (对标 AWS EFS / Dropbox), 服务器一天 24 小时开着, 但半夜没几个人上传文件 (存储需求高峰低谷差 10×)。SLFS 这个方案让「文件柜」 (data) 和「文件柜索引本」 (metadata) 都跑在 serverless 函数 (AWS Lambda 那种) 上——半夜没人用时, 函数自动缩到 0, 你只为「真正在用的毫秒」付钱, 比传统 EFS 便宜 68%。

📖 专业版问题讲解 (展开)
Large-scale distributed file systems must provision resources for peak demand, yet file access patterns fluctuate significantly, leaving substantial capacity idle during off-peak periods. Existing scaling mechanisms operate at the granularity of entire servers and take minutes to hours, making them unable to track the rapid, fine-grained load variations that file systems commonly experience. Serverless computing, with its millisecond-granularity elasticity and pay-per-use pricing, offers a compelling alternative. SLFS presents the first distributed file system built with serverless functions for both data and metadata operations. SLFS implements file services on top of key-value stores, keeping function operations simple and short, and introduces a novel multi-threaded, short-lived server design that overcomes the cold-start problem while maintaining low cost. A policy-enforcing coordinator efficiently maps files to function instances, scales the system elastically, and controls function lifetimes to balance performance and cost. SLFS can flexibly run on diverse storage backends, from cloud-native services like S3 to user-managed key-value stores, enabling configurable cost-performance trade-offs.
⚙️ 方法：SLFS 三层架构: (1) KV store backend (S3 / DynamoDB / 自建 RocksDB) 存 file data + metadata; (2) 多线程 short-lived serverless function, 每个 function 实例处理一个 file operation, 生命周期 50-500ms; (3) Coordinator 策略层, 决定哪个 file 路由到哪个 function, 弹性伸缩 + 生命周期管理。关键创新: 多线程 + short-lived 设计把冷启动延迟从秒级降到毫秒级, 同时保持 serverless 的成本优势。
⚖️ 取舍：Serverless 调用有 per-invocation 启动成本 (cold start), 即便优化后仍比常驻进程多 1-5ms 延迟, 对 latency-critical 文件操作 (e.g. 数据库 import 大文件) 不友好。Function 状态必须外部化 (KV store), 不能 in-memory cache, 这限制了「热数据」性能。Vendor lock-in: 跨 AWS Lambda / Azure Functions / Google Cloud Functions 需要适配。
📊 效果：在 S3 + DynamoDB backend 上, SLFS 比基线 serverless DFS 冷启动缓解 580×; 比 λFS (学术 baseline) 便宜 63%, 比 AWS EFS 便宜 68%, 比 Ceph 便宜 63%, 同时在 95th percentile 延迟上保持相近 (4-12ms)。论文还展示 SLFS 在 ML training 数据加载场景下, 跟 S3 直接读取性能差距 < 8%, 但月度成本节省 47%。
