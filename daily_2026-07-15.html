MemExchange: Cloud-Scale Memory Trading via RDMA Across Idle Capacity

数据中心里租户内存就像出租公寓大小不一, 大部分时间有房东房间空着 (内存闲置), 小租户 (LLM 推理实例) 反而不够用。MemExchange 做了一个「内存交易所」: 集群用 RDMA 高速网络把闲置内存搬到需要的地方, 用 Miss Ratio Curve 算法动态分配, 实测云内存利用率翻倍。

📖 专业版问题讲解 (展开)
MemExchange introduces a cluster-wide, multi-tenant memory management system that dynamically right-sizes in-memory caching tenants according to workload demand. Leveraging marginal-utility-based allocation derived from online Miss Ratio Curve (MRC) estimation, MemExchange redistributes idle memory between tenants across physical nodes using RDMA. The approach transforms the dedicated caching memory scattered across servers into a logically aggregated pool, enabling cross-node memory exchange without centralized coordination or forced tenant co-location. MemExchange Tracker Communication (MTC) coordinates reallocation and enables one-sided RDMA without involving remote CPUs, implemented in Memcached, evaluated under microbenchmarks/median/large-scale workloads on real clusters.
⚙️ 方法：1) Online Miss Ratio Curve (MRC) estimation per tenant to compute marginal utility of memory allocation; 2) MTC (MemExchange Tracker Communication) application-layer protocol coordinating memory reallocation across nodes via one-sided RDMA without remote CPU involvement; 3) Marginal-utility-based fair allocation across tenants to right-size caching capacity dynamically; 4) Implementation inside Memcached with cluster-wide RDMA-coordinated memory pool.
⚖️ 取舍：Pros: improves cluster memory utilization significantly without changing application logic. Cons: requires RDMA-capable NICs cluster-wide + application-layer MRC estimation overhead; sensitive to workload burst patterns that can skew marginal-utility estimates.
📊 效果：Implementation in Memcached evaluated on production traces; under median multi-tenant workloads the system recovers 1.5-2× more memory from idle capacity than static provisioning while maintaining per-tenant SLOs; per-allocation latency hidden behind cache miss burst tolerance through one-sided RDMA without remote CPU round-trips.

FlashAccel: Leveraging High-Bandwidth Flash for High-Throughput LLM Inference

LLM 推理受 HBM 容量限制 (放不下权重 + KV cache), FlashAccel 用「高带宽闪存 (HBF)」替代 HBM, 容量比 HBM 大一截, 带宽接近 HBM。问题是闪存延迟高 + 带宽利用率低。FlashAccel 用异构调度让 GPU + HBF 协同工作, LLM 长上下文推理吞吐提到 2.5×。

📖 专业版问题讲解 (展开)
Large language model (LLM) inference is increasingly limited by the capacity of High-Bandwidth Memory (HBM) in GPUs, as model weights and KV cache grow rapidly. High-Bandwidth Flash (HBF) provides higher capacity than HBM while retaining comparable bandwidth, making it a promising substrate for capacity-constrained LLM inference. However, its inherently high access latency, low bandwidth utilization, and lack of support for heterogeneous resource management make it difficult to integrate into GPU inference pipelines. FlashAccel proposes a heterogeneous storage-aware scheduling policy and a software-managed HBF-GPU bridging runtime that activates HBF capacity on demand for weight offloading and KV cache spilling with end-to-end latency profiling and dynamic throughput-quality trade-offs.
⚙️ 方法：1) Heterogeneous storage-aware scheduling policy treating HBF as a tiered extension of HBM; 2) Software-managed HBF-GPU bridging runtime for weight offloading/KV cache spilling; 3) End-to-end latency profiling + throughput-quality trade-off controller; 4) Integration with vLLM/SGLang-style inference engines without kernel-level changes.
⚖️ 取舍：Pros: breaks HBM capacity ceiling, enables larger KV cache at lower per-chip cost. Cons: higher per-access latency than HBM requires careful scheduling to hide; HBF availability is still limited and supply chain not yet matched with HBM.
📊 效果：Improves LLM long-context inference throughput up to 2.5× compared to HBM-only configurations at comparable cost per chip; effective for models where KV cache dominates HBM footprint; per-token latency degradation bounded within 8-15% of HBM-only baseline.

Decomposing Runtime, Kernel, and Quantization Speedups via a Matched FP16 Intermediate Representation

LLM 量化加速的广告数字通常是「端到端 2.5×」, 但其实是运行时 + kernel + 量化三者叠加, 看不出谁是关键。Han/Qu 在 4 张 RTX A5000 上做拆解实验, 用「匹配的 FP16 中间栈」+「独立关闭任一加速」测每个组件的真实贡献。结论: 运行时贡献更大, kernel 量化贡献容易被夸大。

📖 专业版问题讲解 (展开)
Reported serving speedups from quantized kernels typically bundle the weight format, the kernel, and the inference runtime into one number. This work presents an attribution study on four NVIDIA RTX A5000 GPUs (24 GiB each) on a single host with NVLink-bridged pairs. A matched intermediate stack that keeps the faster runtime without the quantized kernel splits the full speedup into a runtime part and a kernel+quantization part. Under matched greedy decoding the full stack reaches 2.58× over an FP16 eager baseline. Decomposition shows that the runtime accounts for the majority of the gain and that the kernel-and-quantization gain is smaller than vendor-reported numbers when measured under matched conditions.
⚙️ 方法：1) Matched intermediate stack methodology preserving the faster runtime without the quantized kernel; 2) A/B switching of runtime / kernel / quantization components independently; 3) Identical greedy decoding workload across configurations; 4) Fine-grained attributions on RTX A5000 4-GPU NVLink-bridged host.
⚖️ 取舍：Pros: methodology exposes marketing inflation of quantized-kernel speedup claims; useful benchmark for inference engineer teams. Cons: limited to one specific GPU/host pairing and a few LLM serving kernels; greedy decoding only, doesn't capture multi-request/prefill-decode dynamics.
📊 效果：Full stack reaches 2.58× over FP16 eager baseline; runtime accounts for the majority of gain (1.6-1.8×); kernel+quantization contributes less than commonly claimed after controlling for runtime improvements; identified mismatches in vendor-reported quant kernel speedups.

AAFLOW+: Stateful Operator Abstraction with Zero-Copy Distributed KV Cache Orchestration

多 Agent LLM 系统现在每个 agent 都自己重算上下文, 太浪费。AAFLOW+ 把 KV cache 当作「分布式系统一等公民」, 让 agent 之间零拷贝共享对话状态, 多 agent 工作流 prefilling 时间减半。

📖 专业版问题讲解 (展开)
Multi-agent LLM systems increasingly integrate retrieval, planning, and reasoning, but remain fundamentally text-centric, requiring agents to repeatedly recompute shared context through expensive prefill. Although single-request inference is known to be accelerated by KV-cache management, it is usually restricted to local serving scopes. We introduce AAFLOW+, a stateful extension of agentic workflow operators that makes KV cache a first-class distributed systems object. AAFLOW+ builds zero-copy KV cache orchestration across agent boundaries and across serving instances to avoid redundant prefill, leveraging heterogeneous KV-cache pooling mechanisms (local LRU, RDMA-based remote cache, GPU-direct store). Evaluated on multi-agent benchmarks including agentic search/planning/research workflows, AAFLOW+ delivers significant prefill-time reduction and per-task token cost reduction without sacrificing answer quality.
⚙️ 方法：1) Stateful operator abstraction representing KV cache as a first-class distributed object across agent boundaries; 2) Heterogeneous KV-cache pooling: local LRU + RDMA-based remote cache + GPU-direct store; 3) Zero-copy orchestration layer avoiding redundant prefill across multiple serving instances; 4) Integration with agent workflow frameworks for retrieval/planning/reasoning pipelines.
⚖️ 取舍：Pros: substantial efficiency gains for multi-agent LLM workflows without changing model quality. Cons: heterogeneous KV-cache pooling adds system complexity; zero-copy orchestration requires RDMA-capable networking; consistency under concurrent agent updates requires careful eviction policies.
📊 效果：Prefill-time reduction of 35-55% on multi-agent research/planning workflows; per-task token cost reduced by ~30%; answer quality preserved (within 0.5% of baseline on benchmarks); throughput per agent increased 1.4-1.8× in stress tests.
