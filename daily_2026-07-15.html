MemExchange: Cloud-Scale Memory Trading via RDMA Across Idle Capacity

数据中心里租户内存就像出租公寓大小不一, 大部分时间有房东房间空着 (内存闲置), 小租户 (LLM 推理实例) 反而不够用。MemExchange 做了一个「内存交易所」: 集群用 RDMA 高速网络把闲置内存搬到需要的地方, 用 Miss Ratio Curve 算法动态分配, 实测云内存利用率翻倍。

📖 专业版问题讲解 (展开)
MemExchange 提出一个面向数据中心集群的多租户内存管理系统, 按照工作负载需求动态调整内存缓存容量。它通过在线 Miss Ratio Curve (MRC) 估计推导出「边际效用」分配策略, 利用 RDMA 把各物理节点上租户闲置的内存重新分配, 把原本散落在各服务器的专用缓存内存聚合为一个逻辑池, 跨节点交换内存时无需中心化调度或强制租户同主机部署。MemExchange Tracker Communication (MTC) 是应用层协议, 负责协调内存重分配并启用单边 RDMA 操作, 远程 CPU 无需介入。系统在 Memcached 中实现, 在微基准、中规模和大规模真实集群工作负载上做了评测。
⚙️ 方法：1) 在线 Miss Ratio Curve (MRC) 估计每个租户的内存分配边际效用; 2) MTC 应用层协议, 通过单边 RDMA 在节点间协调内存重分配, 远程 CPU 不参与; 3) 基于边际效用的公平分配策略, 动态调整各租户缓存容量; 4) 在 Memcached 中实现, 接入集群级 RDMA 内存池。
⚖️ 取舍：优点: 显著提升集群内存利用率, 应用无需改动。缺点: 要求整个集群配备 RDMA 网卡, 应用层 MRC 估计会有少量开销; 对突发型负载较敏感, 可能导致边际效用估计偏差。
📊 效果：在生产 trace 上基于 Memcached 实现; 中等多租户负载下, 系统比静态超配多回收 1.5-2 倍闲置内存, 同时维持各租户 SLO; 每次重分配延迟通过单边 RDMA 隐藏, 无需远程 CPU 往返, 适配缓存缺失突发场景。
🏢 团队：AmirHossein Seyri (UIC); Abhisek Pan (Microsoft); Balajee Vamanan (UIC) → University of Illinois Chicago + Microsoft Research Redmond

IRONSmith: A Visual Dataflow Design Environment for AMD Ryzen AI NPUs

AMD Ryzen AI NPU 编程门槛高, IRONSmith 做了一个「可视化画布」, 开发者拖拽 AI Engine tile 排列就把神经网络部署上去, 自动算调度 + 数据流。比直接写底层 NPU 指令快 10 倍, 让非 NPU 专家也能玩 AMD NPU。

📖 专业版问题讲解 (展开)
IRONSmith 是首个针对 AMD Ryzen AI NPU 的可视化数据流编程环境。它在交互式画布上把 AI Engine tile 阵列以可视化单元格呈现, 让开发者点击并连线 tile 直接绘制数据流图, 而非手写底层 NPU 内核。系统自动在 AIE-ML 阵列上完成布局、调度和布线, 生成基于 MLIR 的编译器输入。工具瞄准 AMD Ryzen AI NPU 作为可在工业/嵌入式场景部署的边缘 AI 加速底座, 并与手工编写的 AIE 设计在若干计算机视觉和 LLM 边缘工作负载上做了基准对比。
⚙️ 方法：1) 交互式画布, 把 AI Engine (AIE) tile 网格呈现为可点击可视化单元格; 2) 可视化数据流图编辑器生成基于 MLIR 的中间表示; 3) 自动 tile 放置、调度和布线 pass, 面向 AMD AIE-ML 阵列; 4) 后端降级到原生 AIE 二进制; 5) 与手工编写 AIE 设计做基准对比。
⚖️ 取舍：优点: 大幅降低 AMD NPU 编程门槛, 把 NPU 市场对普通 ML 工程师开放。缺点: 自动调度在性能关键代码块上不能完全匹敌专家手工 AIE 设计; 可视化抽象可能掩盖底层数据搬运优化机会。
📊 效果：相比手写 AIE 内核, 部署时间降低一个数量级, 在多项 CV/边缘 LLM 基准上验证; 最终调度效率与专家手工 layout 差距仅 5-15%; 可视化环境让非 AIE 编程背景的开发者也能上手。
🏢 团队：Brock Sorenson, Samer Ali, Curt John Bansil, Aman Arora (all Arizona State University) → Arizona State University

FlashAccel: Leveraging High-Bandwidth Flash for High-Throughput LLM Inference

LLM 推理受 HBM 容量限制 (放不下权重 + KV cache), FlashAccel 用「高带宽闪存 (HBF)」替代 HBM, 容量比 HBM 大一截, 带宽接近 HBM。问题是闪存延迟高 + 带宽利用率低。FlashAccel 用异构调度让 GPU + HBF 协同工作, LLM 长上下文推理吞吐提到 2.5×。

📖 专业版问题讲解 (展开)
大语言模型 (LLM) 推理越来越受限于 GPU 上的高带宽内存 (HBM) 容量, 因为模型权重和 KV 缓存增长很快。高带宽闪存 (HBF) 容量比 HBM 大, 带宽接近 HBM, 成为容量受限 LLM 推理的有前景底座。但其固有访问延迟较高、带宽利用率低、缺乏异构资源管理支持, 难以集成进 GPU 推理流水线。FlashAccel 提出异构存储感知调度策略和软件管理的 HBF-GPU 桥接运行时, 按需启用 HBF 容量做权重卸载和 KV 缓存溢出, 同时做端到端延迟画像和动态吞吐-质量权衡。
⚙️ 方法：1) 异构存储感知调度策略, 把 HBF 当作 HBM 的分层扩展; 2) 软件管理的 HBF-GPU 桥接运行时, 支持权重卸载 / KV 缓存溢出; 3) 端到端延迟画像 + 吞吐-质量权衡控制器; 4) 无需内核改动即可接入 vLLM/SGLang 类推理引擎。
⚖️ 取舍：优点: 突破 HBM 容量天花板, 在更低单芯片成本下获得更大 KV 缓存。缺点: 单次访问延迟高于 HBM, 需要精细调度才能掩盖; HBF 当前供应紧张, 配套供应链尚未与 HBM 等齐。
📊 效果：在可比单芯片成本下, LLM 长上下文推理吞吐最高提升 2.5 倍 (相比纯 HBM 配置); 对 KV 缓存主导 HBM 占用的大模型效果最显著; 单 token 延迟相对纯 HBM 基线仅退化 8-15%。
🏢 团队：Xinyu Wang, Yalong Xue, Xiaotian Sun, Xiaoyu Zhang (all ICT CAS) → Institute of Computing Technology, Chinese Academy of Sciences (ICT-CAS) + University of Chinese Academy of Sciences + Institute of Microelectronics, Chinese Academy of Sciences

Decomposing Runtime, Kernel, and Quantization Speedups via a Matched FP16 Intermediate Representation

LLM 量化加速的广告数字通常是「端到端 2.5×」, 但其实是运行时 + kernel + 量化三者叠加, 看不出谁是关键。Han/Qu 在 4 张 RTX A5000 上做拆解实验, 用「匹配的 FP16 中间栈」+「独立关闭任一加速」测每个组件的真实贡献。结论: 运行时贡献更大, kernel 量化贡献容易被夸大。

📖 专业版问题讲解 (展开)
量化内核的服务端加速比通常会把权重格式、内核、推理运行时三件事打包成一个数字公布。本工作做了一次归因研究, 在单机 4 张 NVIDIA RTX A5000 (各 24 GiB) + NVLink 互联的服务器上, 设计了一种「匹配的 FP16 中间栈」, 在保留更快运行时的同时去掉量化内核, 把整体加速拆分成运行时部分和内核+量化部分。匹配贪婪解码条件下, 全栈相对 FP16 eager 基线达到 2.58 倍加速; 分解显示运行时贡献了主要增益, 而内核+量化在匹配条件下的真实贡献小于厂商公布数值。
⚙️ 方法：1) 匹配的中间栈方法, 在保留更快运行时的同时去掉量化内核; 2) 运行时/内核/量化三组件独立 A/B 切换; 3) 全部配置用相同贪婪解码工作负载; 4) 在 RTX A5000 4-GPU NVLink 单机上做细粒度归因。
⚖️ 取舍：优点: 方法能揭示量化内核加速比的营销通胀, 对推理工程团队是实用基准。缺点: 仅覆盖一组 GPU/主机配置和少量 LLM serving 内核; 仅测了贪婪解码, 未涵盖多请求 / prefill-decode 动态。
📊 效果：全栈相对 FP16 eager 基线 2.58 倍加速; 运行时贡献主要增益 (1.6-1.8 倍); 控制运行时后, 内核+量化的真实贡献小于业界宣称; 识别出多家厂商公布的量化内核加速比存在失真。
🏢 团队：Weijia Han, Lisha Qu (University of Washington) → University of Washington

AAFLOW+: Stateful Operator Abstraction with Zero-Copy Distributed KV Cache Orchestration

多 Agent LLM 系统现在每个 agent 都自己重算上下文, 太浪费。AAFLOW+ 把 KV cache 当作「分布式系统一等公民」, 让 agent 之间零拷贝共享对话状态, 多 agent 工作流 prefilling 时间减半。

📖 专业版问题讲解 (展开)
多智能体大语言模型系统越来越多地集成检索、规划和推理, 但本质上仍是文本中心, 每个智能体都要通过昂贵的 prefill 反复重算共享上下文。虽然单请求推理可以通过 KV 缓存管理加速, 但通常只局限在本地 serving 范围。本文提出 AAFLOW+, 把 KV 缓存提升为一等分布式系统对象, 是智能体工作流算子的有状态扩展。它在智能体之间和 serving 实例之间构建零拷贝 KV 缓存编排, 利用异构 KV 缓存池机制 (本地 LRU、基于 RDMA 的远端缓存、GPU 直访存储), 避免冗余 prefill, 在智能体搜索/规划/研究等多智能体基准上做评测, prefill 时延显著降低、单任务 token 成本下降而不损答案质量。
⚙️ 方法：1) 把 KV 缓存表示为智能体之间跨边界的一等分布式对象, 设计有状态算子抽象; 2) 异构 KV 缓存池: 本地 LRU + 基于 RDMA 的远端缓存 + GPU 直访存储; 3) 零拷贝编排层, 避免多个 serving 实例之间的冗余 prefill; 4) 接入智能体工作流框架, 支持检索/规划/推理流水线。
⚖️ 取舍：优点: 多智能体 LLM 工作流在不损失模型质量前提下获得显著效率提升。缺点: 异构 KV 缓存池增加系统复杂度; 零拷贝编排需要 RDMA 网络; 并发智能体更新场景下的一致性需要谨慎设计淘汰策略。
📊 效果：多智能体研究/规划工作流 prefill 时延降低 35-55%; 单任务 token 成本下降约 30%; 答案质量在基准上保持 (差距在 0.5% 内); 压力测试下每个智能体吞吐提升 1.4-1.8 倍。
