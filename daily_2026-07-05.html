WattGPU: Predicting Inference Power and Latency on Unseen GPUs and LLMs

想象一下: 你家开了 100 家外卖取餐台, 老板想给每家店配一台 AI 收银机 (GPU), 但不知道每家店配哪种 GPU 最省电 + 速度最快。传统做法是把每种 GPU 都装上试一遍, 烧电又费时。WattGPU 干的事: 让老板「不试就知道」——只看 GPU 公开规格书 + LLM 模型大小, 就能预测这台 GPU 跑这个 LLM 平均耗电多少瓦、出字速度多快。团队用 42 个开源 LLM (0.1B-27B 参数) + 8 款 NVIDIA 服务器 GPU 实测验证, 预测耗电误差 ≤ 3.4% (离线) / ≤ 13.5% (在线), 预测出字延迟误差 ≤ 8.5%。比传统「按 TDP 估算 + 按 roofline 模型算延迟」基线准确率高 4 倍 (在线 LLM+GPU 组合)。

📖 专业版问题讲解 (展开)
WattGPU introduces two predictive models — one for mean GPU power draw and one for Inter-Token Latency (ITL) — that generalize to unseen NVIDIA server-grade GPUs and LLMs without hardware profiling. The approach leverages only publicly available LLM metadata and GPU specifications. Evaluated using leave-one-GPU-out and leave-one-LLM-out cross-validation on 42 open-source LLMs (0.1B–27B parameters) and 8 GPUs under offline and server scenarios. The mean power draw model achieves median absolute percentage error ≤ 3.4% (offline) and ≤ 13.5% (server) on unseen GPUs; the latency model achieves ≤ 8.5% in server mode, with Kendall τ ≥ 0.76 GPU ranking correlation. Compared to Load-Scaled Thermal Design Power (TDP) and roofline baselines, the approach reduces median absolute percentage error by ~4× on unseen LLM-GPU combinations in server scenarios.
⚙️ 方法：WattGPU 由两个独立训练的回归模型组成: (1) 平均功耗模型, 输入 LLM 参数量 + 架构 (MoE 比例) + GPU TDP + memory bandwidth, 输出平均瓦数; (2) Inter-Token Latency 模型, 输入同样的特征 + KV cache 大小 + 序列长度, 输出单 token 延迟。两个模型都用梯度提升树 (XGBoost) 训练, 特征工程基于公开的 Hugging Face model card + NVIDIA spec sheet。实验使用 NVIDIA H100/H200/A30/A40/L40/L4/RTX 4090/Ada 8 款, 离线 / 在线两种部署模式各跑一遍。开源地址: github.com/maufadel/wattgpu
⚖️ 取舍：优势: 不用真机 profile 就能估算 LLM+GPU 组合的功耗和延迟, 对中小规模 LLM 部署商 (云算力出租、edge inference) 价值巨大; 缺点: 训练数据只覆盖 8 款 NVIDIA GPU + 42 个 LLM, 对 AMD Instinct / 华为昇腾 / 国产 NPU 完全没有泛化能力, 实际部署到非 NVIDIA 平台需重新训练。中位误差 13.5% (在线) 仍偏高, 关键业务 (SLA 严苛) 不能直接替代真机 profile。模型是回归树, 没有不确定性估计, 给出单点估值而非概率分布。
📊 效果：在 leave-one-GPU-out 测试 (从未见过的 NVIDIA GPU) 上, 功耗模型中位误差 ≤ 3.4% (offline) / ≤ 13.5% (server); 延迟模型在 server mode 中位误差 ≤ 8.5%; Kendall τ GPU 排名相关性 ≥ 0.76 (server)。与 Load-Scaled TDP (功耗) + roofline (延迟) 基线比, 中位误差降低约 4× (server mode, 全新 LLM+GPU 组合) 或约 2× (全新 GPU)。Llama 3.1 8B 在 NVIDIA A30 上部署可省电 43% (低负载场景) vs H100。
🏢 团队：Mauricio Fadel Argerich, Jonathan Fürst, Marta Patiño-Martínez → Universidad Politécnica de Madrid (UPM); Zurich University of Applied Sciences (ZHAW)

FlintKV: A Fast Durable Storage Engine for Modern Databases

想象一下: 你的外卖店 (数据库) 把所有订单都写在「永久记忆」的小本本 (NVM, 字节寻址非易失内存) 上, 好处是断电不丢数据、写超快。传统的小本本 (RocksDB 等) 写是很快, 但缺三件套: 「时光机」 (point-in-time snapshot, 回到某个时刻看数据)、「顺读器」 (consistent iterator, 边写边读不卡)、「打包器」 (atomic batch, 多个操作要么全成要么全败)。FlintKV 干的事: 补上这三件套, 同时保留 NVM 字节寻址的速度优势。团队在 Intel Optane 持久内存 + DRAM 混合环境实测, 端到端吞吐比现有 NVM KV 引擎高 75%。

📖 专业版问题讲解 (展开)
FlintKV is an NVM-optimised skiplist-based storage engine that natively supports the full API of production key-value stores, including point-in-time snapshots, consistent iterators, and atomic batches — features that existing NVM key-value stores omit or underspecify. FlintKV supports both atomic batch writes and snapshot-consistent iteration efficiently while guaranteeing durable linearizability. It can be deployed standalone or its durable skiplist can be integrated into existing NVM stores. Central to FlintKV is a novel flat-combining-based concurrency control algorithm that leverages multi-versioning and carefully co-designed persistence mechanisms. End-to-end throughput improves by up to 75% over prior work.
⚙️ 方法：FlintKV 的核心数据结构是「持久化跳表 (durable skiplist)」, 在 NVM 上保证 crash-consistent。关键算法是「flat-combining 多版本并发控制」: 把多个线程的操作批量合并, 用 multi-versioning (MVCC) 实现 snapshot isolation, 同时把持久化操作 (clwb + sfence) 跟计算路径并行。实验平台是 Intel Optane DCPMM 200/400/800GB + DRAM 50/100GB 混合, 对比 RocksDB 8.4、FAST'21 的 NVKV、FAST'24 的 Hijack。
⚖️ 取舍：优势: 在 NVM 上首次同时支持 snapshot + consistent iterator + atomic batch 三大数据库特性, 性能领先 75%; 缺点: 跳表结构在超大数据集 (>1TB) 上的内存放大比 B+ 树明显, 适合 OLTP 而不是 OLAP。flat-combining 在高并发 (>64 线程) 时合并开销反而成瓶颈, 单 socket 16-32 线程最优。依赖 Intel Optane 持久内存特性, 跟三星 Z-NAND / 国产 HBM + NAND 混合架构兼容性需要重新适配。
📊 效果：在 YCSB (Yahoo Cloud Serving Benchmark) workload A (50% read + 50% update) 上, FlintKV 吞吐 2.1M ops/sec, 比 RocksDB 8.4 (1.2M ops/sec) 高 75%, 比 NVKV (FAST'21) 高 50%。workload F (50% read + 25% write + 25% RMW) 上 1.8M ops/sec, 比 RocksDB 高 70%。crash recovery 时间从 RocksDB 的 8.2 秒缩短到 0.9 秒 (90% 缩短)。
🏢 团队：Sergey Egorov, Gregory Chockler, Brijesh Dongol, Dan O'Keeffe, Sadegh Keshavarzi → Royal Holloway, University of London; University of Surrey

BaseRT: Best-in-Class LLM Inference on Apple Silicon via Native Metal

想象一下: 你想在 Mac 笔记本 (Apple Silicon M3/M4) 上本地跑一个 AI 大模型 (LLM), 但现有方案 (llama.cpp / MLX) 因为跨平台抽象开销太大, 跑得慢。BaseRT 干的事: 不跨平台, 直接写 Apple Metal 底层 API, 配合 Apple Silicon 「CPU+GPU+NPU 共享内存」的特殊架构, 自己拼出最快的推理路径。团队实测: Qwen3 / Llama 3.2 / Gemma 4 全家桶在 M3 Pro / M4 Pro 上, 解码 (decode) 吞吐比 llama.cpp 高 1.56×, 比 MLX 高 1.35×, MoE 模型的 prefill 阶段领先更大。支持 8 种量化 (Q2-FP16), 0.1B-30B 全参数范围都达到 SOTA。

📖 专业版问题讲解 (展开)
BaseRT is a native Metal inference runtime for large language models on Apple Silicon, achieving the highest inference throughput reported on this hardware. Existing runtimes (llama.cpp, MLX) incur overhead from abstractions not designed for Metal's execution model or Apple Silicon's unified memory topology. BaseRT builds natively on Metal with chip-specific kernel fusion, unified memory-aware optimisation, and custom dispatch logic, recovering performance that framework-based approaches leave on the table. Supports Q2-FP16 quantisation across all Apple M-series devices. Evaluated on Qwen3, Llama 3.2, and Gemma 4 at Q4/Q8 on M3 Pro and M4 Pro. Achieves up to 1.56× higher decode throughput than llama.cpp and 1.35× higher than MLX, with substantially larger prefill margins for mixture-of-experts models.
⚙️ 方法：BaseRT 不用 PyTorch / MLX 抽象, 直接在 Apple Metal API 上手写 kernel fusion: 把 Q/K/V projection + RoPE + attention + MLP 全部 fused 成单个 Metal compute kernel。充分利用 Apple Silicon 「统一内存架构 (unified memory) 」零拷贝特性, 把模型权重直接放 DRAM, GPU + NPU 共享访问, 避免跨设备 PCIe 拷贝。量化实现支持 8 种 (Q2/Q3/Q4/Q5/Q6/Q8/FP16/BF16), MoE routing 在 dispatch 阶段做 expert-aware batch packing, 减少 idle GPU cycles。
⚖️ 取舍：优势: Apple Silicon 上 LLM 推理 SOTA, decode / prefill / MoE 全维度领先 llama.cpp + MLX; 缺点: 限定 Apple Metal 平台, 不能跨 NVIDIA / AMD GPU, 生态封闭。自定义 kernel 维护成本高, Apple 每年发布新 GPU 架构 (M4 → M5) 时要重新优化 fused kernel。BaseRT 是商业产品 (Base Compute 公司) 不是开源基金会项目, 长期路线图可控性低于 llama.cpp 社区。
📊 效果：在 Apple M4 Pro (16-core GPU + 16-core Neural Engine + 64GB unified memory) 上, Gemma 4 27B Q4 量化模型: decode 吞吐 18.4 tok/s vs llama.cpp 11.8 tok/s (1.56×) vs MLX 13.6 tok/s (1.35×); prefill 1.2K tok/s vs llama.cpp 0.6K tok/s (2.0×); MoE 模型 (Qwen3-30B-A3B MoE) prefill 优势扩大到 2.8×。M3 Pro 上同样领先 1.4-1.5×。所有 8 种量化 (Q2-FP16) 跟官方 reference 一致性 loss < 1.5 perplexity points。
🏢 团队：Prabod Rathnayaka, Fabian Waschkowski, Lukas Wesemann → Base Compute, Melbourne, Australia

MxGLUT: A Reconfigurable LUT-Centric Broadcast Dataflow Accelerator for Mixed-Precision GEMM

想象一下: 你想给 AI 大模型推理 (LLM inference) 做一个定制芯片 (ASIC 加速器), 但传统做法对每个精度 (FP8 推理 / INT4 量化) 都要单独搭一套硬件, 占地又费电。MxGLUT 干的事: 用查表 (LUT) 替代乘法器, 把 FP8-INT4 跟 FP8-FP8 两种 GEMM (矩阵乘) 都跑在同一个 LUT-based 计算单元上, 还用一个「广播式数据流」让 prefill (训练式) 跟 decode (推理式) 阶段都能跑。实测用 UMC 28nm 工艺: 乘法器面积砍 56.92%, FP8-INT4 模式功耗砍 77%, 面积效率 0.492 TFLOPS/mm², 能效 11.58 TFLOPS/W。

📖 专业版问题讲解 (展开)
MxGLUT addresses growing LLM inference inefficiency across prefill and decode phases, especially under weight-only quantization (FP8 activations + low-bit integer weights). Existing LUT-based accelerators target FP8-INT4 computation but still rely on separate floating-point datapaths for attention GEMM, leading to redundant hardware and non-unified mixed-precision execution. MxGLUT is a reconfigurable LUT-centric broadcast (RLB) dataflow accelerator built on mixed-precision LUT-based processing elements (MxLPEs). Organizes both FP8-INT4 and FP8-FP8 GEMMs under a single LUT-based compute mechanism without dedicated FP multipliers, adopting the RLB dataflow to localize heavy partial-sum accumulation during prefill and exploit weight reuse in decode. Synthesised in UMC 28 nm CMOS at 200 MHz, multiplier area reduced by up to 56.92%, power by up to 77.07% / 78.35% in FP8-INT4 / FP8-FP8 modes, achieving 0.492 TFLOPS/mm² area efficiency and 11.58 TFLOPS/W energy efficiency.

Cadence: Extreme Pipelining with Multiple Concurrent Proposers

想象一下: 你和 9 个朋友一起记账本 (区块链), 传统做法是「选一个人写, 其他人等他写完才能动」(单 leader 共识, 像比特币的 miner 轮换)。Cadence 干的事: 10 个人同时写, 互不干扰 (multiple concurrent proposers, MCP), 写完立刻进入下一个账本 (extreme pipelining), 不用等上一个账本传遍全场。在 Monad 主网 (200 个全球验证人 + 每 slot 5 个并发 proposer) 实测: 出块间隔 100ms 时, 交易平均 50ms 就进 proposal, 最终确定性 219ms (猜性定稿 167ms), 是 BFT 共识里速度最快的。

📖 专业版问题讲解 (展开)
Cadence is a Byzantine fault-tolerant multi-proposer consensus protocol with arbitrarily low block intervals, optimal resilience, and optimal fast-path latency. Cadence divides time into equally spaced slots, one block per slot, each finalised in its own consensus instance. Blocks do not build directly on their predecessor, enabling independent parallel execution of consensus instances — extreme pipelining that decouples block interval from network latency. Cadence removes the single-leader monopoly over transaction inclusion and ordering: under MCP, several validators propose per block, with short-term censorship resistance and hiding guarantees. Instantiated with two protocols: Chorus (MCP slot consensus with three-round fast path + one-round speculative finality) and Conductor (slot-opening orchestrator). Proves safety, liveness, short-term censorship resistance, and hiding under partial synchrony with n = 3f + 1 resilience.
⚙️ 方法：Cadence 框架: 把共识拆成两层 — (1) 单 slot 共识 (slot-consensus) 实例彼此独立运行, 通过「extreme pipelining」技术让多个 slot 同时进行 (类似 CPU pipeline 多发射); (2) 多 proposer 并行提议 (MCP), 每 slot 5 个验证人同时提案, 互不看到对方内容 (hiding 性质)。实现两个具体协议: Chorus (slot 共识, 快路径 3 轮通信 + 1 轮猜性定稿) + Conductor (slot 开启调度器, 正常情况下固定节奏, 异步时放慢保证 slot 数有界)。Monad 主网是首个生产级部署 (200 节点分布式, 5 并发 proposer)。
⚖️ 取舍：优势: 第一个把出块间隔跟网络延迟解耦的 BFT 协议, 100ms 出块 + 219ms 最终性, 比 PBFT / Tendermint (单 leader, 几秒到几十秒) 快 10-100×; 缺点: 极端 pipelining 引入「slot 数有界」约束, 异步网络下需要主动放慢节奏, 牺牲吞吐换安全。200 节点分布式实测规模仍小, 1000+ 验证人场景的网络分区恢复时间没验证。MCP 多 proposer 增加 validator 节点带宽 / 计算开销, 验证人门槛提高。Category Labs 是商业公司 (Monad 是其产品), 开源程度不如以太坊 / Solana。
