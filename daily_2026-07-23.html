AMD与Anthropic达成战略合作：未来部署2吉瓦AMD Instinct MI450系列GPU，AMD同时向Anthropic追加最高50亿美元股权投资

AMD 7/22 (美东时间上午 9 点) 发布公告, 跟 Anthropic 达成战略合作: Anthropic 将在 AMD Helios 机架级方案上批量部署 AMD Instinct MI450 系列 GPU (含 MI455X), 总规模最高 2 吉瓦 (GW), 首批 1 GW 部署将在 2027 年上半年启动。Helios 一体机整合 MI455X GPU + EPYC "Venice" CPU + Pensando 网络 + ROCm 软件 + 液冷, 专攻下一代 AI 训练与推理。同时双方启动多年工程协作: 用 Claude 优化 AMD Instinct 的 ROCm 软件开发, AMD 集团广泛采用 Claude 替换工程与产品研发流程; AMD 承诺对 Anthropic 战略股权投资最高 50 亿美元 (USD 5 billion)。Lisa Su (AMD 董事长兼 CEO) 与 Tom Brown (Anthropic 联合创始人兼首席算力官) 共同出席签约。这是继 AMD MI355X 之后 Anthropic 第一次上到 Helios 千兆瓦 (gigawatt) 体量, 也是 AMD 第一次把 Helios 平台开放给 Frontier 模型公司跑主流量产训练/推理负载。

💡 AMD 拿下 Anthropic 是 2026 年算力供需格局最重要的「再平衡」事件之一。Anthropic 之前主要跑 NVIDIA H100/H200, 这次双轨部署等同把「NVIDIA 主供」改成「NVIDIA + AMD 双轨」。AMI Helios MI455X + EPYC Venice + Pensando + ROCm 等同 AMD 第一次把 rack-scale 整套参考设计 (类似 NVIDIA HGX/DGX) 押到 Frontier 模型公司的主训练/推理路径, 而不是只跑 fine-tune 或推理 offload。对 AI Infra 行业的连锁反应: (1) NVIDIA 在头部模型公司的「独家供货」格局首次被实质性松动; (2) ROCm 软件栈长期偏弱的短板, 现在 Anthropic 用 Claude 反向给 AMD 加速补齐, 等同「AMD 出算力 + Anthropic 出代码」双向交换; (3) AMD 在 2nm/MI400 阵营开始形成真正的 frontier 客户反馈循环, 跟 NVIDIA Rubin 平台的代差可能从 12-18 个月缩到 6-9 个月; (4) AMD 50 亿美元股权投资 Anthropic = 大客户交叉持股新范式, OpenAI×NVIDIA 投资 / NVIDIA×H100 长期协议之后的第三种「算力换股权」模板, 后续 Cerebras / Tenstorrent / Groq 都有可能跟进类似动作。

Keeping the Cache Warm Pays: Keepalive Economics for Agentic Workloads

想象一下你点了一杯奶茶, 店家先帮你摇好, 这杯奶茶的「半成品状态」(对应大模型里的 "前缀 KV 缓存") 会被店家留着 5 分钟, 等你加完配料 (agent 调用工具) 回来续杯, 店家能从原杯里加, 价格直接打 1 折、速度也快 5 倍。但如果你离开 30 分钟, 店家直接倒掉那杯, 回来续杯又得重新付钱摇一杯新的。Anthropic、OpenAI、Google、DeepSeek 这四家大模型提供商都有类似的「前缀 KV 缓存」机制, 默认 5 分钟过期。但 AI agent 的工作场景里, agent 调一个外部工具 (跑测试、等审批) 经常要几分钟, 缓存就会被 evict 浪费。本文提出客户端 keepalive 策略, 在后台定时 ping 一下这个前缀, 把 token 留在缓存 TTL 之内, 单个 agent 的 token 成本最多能降到 1/12.5, 同时把压力回传给大模型提供商, 倒逼他们未来 8-12 个月内可能改成「按缓存驻留时长」计费。

📖 专业版问题讲解 (展开)
前沿大模型提供商普遍启用「前缀 KV 缓存」(Anthropic/OpenAI/Google/DeepSeek 默认 TTL 5 分钟), 把带相同前缀的续请求成本压到约 10%、跳过 prefill (预填充) 阶段。Agent 工作流的痛点是 "思考-行动-等待" 循环: agent 发请求 → 跑工具或等审批 → 几分钟后再回来发后续请求, 这段时间缓存大概率已被 evict, 等于要重付 prefill 成本。本文对四家前沿大模型做了系统的实测, 推导出 keepalive 间隔的最优选择是「最大间隔但仍短于 TTL」(对 Anthropic 5 分钟 TTL 来说约 4 分钟, 而不是常见 30 秒), 给出盈亏平衡 (break-even) 公式 (≈ TTL × (写/读比 - 1)), 并指出 keepalive 普遍采用带来一个负外部性: 因为缓存驻留按读次而不是按 token-小时计费, 一个 agent 把缓存打满会让 LRU (最近最少使用) 驱逐策略无内容可排, 倒逼提供商必须显式改成「按 token-小时」的缓存驻留计费才能维持成本回收, 8-12 个月内 Anthropic/OpenAI/Google 极可能推出新的缓存计费 SKU。
⚙️ 方法：作者对四家前沿大模型 (Anthropic / OpenAI / Google / DeepSeek) 的真实 API 接口端点跑受控实验, 把 keepalive 间隔变量从 30 秒到 30 分钟分了 7 档跑, 测出「续请求首 token 延迟」和「单请求有效 token 单价」两项指标, 在 Anthropic 的 5 分钟 TTL 场景下画出 (成本下降, 延迟下降) 的二维曲线。盈亏平衡模型用排队论配合实际账单推导, 给出「空闲时间乘 (写/读比减 1)」的闭式公式, 然后对四家各代入其 TTL 和单位成本具体算。最终结果用纯文本表格和算式展示, 没有用图。
⚖️ 取舍：优点: 客户侧零基础设施改动, 12.5 倍成本下降是真金白银, 对 agent 工作流是立竿见影的可部署方案。缺点与遗留问题: (1) keepalive 打满的用户的 LRU 驱逐负外部性会破坏提供商的成本回收, 可能出现「单提供商任意租户达到 80% 缓存打满就不得不切按 token-小时计费」这种供应侧波动; (2) 四家提供商的 TTL 当前都按「对空闲时间」线性算, 但多租户、工具调用中插入冷缓存、warm-up 期间 prefill 等复杂场景下需要扩展论文模型; (3) 提供商端缓存实际已实现按读、按 token 计费, 但代价是高频热 token 的缓存驻留不单独计费, 这是经济学模型未涵盖的缺口, 后续研究要把这块纳入逐 token 核算。
📊 效果：实验数据: keepalive 间隔 = 4 分钟 (Anthropic 5 分钟 TTL 下) 时单请求成本最多下降 12.5 倍, 续请求延迟同步下降 5-8 倍; 间隔 30 秒时成本反而略高 (因为 ping 自身成本抵消了部分缓存收益), 4 分钟是最优拐点。盈亏平衡空闲时间: Anthropic ≈ 46 分钟, OpenAI / DeepSeek ≈ 36 分钟, Google ≈ 33 分钟, 意味着 agent 在这个时间窗口内不做事让缓存自然过期, 不划算 (重 prefill 比 keepalive 更贵)。作者还从博弈论模型证明「理性采用 = 普遍采用」, 因为 keepalive 收益只跟自己账户绑定, 没有纳什均衡 (博弈论均衡) 障碍, 一个用户全采用, 必然导致所有用户都采用, 反而推倒提供商「按读次计费」的旧商业模式。

InstantInfer: Enabling Fast LLM Cold Start with Communicating Finite Automata

想象一下奶茶店高峰时段: 30 多个客户同时点单, 每个单都是复杂多步骤 (调 5-6 种配料), 店员要按顺序一步步操作, 一单要 30 秒, 客户体验很差 (大模型里这种 "首 token 等待时间 TTFT" 能到 50 秒起步, 70 亿参数模型 160 秒, 3000 亿模型要 300 秒)。这是大模型推理服务的「冷启动」现象: 一开始要把模型从磁盘加载到 GPU, 初始化好推理状态, 一般要 50-300 秒。本文提出「通信有限状态机」(CFA) 的抽象, 把推理服务多个组件 (进程树创建、张量加载、模型切换) 视为一组互相通信的小「水位机」, 然后改造 vLLM 冷启动代码让组件并发执行, 不破坏正确性, 冷启动延迟最多下降 7.2 倍。

📖 专业版问题讲解 (展开)
vLLM 等大模型推理服务的冷启动延迟在 100 亿以下、700 亿、3000 亿参数模型上分别约 50 秒、160 秒、300 秒, 这是无服务器推理 (serverless, scale-to-zero 即缩容到零、spot 即抢占式实例) 场景的「准入门槛」瓶颈。本文提出「通信有限状态机」抽象, 把 vLLM 三个组件 (进程树创建、张量加载、模型切换) 描述成一组互相交换消息的有限状态机, 在 CFA 模型下推导「安全并发条件」, 然后实现一个 CFA 编程框架, 把 vLLM 的串行冷启动改造为「保留原顺序语义, 但跨组件可并发执行」的结构。理论证明和 vLLM 实测都显示, 在 100 亿/700 亿/3000 亿模型上冷启动时间最多下降 7.2 倍, 跨 GPU、跨负载、跨规模都稳定, 没有引入竞态 (race condition) 或加载顺序错误。这是 vLLM 上线以来首次系统性的「冷启动优化」工作, 也是国产推理工程团队 (北京大学 + ScitiX) 在 SGLang / vLLM 体系外另辟蹊径的一个突破口。
⚙️ 方法：方法分三步: (1) 把 vLLM 的 3 个冷启动阶段 (进程树创建、张量加载、模型切换) 用 CFA 抽象建模, 每个组件是一个确定性有限状态机, 组件之间用消息传递协调; (2) 推导「跨组件安全并发条件」, 保证「跨组件并发执行」但不破坏 vLLM 原有的顺序语义 (sequence-preserving 即顺序保持) 的充要条件; (3) 在 CFA 框架内重新组织 vLLM 冷启动代码, 用框架沙箱让「哪一步必须串行、哪一步可以并行」由 CFA 静态分析自动决定, 而非工程师手写线程锁。理论证明给出一个「安全并发定理」, 论文附件给出完整证明。
⚖️ 取舍：优点: 7.2 倍冷启动加速是无服务器推理的核心准入门槛, 落地成本低 (替换 vLLM 冷启动代码路径), 对缩容到零、抢占式、突发流量场景尤其有效; 北大 + ScitiX 国产团队首次给出「基于 CFA 框架的推理优化」学术级工作。缺点与遗留问题: (1) CFA 框架假设「组件间只交换明确消息、没有共享可变状态」, 但 vLLM 实际有些延迟初始化的状态机不符合 CFA 假设, 论文用回退走传统串行模式 (占 8% 路径); (2) 论文没有覆盖多模态、多 LoRA (低秩适配) 场景, 这些场景的冷启动比纯文本大模型更复杂 (多模型、多插件、多个分词器); (3) 实验主要在 NVIDIA GPU 上跑, AMD 和国产 GPU 适配未验证, 这是大模型推理去 NVIDIA 化大趋势下未来必经的扩展工作。
📊 效果：实验在 vLLM v0.6 上跑, 配合 Llama-70 亿、Qwen-3000 亿等模型, GPU 覆盖 A100、H100、H200 三代。核心结果分三档: (1) 冷启动延迟在 100 亿以下模型下降 4.8 倍 (10 秒对比 50 秒基准), 700 亿参数模型下降 5.5 倍 (29 秒对比 160 秒), 3000 亿参数模型下降 7.2 倍 (42 秒对比 300 秒); (2) 冷启动阶段 GPU 利用率从 18% 提升到 71%, 关键原因是张量加载和模型切换两个阶段并发跑在了 GPU 算力上, 没有让 GPU 闲置; (3) 冷热混合负载下端到端 P99 (第 99 百分位) 延迟在突发流量场景下从 138 毫秒降到 64 毫秒, 原因是冷模型被「预热缓存」而不是「重新初始化」。论文公开了 InstantInfer 工具链, vLLM 社区已开合并请求 PR。

Unstructured Hydrodynamics on Spatial Dataflow Architectures: A Joint Code and Data Decomposition Approach

Cerebras 这台晶圆级 AI 芯片 (WSE 即「晶圆规模引擎」), 把算力和内存都封装在同一块超大晶圆上, 内存带宽巨大、不受传统芯片的离芯片瓶颈限制。这种「空间数据流架构」对规则的网格计算很在行, 但对实际科学计算里常见的「不规则网格」就有问题——后者有高维通信模式 + 间接访存, 无法直接 map 到 WSE 的二维 PE (处理单元) 网格。本文用「以模型为中心」的方式, 同时拆解「代码」(子程序) 和「数据」(应用场) 两个维度, 用填充空间曲线 (类似希尔伯特曲线) 最小化通信, 同时做内存优化。结果是把美国国家实验室的 LULESH (一种不规则流体动力学基准程序) 真正能跑在 Cerebras WSE 上, 而且比传统 GPU 还快。

📖 专业版问题讲解 (展开)
空间数据流架构 (如 Cerebras WSE) 是 2024-2026 年新兴的「网状互连、固定内存的处理单元」硬件范式, 把内存局部性推到极致, 天然适合结构化网格 / 模板类 HPC (高性能计算) 算子。但现实科学计算 80% 以上 (流体、有限元、地震、分子动力学) 是不规则网格, 间接访存 + 高维通信模式让 WSE「没有处理单元网格直接对应, 无法在网状对齐的局部性上跑」。本文从「以模型为中心」视角出发, 联合分解「代码拆分 (子程序切分)」+「数据拆分 (应用场)」两个维度, 通过通信与内存联合建模推导「最小通信 + 最少冗余内存」的拆分方案, 用高维填充曲线实现高维数据切到二维处理单元网格, 并加入编译器层级的内存优化减少片上内存压力。实验: 把 LLNL 的 LULESH 基准程序整套映射到 Cerebras WSE, 速度比 NVIDIA H100 更优。
⚙️ 方法：方法分 3 个层次: (1) 通信与内存联合模型, 把「代码拆解」(子程序切分) 和「数据拆解」(应用场切分) 视为两个独立但有交叉约束的优化问题, 同时求解; (2) 用高维填充曲线 (皮亚诺 / 希尔伯特曲线) 实现「n 维应用场 → 二维处理单元网格的连续填充」, 保证局部性; (3) 加入编译器层级的内存优化, 把频繁访问的场切到片上内存, 减少跨处理单元的全局往返。整套方法作为一个「代码/数据协同切分」流程实现在 WSE 的 SDK 里, 用户用领域特定的描述 (网格 + 算子) 输入, 工具链自动切分。
⚖️ 取舍：优点: 首次提出「代码 + 数据协同分解」在空间数据流架构上的通用方法论, 直接把 80% 的真实 HPC 工作负载从「GPU 专属」拉回晶圆级阵营, 是 Cerebras WSE 在 HPC / 科学计算领域的决定性扩展; 同时把 LULESH 这种美国国家实验室级基准拉上 WSE, 等同「美国国家实验室 + 美国本土 AI 硬件厂」联手在 NVIDIA GPU 主导的科学 HPC 体系外开辟第二条路径。缺点与遗留问题: (1) 方法论假设应用的依赖图是稀疏且规则的, 对高度动态的网格自适应细化 (AMR) 类应用还需扩展; (2) 编译器对「场切分」的自动推导目前只能处理最多 4 维空间, 多于 4 维 (例如 6 维的粒子云模拟方法) 仍需手动指定; (3) 实测只在 Cerebras CS-3 上跑通, 是否能移植到其他空间数据流架构 (如 SambaNova RDU) 未验证, 涉及「以模型为中心分解」是否其他硬件同样适用。
📊 效果：实验在 Cerebras CS-3 (WSE-3, 90 万个处理单元) 上跑 LULESH (一个横跨 7 个数量级的基准, 用来检验 LLNL 武器模拟 + 星球实验室天体物理代码), 网格规模从 32 立方跑到 1024 立方。关键结果: (1) 256 立方的网格上 Cerebras WSE 跑 LULESH 比 NVIDIA H100 单节点快 2.3 倍, 关键原因是在 WSE 上通信模式避开了离芯片网络; (2) 第一份「不规则网格基准真正在晶圆级上跑通」的工作, 之前所有 WSE 上的不规则代码都是「应用特定移植」, 本文首次给出「通用方法论」; (3) 编译器重新生成的访存模式让 LULESH 关键算子 (能量 + 应力 + 沙漏控制) 在 512 立方网格上 WSE 加速比达到 3.7 倍 (对比 H100), 同时 WSE 自身的「每处理单元内存」瓶颈没触发 (整个网格全部驻留在处理单元本地)。

What Governs Decode Throughput in Absolute-Offset GPU LZ77? A Work-Granularity Mechanism and Hardware Implications

在 GPU 上做 LZ77 解压 (一种常见的数据压缩算法), 经典做法是一个 warp (32 个 GPU 线程) 一起解一段压缩流。但很多「匹配段」都很短 (比如 6.5 字节), 这时 32 个 lane 里大部分闲置——一个 8 字节的匹配段用 32 lane 去做, 只有 8 lane 在干活, 24 lane 在「等同事」。本文在 NVIDIA H100 上做受控实验, 发现真正控制 GPU 解压吞吐量的不是「占用率」、「计算量」、「地址分散」、「launch 并行度」, 而是「工作粒度」——具体说就是「平均匹配长度」。基于此发现, 作者改「编码端」的最小匹配长度 (按距离类别区分), 让每次匹配至少 12-16 字节起步, 实测到 enwik9 文本和 FASTQ 基因组数据上 GPU 解压吞吐最多升 78%, 而且压缩比也提升了 1.8%。

📖 专业版问题讲解 (展开)
本文针对 ACEAPEX 系列 GPU 原生压缩算法 (绝对偏移 LZ77 解压, 设备常驻, 亚毫秒级区域随机定位), 通过 NVIDIA H100 上的受控消融实验, 系统验证「控制解码吞吐的不是占用率 / 计算量 / 地址分散 / launch 并行度, 而是工作粒度 (平均匹配长度)」。理论给出「匹配段短 → warp 通道利用率低 → 整 warp 走最慢通道」的解释。合成复制算子实测: 平均匹配长度从 32 字节增长到 1024 字节时, 吞吐量从 212 GB/s 上升到 744 GB/s (3.5 倍跨度), 完整验证「工作粒度是主因」假设。真实数据端: enwik9 平均匹配长度 6.5 / FASTQ 10.1, 都远低于理论最优 32+ 字节, 因此 GPU 真实吞吐被严重欠利用。基于此机理, 作者提出「编码端最小匹配长度调节杠杆」: 把最小匹配长度按距离类别 6/8/10/12 字节提升到 12/16/24/32 字节, 同时实现「压缩比提升 1.8%」和「GPU 端解码吞吐提升 (enwik9 78%, FASTQ 25%)」。所有数字都做位级精确 (FNV 哈希) 验证 + git 可复现, 工程严谨度非常高。
⚙️ 方法：实验方法分两步: (1) 消融研究 — 在 NVIDIA H100 上用微基准单独控制「占用率」、「计算强度」、「地址分散」、「launch 并行度」、「工作粒度 (匹配长度)」, 每次只变一个, 看解码吞吐变化, 验证「工作粒度是单一主因」假设; (2) 编码端杠杆 — 修改 ACEAPEX 的最小匹配长度参数, 按距离类别把最小匹配由 6/8/10/12 字节提升到 12/16/24/32 字节, 在 8 个真实数据集 (enwik9 / FASTQ / DNA / 蛋白等) 上重新编码, 对比新 GPU 端吞吐 + 新压缩比。所有数字加 FNV 哈希 + 字节级 git 验证, 整套实验可复现。
⚖️ 取舍：优点: 单一机理明确 (工作粒度是 GPU 端解码主因), 给「编码端调节杠杆」, 全 8 个数据集验证, 数据严谨 (位级精确 + git 可复现), 实操价值高 (基因组数据 GPU 端解压提速 + 压缩比提升); 同时「工作粒度假设」是一个通用 GPU 算子设计经验法则, 后续 GPU LZ77 / GPU BWT (Burrows-Wheeler 变换) / GPU 字典编码都可以套用。缺点与遗留问题: (1) 单作者独立研究员工作, 实验配置覆盖度 (只 H100) 不够广, 缺乏 A100 / B200 / AMD MI300 跨硬件验证; (2) 「工作粒度主因」假设是在「设备常驻、匹配阶段」场景下验证, 「熵解码 + 主机传输」不在论文测量范围, 这两块对 GPU 端端到端性能贡献非平凡; (3) 长匹配长度在很多流式实时压缩场景不适用 (如实时流媒体 GPU 端压缩), 「编码端杠杆」在那个场景下是反优化; (4) 论文实操层是「速度测试 + 性能微调」, 没有解决「不同应用对延迟 vs 吞吐的不同偏好」这个问题, 后续如果要部署还需要根据场景切换不同最小匹配长度配置。
📊 效果：核心结果有五项: (1) 工作粒度假设验证: 消融实验显示只有匹配长度变化时, GPU 解码吞吐变化 3.5 倍 (212 GB/s 升到 744 GB/s), 其他四个变量 (占用率、计算量、地址分散、launch 并行度) 各自变化 ±10% 都没有显著效应; (2) enwik9 数据集: 最小匹配长度从 6 字节提到 12 字节, GPU 解码吞吐从 142 GB/s 升到 178 GB/s (提升 25.3%), 压缩比同时升 1.8%; (3) FASTQ 数据集: 最小匹配长度从 8 字节提到 16 字节, 解码从 142 GB/s 升到 178 GB/s, 压缩比升 2.1%; (4) 全部 8 个数据集都没有反例, 「编码端杠杆不是取舍, 是双赢」; (5) 同时给出「GPU 端设备常驻区域随机定位 0.334 毫秒 / 16 KB 块」的位级精确验证, 这是论文系列第三部工作。

A Second-Moment Theory for Floating-Point Reduction Trees

想象一下把 1024 个浮点数加在一起: 你可以两两相加 1024 → 512 → 256 → ... → 1 (平衡二叉树), 也可以按顺序 1 + 2 + 3 + ... (顺序树), 结果会略有不同——理论上数学应该相等, 但浮点数有舍入误差, 顺序不同误差也不同。这是 GPU 上做大规模求和躲不开的问题, 尤其 HBM3 / HBM4 这种 8/16/32/64 位各种精度混用时, 误差会被树形放大。本文用「二阶矩」(均方误差) 理论, 推导出在任意二叉归约树 (求和树) 下浮点求和误差的精确公式, 给出「哪种树形在哪种数据类型下最优」的明确答案: 平衡树对零均值输入最好, 顺序树适合非零均值, 几何级数排布适合零均值, 哈夫曼编码在加权方差下最优。

📖 专业版问题讲解 (展开)
本文发展二叉归约树 (即逐步两两求和的结构) 上浮点求和的二阶矩 (均方误差) 精确分析, 填补了「标准最坏情况界忽略部分和顺序」的理论空白。在常数 ν 圆整模型 (舍入前值的局部方差 = νu²x²) 下, 推导任意二叉归约树 T 的期望均方误差闭式 (closed-form, 指公式直接给出精确数值): 对独立同分布输入 (均值 μ, 方差 τ²), 期望均方误差 = τ² · Λ₁(T) + μ² · Λ₂(T), 其中 Λ₁(T) 是叶子总深度, Λ₂(T) 是内部子树大小的平方和。基于此, 给出「平衡树 / 顺序树在零均值输入下达到零均值输入极值」、「几何级数排布在零均值输入下最优」、「非零均值的阶段指数减半」、「哈夫曼编码在加权方差下最小化方差加权深度」等 4 类树形结构的最优性证明。同时把核函数扩展到矩阵乘 (通过操作数 Gram 矩阵, 即操作数自身内积组成的矩阵), 在双精度 64 位、单精度 32 位、软件仿真半精度 16 位、bfloat16 脑浮点上做实测验证: 模型对树形结构的排序全部对得上, KT 在 AR(1) 部分和代价上跟踪到一阶匹配, 矩阵乘端到端预测偏差 ≤3%。这是浮点归约领域近 30 年最重要的「树形结构优化」理论工具, 对 GPU 混合精度训练 / 推理的数值稳定性、HPC Kahan 求和库、机器学习矩阵乘算子自动调参都有直接指导价值。
⚙️ 方法：理论方法分 4 块: (1) 均方误差递推推导 — 在二叉归约树 T 上建立「条件无偏圆整」假设, 推导每一步圆整方差的精确递推; (2) 求和闭式 — 用归纳法推出「共同祖先核 K_T(i,j)」表达内部节点共祖先关系, 得到期望代价 = p^T K_T p 的代数形式; (3) 树形最优性证明 — 对平衡 / 顺序 / 几何级数排布 / 哈夫曼 4 种典型树形, 在不同输入分布下给出闭式 Λ₁ / Λ₂ 并证明极值; (4) 矩阵乘扩展 — 把单树归约核扩到矩阵乘的操作数 Gram 矩阵, 复用 K_T 核同时建模操作数精度失配。
