CrossPool: Efficient Multi-LLM Serving for Cold MoE Models through KV-Cache and Weight Disaggregation

想象一下：你开了个奶茶店, 店里有 100 种奶茶配方 (大模型的「参数」), 还有 50 个调饮台 (GPU)。每个客人来点单, 调饮师要边记配方 (权重) 边记客人加了多少糖多少冰 (KV-cache, 也就是「对话上下文」)。问题来了——配方本和点单簿都堆在同一个调饮台上, 经常互相挤占, 一忙起来调饮师手忙脚乱, 出餐变慢。这篇论文讲的就是怎么把「配方本」和「点单簿」分开放到不同的台面上, 让多个冷门配方 (很少有人点的奶茶) 共享一套配方本, 同时点单簿还能弹性扩容——结果就是客人再多也不怕, 出餐速度提升 10 倍以上.

📖 专业版问题讲解 (展开)
LLM 服务场景中, 大量稀疏激活的 MoE 模型长期处于 cold 状态——大多数模型请求稀疏, GPU 显存被静态 weights 和动态 KV-cache 共占一池. 静态权重稳定占用显存, 而 KV-cache 随请求动态伸缩; cold 模型之间峰值极少同时出现, 传统的 worst-case KV 预留方式严重浪费显存, 共享 KV-cache 池能聚合实际请求容量. 但 weights 和 KV-cache 同池时, 静态权重与动态 KV 互相挤占; 在 cold、低并发长上下文场景下, MLA / MQA 等 KV-head 压缩的 attention 算法只能暴露部分副本的 KV 容量, 实际可用上下文容量大幅缩水.
⚙️ 方法：CrossPool 把 MoE 模型的 FFN weights 和 KV-cache 分离到两个 GPU 显存池: weights 池聚合多个 cold 模型的 FFN 权重, KV-cache 池动态服务活跃请求, attention 留在 KV 池内做本地 KV 访问, FFN 在 weights 池执行 (FFN 占 MoE 参数主导). 池边界交换 hidden state 而非 KV tensor. 三个核心设计: 异构 KV-cache 规划器 + 虚拟化、跨池通信的 layer-wise pipeline scheduler (隐藏 hidden state 传输开销)、带 control lowering 的 persistent kernel (降 CPU-GPU 控制开销).
⚖️ 取舍：权重 / KV 分离增加跨池 hidden state 传输, 对 latency 敏感的小模型推理是负向; 但对冷启动、长上下文、低并发多模型场景显存利用率提升显著——同样的 GPU 显存预算能服务更多 cold 模型. 跨池通信需要 layer-wise 调度, 对调度器实现复杂度要求较高.
📊 效果：相比基于 kvcached 的多 LLM serving 方案, CrossPool 在 cold MoE 模型上 P99 TBT (Time-Between-Tokens) 最多降低 10.4 倍. 在 bursty 长上下文请求下能维持高吞吐, weights 池 + KV 池设计解决了 cold 模型显存浪费这一行业核心痛点.

Aquifer: Hierarchical Memory Pooling with CXL and RDMA for MicroVM Snapshots

你公司买了 100 台服务器, 每台 256GB 内存. 但实际用下来发现, 平均每台真正用到的内存只有 160GB——剩下 30% 的内存都「插着但没人用」. 因为 CPU 满了, 这些空内存就成了「挂着的肉」. 怎么办? 有人想: 能不能让 100 台服务器的内存「互通有无」, 谁缺谁借? 问题: 走 CXL 总线只能在 8-16 台小范围内借, 走 RDMA 网卡虽然能跨整个机房但每次借都要打「申请单」很慢. 这篇论文的方案是——把两种借法组合起来用: 借得急的 (热点内存) 走 CXL 快速通道, 借得不急的 (冷数据) 走 RDMA 慢速通道. 结果: AWS Lambda 那种「按函数调用」的冷启动速度提升 2.2 倍.

📖 专业版问题讲解 (展开)
生产云集群里 25-35% 的 DRAM 处于 stranded 状态 (内存插着但 CPU 已满订阅). 内存池化 (memory pooling) 是缓解办法, 但单靠 CXL 或 RDMA 都不够: CXL 提供低延迟 load/store 透明访问但局限于 pod 内 (8-16 socket), 缺乏跨主机硬件一致性; RDMA 集群范围可达但每次访问要 software interposition, page fault / kernel 介入开销大. 两者结合是未来方向, 但在 MicroVM serverless 场景 (snapshot restore 是 cold-start 主导瓶颈) 还未被探索. Aquifer 填补这一空白.
⚙️ 方法：Aquifer 是首个用 CXL+RDMA 分层内存池服务 MicroVM snapshot 的系统. 关键观察: snapshot 页面绝大多数是 zero page 或 cold page——hotness-based snapshot 格式消除 zero page, 只把 hot working set 放 CXL 池, cold page 放 RDMA 池. CXL 2.0 multi-headed devices 缺乏硬件 cache coherence, Aquifer 设计 ownership-based coherence protocol 保证正确性. Copy-based page serving: MicroVM resume 前预装 hot page from CXL, 异步 demand-page cold page from RDMA.
⚖️ 取舍：CXL 域限制在 8-16 socket, 跨 pod 必须走 RDMA, 路径分两层导致 cold-page 访问延迟比纯 CXL 高; 但比纯 Firecracker (无池化) 快, 比纯 RDMA 池化便宜. Hot/cold 区分需要 snapshot 分析, 对 snapshot 内页分布均匀的 workload 收益有限. Ownership coherence 协议增加一致性开销但避免全硬件 coherence 复杂度.
📊 效果：在 emulated CXL+RDMA 硬件上, Aquifer 端到端 invocation time 几何均值加速: 比 Firecracker 快 2.2 倍, 比次优方案快 1.1 倍. 解决了 Microsoft Azure / Google / Meta / 阿里 25-45% DRAM 浪费问题, 在 warehouse scale 单百分点改进可省上亿美元.
🏢 团队：Junliang Hu, Huaicheng Li, Ming-Chang Yang → The Chinese University of Hong Kong, Department of Computer Science and Engineering; Virginia Tech, Department of Computer Science

The Serialized Bridge: Understanding and Recovering LLM Serving Performance under Blackwell GPU Confidential Computing

你想把一个 LLM 部署到一台「上锁的服务器」里 (机密计算, GPU-CC)——客户的数据绝对安全, 哪怕机房管理员都看不到. 听起来很美好, 但实测下来发现: 装上这把「锁」之后, 单卡算力几乎没损失 (B300 BF16 matmul 还能跑到 0.998 倍速度), 可是 LLM 端到端推理速度掉了 13-27%, 加载 GPT-OSS-120B 这种大模型居然要等 287 秒. 问题出在哪? 不是 GPU 慢了, 而是 CPU 给 GPU 传数据的那个「桥」被锁住了, 变成单车道——以前能并行传 16 路数据, 现在只能一辆一辆过. 解决方案: 调度器加个标志位 + 加一个 worker 线程专门「放行」, 最高能恢复 92% 的性能损失.

📖 专业版问题讲解 (展开)
GPU Confidential Computing (GPU-CC) 在 GPU-local 性能上几乎零损失: NVIDIA B300 上 BF16 matmul 跑出 0.998x 非机密性能. 但端到端 LLM serving 在 Intel TDX + GPU-CC 下吞吐仍损失 13-27%, KV-cache restore 延迟翻倍. 本文在 RTX Pro 6000 和 B300 HGX 两个 Blackwell 平台上定位这个 gap 的根因——不是 GPU compute 而是 confidential VM-GPU bridge. GPU-CC 把 host/device 移动变成串行、高 setup cost 的通道, secure copies 在 CUDA context 内无法并发, 异步传输在 runtime boundary block, 小规模 crossing 还要付固定 toll.
⚙️ 方法：论文用一个机制解释全部损失: GPU-CC 下 confidential VM-GPU bridge 串行化. 三个性质定义行为: (1) context 内无并发 (16 CUDA stream 同时 32-byte device-to-host copy 仍 0 parallelism), (2) async transfer block, (3) 小 crossing 固定 toll. 实验工具: vLLM dense decode 跑 44x slower 小 alloc-and-copy 操作, 用 targeted patches 排除其他解释. 两个恢复方案: scheduling flag 恢复 57% gap, worker-thread drain 在高并发场景恢复 92%.
⚖️ 取舍：GPU-CC 启用后所有 GPU-local optimization (CUDA stream concurrency, async DMA, persistent kernel) 都被 secure bridge 的串行化抵消; 即便 GPU 算力本身几乎无损失, 端到端 inference 仍退步 13-27%. Blackwell 让 confidential tax 占比上升 (相同 vLLM stack 在 RTX Pro 6000 损失 ~10%, 在 B300 损失 ~26%). KV-restore 慢 +131%, model load 慢 34x.
📊 效果：B300 上 BF16 matmul 0.998x non-CC, 96K matmul CUDA graph 1.0012x. 但 dense LLM decode 损失 13-22%, MoE serving 损失最高 25%, KV-cache restore 延迟上升 131%, GPT-OSS-120B 加载默认 287 秒. Worker-thread drain 在 qualified 高并发场景恢复 92% gap. 同时评估了 B300 上 confidential multi-GPU NVSwitch tenants (510 GB/s NVLink P2P inside CVM, 并发 isolated tenants), 找出 fabric-attestation gap 是产品化最大拦路虎.

FP8 is All You Need (Part 2): Efficient Ozaki-Bailey FFT Through Tensor-core Garner Reformulation and Kulisch Escape Route

你买了个 NVIDIA B300 显卡准备跑科学计算 (气候模拟、分子动力学), 想用它的 FP64 (双精度浮点) 算力. 结果一看规格傻眼了——B300 的 FP64 只有 1.3 TFLOPS, 比上一代 B200 砍了 30 倍. 原因: NVIDIA 把硅片面积都让给 AI 用的 FP8/FP4 了, FP64 被边缘化. 那科学计算怎么办? 日本超算教父松岗聪 (TSUBAME/富岳之父) 给出了一个天才解法——既然 FP64 慢, 那就用 FP8 拼! 把 FP64 数字切成 8 段, 每段用 FP8 算, 最后拼回去 (Garner reconstruction). 精度还能保证比原生 FP64 还准. 这篇论文把这个招法扩展到了 3D 傅里叶变换 (气候模拟的核心算子), 还用一个叫 Kulisch 累加器的招数, 把拼装阶段的瓶颈也突破了.

📖 专业版问题讲解 (展开)
NVIDIA Blackwell Ultra (B300) 把 FP64 vector 吞吐降到约 1.3 TFLOPS/GPU, 相比 B200 砍 30 倍, 远低于 bandwidth-limited FP64 workload 保持 memory-bound 的阈值. Ozaki Scheme II (Ozaki-II) 仿真框架在 B300 上通过把 dense matrix multiply 仿真到 FP8 tensor core + mantissa-sliced Chinese-remainder reconstruction 恢复 FP64 等效吞吐. Companion paper Part (1) 已覆盖 dense GEMM、batched GEMV、structured stencils、SpMV; 本文补足第五个原语: 3D FFT, 两篇合起来构成 post-FP64 stack 的端到端 kernel 覆盖审计.
⚙️ 方法：Ozaki-Bailey FFT——通过 Bailey six-step decomposition 把 3D FFT 拆为 1D FFT GEMMs, 所有 GEMM 都跑在 FP8 tensor core. Bailey factorization 暴露一个 dense GEMM 没有的 regime: per-output Garner reconstruction cost 按 r^2/k scale (k 是 Bailey 内维度), 当 k=32 时 η-roof (三参数 TME model 的第三个 roofline) 成为 binding constraint. Garner reconstruction 拆 Phase A (FP8/INT8 tensor core 内积, 1024^3 on B300 ~1ms) 和 Phase B (per-output 归约). Kulisch fixed-point complete-arithmetic accumulator 是 Phase B 改革——维持全 FP64 精度但跑在 INT32 SIMT pipe (避开 FP64 silicon 退化).
⚖️ 取舍：Ozaki 仿真要在 FP8 tensor core 上重写原 FP64 算法, 编程模型复杂, kernel 优化工作大; 但相比 B300 砍 30x 的原生 FP64, 仿真后 ~1.36 PFLOPS FP8 throughput 让科学计算再次可行. Bailey factorization 增加 Phase B (Garner reduction) 开销, 但 Kulisch INT32 SIMT pipe 足够快 (B300 ~75 TOPS vs floor 66 TOPS, 余 14% margin). Rubin 33 TF @ 22 TB/s 才是真正的 FP64 roof, B300 离 roof 差 10 倍.
📊 效果：B300 FP64 1.3 TF vs 12.5 TF memory roof, 差 10x. Rubin 33 TF @ 22 TB/s 落在 roof 4% 内. B300 INT32 ~75 TOPS > Kulisch floor 66 TOPS, FP8 tensor ~5 PFLOPS > FP8 floor 1.36 PFLOPS by 3.7x. 1024^3 3D FFT 投影 ~18 ms wall time at full FP64 accuracy (vs naive FP64 sum 准确性更优). A-Roof analysis + Kulisch escape route 把 post-FP64 era 的带宽 parity floor 定在 O_FP64 = O_I × B_HBM ≥ 1.56 B_HBM.

Accelerating Disaggregated RL for Visual Generative LLMs with Diffusion-Based Parallelism and Trainer-Assisted Generation

你想训练一个会画画的 AI (DALL-E 那种扩散模型). 训练这种 AI 用强化学习 (RL) 时, 系统要做两件事: (1) 让 AI 生成一堆图 (rollout), (2) 给这些图打分、算梯度、更新模型 (training). 现有的训练框架把这两件事绑在同一台机器上——GPU 又要画图又要算梯度, 互相等, 浪费算力. 这篇论文的解法是: 把「画图的机器」和「评分的机器」分开 (PD 分离 / 存算分离思路), 用一种叫 diffusion-based parallelism 的方式让画图机多机协作, 再加一个 trainer 模型帮画图机纠错. 这样 AI 画画的训练速度能大幅提升, 也更容易扩展到更大的模型和数据集.

📖 专业版问题讲解 (展开)
强化学习 (RL) 已成为后训练主流范式, 推动了 veRL 等高性能 RL 系统出现, 服务自回归 LLM. 同时, diffusion-oriented RL 算法 (DanceGRPO, FlowGRPO) 把 RL 从语言推理扩展到 diffusion 视觉/流式生成. 但面向 diffusion generative LLM 的高效 RL 系统仍是空白——现有实现如 veRL-Omni 仍依赖 colocated execution, 简化了同步但把 rollout 和 training 紧耦合, 难以扩展到大模型/大 batch 场景.
⚙️ 方法：提出针对 diffusion generative LLM 的 disaggregated RL 架构, 核心创新是 diffusion-based parallelism: rollout 阶段使用 diffusion model 的并行去噪 (而非自回归), trainer-assisted generation 用单独的 trainer 模型辅助 rollout 模型生成, 降低单步决策的方差. Rollout 和 trainer 解耦到独立 worker 池, 通过异步通信 + 共享 KV-cache 复用实现高吞吐.
⚖️ 取舍：Disaggregation 增加 rollout-trainer 通信开销, 但通过 KV-cache 复用和异步流水线把通信隐藏在 diffusion 迭代时间内. Colocated 版本同步简单但无法弹性扩展, 难以支撑百亿参数 diffusion LLM 的 RL 训练. Diffusion rollout 比 autoregressive 更易并行但采样质量略低, trainer-assisted generation 弥补这一 gap.
📊 效果：在 diffusion generative LLM 的 RL 训练上, disaggregated 架构相比 colocated baseline 在同样 GPU 预算下吞吐提升显著 (具体数字见论文 benchmark), 把 RL 后训练从语言推理扩展到视觉生成, 为下一代多模态 RL 训练铺路. Huawei 提供算力支持, 哈工深 + 港科广 + 华为三方合作体现中国 RL infra 系统研究的产业纵深.
