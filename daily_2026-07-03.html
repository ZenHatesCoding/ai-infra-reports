ELDR: Expert-Locality-Aware Decode Routing for PD-Disaggregated MoE Serving

想象你开了个奶茶店, 店里用的是 MoE 配方 (mixture-of-experts, 业内叫「混合专家」): 一杯奶茶背后其实是几十个调饮专家, 路由器 (gate) 根据客人点的口味只挑 2-3 个调, 而不是把所有调饮师都叫过来。现在业界流行「PD 分离」(prefill-decode disaggregation): 点单 (prefill) 和出品 (decode) 分别在两个独立的奶茶吧台做, 一个吧台专门接单, 一个吧台专门出品。但问题来了 — 现有路由器只看哪个吧台不忙, 就把单派过去; 但 MoE 模型里不同吧台调出来的口味不一样, 同样接 10 单, 集中让擅长「果茶」的调饮师做比让擅长「奶茶」+「果茶」的混合调饮师做要快得多。ELDR 这个工作就是说: 在 PD 分离的出品环节, 路由器不光要看到吧台忙不忙, 还要看单子「口味分布」相似不相似 — 口味相近的单子尽量堆到同一个吧台去, 这样调饮师切换配方的次数少, 出杯更快。

📖 专业版问题讲解 (展开)
PD-disaggregated LLM serving splits prompt processing (prefill) and token generation (decode) onto separate worker pools. Existing decode-side routers balance only load; for mixture-of-experts (MoE) models this is insufficient because equally loaded workers can differ in latency — each decode step loads weights of every distinct expert the batch activates. We present ELDR, an expert-locality-aware decode router for PD-disaggregated MoE serving. From a request's prefill expert activations, ELDR builds an expert signature predicting the experts it will activate during generation. Offline, balanced K-means partitions signature space across decode workers; online, locality-band routing sends each request to the least-loaded worker among those best matching its signature. A signature cache, co-indexed with the KV cache at KV-block granularity, keeps signatures exact under prefix caching. Implemented in vLLM and evaluated on deployments of up to 40 GPUs, ELDR reduces median TPOT by 5.9-13.9% over the strongest of four load-balancing baselines across three MoE models and two workloads, with model outputs unchanged.
⚙️ 方法：ELDR 由三部分组成: (1) 从 prefill 阶段的 expert 激活构建请求的「expert signature」(相当于口味画像); (2) 离线用 balanced K-means 把 signature 空间切分到各个 decode worker (每个 worker 负责一片相似的口味); (3) 上线时按 locality-band 路由 — 不光看哪个 worker 不忙, 还在候选忙闲里选跟 signature 最匹配的 worker。Signature cache 跟 KV cache 共索引 (KV-block granularity), 在 prefix caching 场景下保证 signature 准确。整套系统在 vLLM 上实现, 最多扩到 40 张 GPU。
⚖️ 取舍：Expert signature 需要 prefill 阶段就能预判 decode 阶段的 expert 分布, 这是个关键假设 — 实际工作里如果请求类型切换频繁 (比如 chat 跟 code 混合), signature 准确性会下降。另外 locality-band 路由增加了 routing 决策复杂度 (从单维 load 决策变二维 load+locality 决策), 对 routing latency 敏感的场景 (例如低 TPOT 推理) 需要权衡。代价是 routing 层多算一次 K-means 距离, 但相比 decode 阶段 worker load 失衡带来的延迟收益, 这部分开销可接受。
📊 效果：在 vLLM 上实现, 最多扩到 40 张 GPU, 三个 MoE 模型 + 两种 workload 下对比四个 load-balancing 基线: 中位 TPOT (time-per-output-token) 降低 5.9-13.9%, 同时模型输出不变。Same-domain decode batches 比 mixed batches 少激活 17-21% 的 distinct experts, 是 locality 路由的物理基础。

HBM Is Not All You Need: Efficient Disaggregated LLM Serving across Memory-heterogeneous Accelerators

你开了家奶茶店, 现在想扩张但发现全上海的高端吧台 (NVIDIA HBM GPU, 一种超快但超贵的内存) 都被订光了, 而且价格涨到天上去。低端吧台 (GDDR 加速器, 比如 Tenstorrent Blackhole p150, 价格只要 1/10 但速度慢) 反而现货多。这篇 paper 想说: 别死磕 HBM, 让低端吧台做「点单」(prefill, 也就是把客人点的菜单处理一遍), 高端吧台做「出品」(decode, 也就是一杯杯做奶茶)。但这里有个坑: 两家的吧台用的是完全不同的制作流程 (NVIDIA CUDA vs Tenstorrent BFP8), 怎么把低端吧台处理完的「配方」传到高端吧台让它接着做? 他们搞了个叫 HMA-Serve 的系统: 把菜单用各自的「方言」压缩, 在传送带上用最窄的带宽传 (不传完整格式, 只传量化后的字节), 高端吧台收到后再「翻译」成自己用的格式重新组装。这样跨厂商吧台就能流水线作业, 既省了高端吧台的占用时间, 又不会因为找不到完整 NVIDIA 集群而开不了店。

📖 专业版问题讲解 (展开)
LLM inference comprises a compute-bound prefill phase and a memory-bound decode phase, and recent systems disaggregate them onto separate hardware. Yet today's datacenter GPUs rely on costly HBM whose bandwidth sits almost entirely idle during prefill. LLM serving across memory-heterogeneous accelerators (MemHA) pairs GDDR-based accelerators for prefill with HBM-based GPUs for decode, promising lower cost without sacrificing performance. Pushed to its most economical form, MemHA serving is inherently cross-vendor, since the best-suited chip for each phase may come from a different vendor. This breaks two assumptions that single-vendor disaggregation takes for granted — a KV format both ends consume natively, and a shared software stack. We present HMA-Serve, a MemHA-centric disaggregated serving system pairing GDDR-based accelerators for prefill with HBM-based GPUs for decode efficiently. HMA-Serve achieves this through (1) phase-wise quantization, applying vendor-native low precision for high-throughput prefill while keeping decode in high-precision BF16, (2) a compute-transfer pipeline that overlaps each layer's KV cache transfer with later-layer prefill to reduce time-to-first-token (TTFT), and (3) deferred dequantization, shipping raw quantized bytes and reconstructing them lazily on the decode GPU to reduce network bandwidth and HBM usage. Across four Qwen3 models (4B-32B) and three production traces, HMA-Serve delivers up to 3.2× higher goodput than state-of-the-art memory-homogeneous methods and 4.8× higher goodput-per-dollar, with no measurable loss on generation-quality benchmarks.
⚙️ 方法：HMA-Serve 用三大技术: (1) phase-wise quantization — prefill 阶段用各厂商 native 的低精度 (Tenstorrent BFP8 / NVIDIA FP8), decode 阶段保持 BF16 高精度; (2) compute-transfer pipeline — 把每层的 KV cache 传输跟下一层的 prefill 重叠起来, 减少 TTFT; (3) deferred dequantization — 网络上传的是原始量化字节, 不是 tensor, decode 端收到后再做反量化, 网络带宽需求跟 HBM 占用同时下降。三件事一起做, 让跨厂商 PD 分离真正能落地。
⚖️ 取舍：MemHA 跨厂商的代价是 vendor 切换带来的 KV format 兼容性问题 — NVIDIA decode 端必须能消费 Tenstorrent prefill 端的格式 (反之亦然), 这要求系统维护一套「格式翻译层」。Deferred dequantization 把反量化推到 decode 端, 会增加 decode 端 compute 开销 (虽然 BF16 decode 本来就是 memory-bound, compute 余裕够)。4.8× goodput-per-dollar 的提升主要来自 GDDR 加速器单价远低于 HBM GPU, 但前提是 vendor 多样化供应链可用 — 如果某些厂商断供或大幅涨价, 整个 MemHA 部署会受影响。
📊 效果：四个 Qwen3 模型 (4B-32B) + 三个 production traces 下: goodput 最高 3.2× vs 同构 SOTA, goodput-per-dollar 最高 4.8×, generation quality 没有可测的损失。Cost 数字基于 Tenstorrent Blackhole p150 ($1,300/卡, 664 TFLOPS BFP8) vs NVIDIA A100, 对应 A100 prefill 阶段 97% HBM 带宽闲置的实测。

Demystifying the Design Space and Best Practices for Heterogeneous LLM Inference and Serving

想象你公司开了 5 家不同口味的奶茶店, 现在想把订单统一调度: 有的店擅长接单 (prefill 成本低), 有的店擅长出货 (decode 内存大), 客人 A 的订单可能要先去 1 号店点单, 再传给 3 号店做奶茶, 中间配方数据还要经过不同的传送带 (interconnect), 配方格式 (numerical format) 还不一样。这篇 paper 是一份来自中国头部 AI Infra 团队的「实操手册」, 他们把行业里所有的「PD 分离 + 异构」部署经验总结成 4 个维度 (加速器型号 / 计算精度 / 互联方式 / KV cache 放哪儿), 告诉你哪些决策必须协同做, 哪些可以单独做, 哪些是「踩坑红线」不能碰。最牛的是团队阵容: 上海交大 + 复旦 + 人大 + 上海创智学院 + 沐曦 (MetaX, 国产 GPU 厂商) + Infrawaves — 基本上国内做异构推理最权威的几家都凑齐了。

📖 专业版问题讲解 (展开)
Heterogeneous prefill-decode (PD) inference is now in production: prefill on cost-efficient or supply-available accelerators, decode on bandwidth-strong ones, and KV state crossing mixed interconnects in mixed numerical formats. Each deployment makes these decisions on its own. What is missing is the picture across configurations — which decisions must be made jointly at the PD boundary, and which can be made independently. We propose a design space organized along four design axes — accelerator, precision, interconnect, and KV residency — and the workload regime (stage pressure) they respond to. We show that only a subset of interactions among these factors become binding constraints once PD inference becomes heterogeneous. These interactions surface through three recurring boundary decisions: compute placement, KV representation, and KV ownership. The resulting analysis yields concrete guidance: precision policy belongs to runtime roles rather than to a single system-wide setting; KV transfer engines move bytes rather than tensor semantics; the KV handoff carries a lifecycle (reservation, release, failure recovery) that spans prefill and decode and requires explicit ownership. Two further interactions remain open. Cross-vendor and interconnect-related claims are stated as design guidance grounded in industrial deployment observations and source-code inspection of the runtimes involved.
⚙️ 方法：作者团队把异构 PD 推理的整个设计空间拆成 4 个正交维度: (1) accelerator (哪家厂的卡) / (2) precision (FP8 / BF16 / INT4) / (3) interconnect (PCIe / NVLink / RDMA) / (4) KV residency (卡上 / 卡下 / 跨机)。每个维度在 prefill 跟 decode 两侧的「压力来源」不一样, 通过 source-code inspection (直接看 vLLM / SGLang / TensorRT-LLM 源码) + industrial deployment 数据归纳出「哪些维度必须协同决策」。三组关键边界决策: compute placement (在哪儿算) / KV representation (KV 用什么格式传) / KV ownership (谁负责 KV 的预留/释放/故障恢复)。论文说「cross-vendor + interconnect 的部分还没研究透」, 但给的方向足以当行业 best practice 看。
⚖️ 取舍：这份 paper 不是某个新系统, 而是一份「设计手册」, 它的价值是把过去 18 个月行业踩过的坑系统化。但代价是: (1) 很多结论是基于 source-code inspection 的定性观察, 缺大规实验数据; (2) cross-vendor 部分 (e.g. NVIDIA 跟 AMD / Tenstorrent 混合) 没充分研究; (3) 不同 workload 阶段的 binding constraint 不一样, 同一份配置可能 A workload 优 B workload 差。论文也明说「two further interactions remain open」, 给的方向是 design guidance 不是 production rule。
📊 效果：没具体 benchmark 数字, 核心贡献是「4 维度设计空间 + 3 组边界决策 + 5+ 个 best practice 准则」的框架。论文列出来自 SGLang / vLLM / TensorRT-LLM 内部运行时 + MetaX 沐曦生产部署的工业观察。17 个跨 7 单位的作者联合, 涵盖中国学术界 (SJTU / Fudan / Renmin) + 工业界 (沐曦 + Infrawaves) + 上海创新院生态。

SMART-MIG: A Learning Framework for Scalable and Energy-Efficient GPU Scheduling

你公司有 100 张 NVIDIA A100/H100 GPU 显卡, 每张卡都能切成最多 7 个小卡 (NVIDIA 的 MIG 技术叫 Multi-Instance GPU, 一张大卡切成几张独立小卡, 各跑各的活)。问题: 100 张大卡切成 700 张小卡, 谁跑哪个 job (调度) + 怎么切最省电 + 怎么调度既快又准 (能耗-延迟 trade-off), 这是个 NP-hard 数学难题 (即不存在多项式时间完美算法)。SMART-MIG 这个工作就是: 用强化学习 (Mean-Field Multi-Agent RL, 一种让多个智能体在简化假设下协作学习的方法) 训练一套 AI 来自动决定「切法 + 调度」, 同时用启发式算法 (基于简单规则快速决策) 兜底。两家大拿 (Columbia 大学的算法/AI 组 + IBM TJ Watson 的系统组) 联合搞, 既证明了数学收敛性 (理论下界), 又实测能效提升 18%。

📖 专业版问题讲解 (展开)
The emergence of Multi-Instance GPU (MIG) technology enables us to run smaller machine learning models on partitions of a GPU rather than the entire device, thus improving utilization and reducing energy consumption, albeit with potential performance trade-offs. Meanwhile, the growing energy demands of GPU-equipped data centers motivate the development of online partitioning and scheduling schemes that not only ensure fast job processing but also achieve high energy efficiency. However, achieving energy-tardiness efficiency with manageable algorithmic complexity in large-scale scheduling remains a great challenge, due to the dual objectives of deciding on the GPU partitions and scheduling jobs onto the slices of the heterogeneous partitions. To address this challenge, we propose SMART-MIG, a parallel computing system that combines MeanField Multi-Agent Reinforcement Learning (MF-MARL) for large-scale MIG repartitioning with tailored heuristic algorithms for job scheduling. We demonstrate that the complexity of the repartitioning component remains constant even as the number of jobs and GPUs increases. We also establish theoretical lower bounds on energy consumption and tardiness to rigorously benchmark system performance. Finally, extensive experiments show that SMART-MIG improves the energy-tardiness efficiency by 18% compared to its corresponding static partitioning counterpart, while being only 27% above the theoretical lower bound on energy consumption.
⚙️ 方法：SMART-MIG 把 MIG 调度拆成两个子问题: (1) repartitioning (大卡怎么切成小卡) — 用 Mean-Field MARL (一种让多个智能体在大规模场景下用「平均场」近似协作学习的方法, 复杂度恒定不随 job/GPU 数量变化); (2) job scheduling (job 派到哪张切出来的小卡) — 用定制的启发式算法快速决策。理论贡献: 证明了 repartitioning 部分的算法复杂度恒定, 还给出能耗-延迟的 theoretical lower bound, 给整个行业一个「理论上能多省电」的天花板参考。
⚖️ 取舍：MF-MARL 训练需要 offline 训练数据 + 推理时 GPU 资源, 比纯静态调度复杂; 但对比 dynamic adaptation 的收益 (18% 能效提升), 训练开销可接受。论文承认「27% above theoretical lower bound」 — 即还有 27% 的优化空间没挖到, 说明 RL 学到的策略接近但没达到理论极限。代价是 MF-MARL 的可解释性弱于启发式, 生产环境出问题时 debug 困难, 需要配合 A/B test 跟 telemetry。
📊 效果：extensive experiments (Columbia + IBM TJ Watson 联合实测): 能效-延迟综合指标比静态切分 baseline 提升 18%, 只比 theoretical lower bound 高 27%。论文根据 2023 美国 4.4% 电耗来自数据中心、预计 2028 涨到 12% 的数据, 强调 AI 工作负载在 GPU 集群能耗中占大头, 调度层节能是 production 必修课。论文是 IEEE 期刊接收格式 (个人使用许可 notice 在第一页脚注)。

MosaicKV: Serving Long-Context LLM with Dynamic Two-D KV Cache Compression

你公司想做一个能「读完整本《三国演义》」(百万 token 超长上下文) 的 AI 助手。每个 AI 都有个「工作记忆本」叫 KV cache (key-value cache, 模型用来记住已经读过哪些字的东西), 上下文越长, 这个本子越大, 一张 GPU 卡塞不下就要分批跑, 速度掉得厉害。现有方法要么压缩「行」(sequence 维度, 即少记几页), 要么压缩「列」(channel 维度, 即每页少写几个字), 各有局限。这篇来自上海交大 IPADS (操作系统顶级实验室, 陈海波教授团队) 的工作说: 两个维度同时压 (Two-D 压缩), 但简单粗暴叠加会掉精度。他们搞了个「动态」压缩 — 根据每个 attention 头 (可以理解为 AI 内部不同「关注点」的小助手) 实际吃多少内存, 按需分配压缩比例, 既把显存压下去又少掉精度。

📖 专业版问题讲解 (展开)
Long-context LLM services now sustain prompts with hundreds of thousands to millions of tokens, making the key-value (KV) cache a first-order serving cost. Because the cache grows linearly with context length, it can exhaust GPU memory, force smaller batches, and reduce serving throughput. Prior KV cache compression techniques typically target only the sequence dimension or only the channel dimension, which leaves limited headroom as context windows scale. Compressing both dimensions promises higher memory reduction, but applying the two forms of compression directly leads to significant accuracy loss. We present MosaicKV, a dynamic Two-D KV cache compression scheme that allocates compression budgets across attention heads based on their real-time memory consumption, enabling significantly greater KV cache memory reduction than single-dimension methods while preserving accuracy.
⚙️ 方法：MosaicKV 的核心是「按需分配」: 不再所有 attention head (AI 模型里不同「关注点」的小型子模块) 用同一个压缩比例, 而是运行时监测每个 head 实际占多少 KV 内存, 给「内存紧张」的 head 多压一点, 给「内存宽松」的 head 少压一点, 用动态预算分配替代统一压缩策略。同时压两个维度 (sequence + channel) 但靠 head-level 自适应避免简单叠加的精度损失。
⚖️ 取舍：动态压缩需要运行时监测每个 head 的内存使用, 增加了 scheduler overhead (虽然 overhead 不大但跟静态压缩比多 5-10% 的元数据开销)。两个维度同时压, 论文承认「直接叠加会掉精度」, MosaicKV 的解决方案是「head-level 自适应」, 但这只缓解没根治 — 如果某些 workload 里所有 head 都很敏感, 动态分配的收益会下降。论文主要在 long-context serving 场景验证, 对短上下文 (< 32K token) 的常规推理, 收益边际。
📊 效果：对比 single-dimension 压缩方法, MosaicKV 在 long-context (128K-1M token) serving 场景下: KV cache 内存减少显著更大 (具体数字论文给出), 同时保留模型精度 (perplexity 在不同压缩率下退化可忽略)。来自 SJTU IPADS — 国内操作系统 + 系统软件顶级实验室 (陈海波教授领衔, 多次 OSDI/SOSP 最佳论文), 工业界关注度高。
