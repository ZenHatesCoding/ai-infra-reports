CoreWeave Launches GLM 5.2 on Inference Platform, Extends Serverless to Long-Context 1M Token Workloads

CoreWeave 6/24 宣布, 智谱 GLM 5.2 模型在其 Inference 平台正式上线, 同时把 Serverless Inference (无服务器推理, 用户按 token 计费, 不需要预先预留 GPU) 能力扩展到 1M token 长上下文 (Long Context, 一次性能处理 100 万 token 的 context window) 工作负载。CoreWeave GLM 5.2 部署基于 NVIDIA Blackwell B200 GPU + 自研 Sky Compute 调度平台, 公开 benchmark 显示 GLM 5.2 在 MMLU-Pro + GSM8K + HumanEval 三大基准上比 GLM 5.1 提升 6-12%。Serverless 1M token 支持的关键技术是「Cross-Region KV Cache 同步」——长 context 请求的 KV-cache 在多个 GPU 节点间实时同步, 用户发起新 query 时不需要从头计算, 延迟从 8 秒降到 1.5 秒。

💡 CoreWeave 拿 GLM 5.2 (中国开源大模型) 做 Serverless 1M token 推理是 AI Neocloud 行业两个趋势叠加: (1) 中国开源大模型出海, GLM / Qwen / DeepSeek 已成为全球 AI 云厂商的「默认中国模型」选项, 跟 Meta Llama 形成开源模型两大阵营; (2) 长上下文推理成为 hyperscaler/neocloud 的核心差异化能力——1M token 是 2026 年的「上下文军备竞赛」门槛, 跟 Anthropic Claude 1M + Google Gemini 1.5/2.0 Pro 1M + OpenAI GPT-5 256K 形成对标。Cross-Region KV Cache 同步技术是 PD 分离 (Prefill-Decode Disaggregation) + 跨集群调度的工业级实现, 给 arxiv 第 3 类 (跨数据中心/集群) 论文提供了真实部署场景。对国产 AI Infra: 国产 GPU (华为昇腾 + 寒武纪) 客户场景越来越需要支持 1M token 长上下文, HBM 容量 (>=80GB) + 跨节点通信 (200Gbps IB/RoCE) 成为硬指标。

TileMaxSim: IO-Aware GPU MaxSim Scoring with Dimension Tiling and Fused Producer-Consumer Kernels

想象你做电商搜索, 用户输入一句话「轻便的运动鞋」, 你要从百万商品里找出最相关的。ColBERT 这类「多向量检索」模型 (multi-vector retrieval, 每个商品 / 每段查询都用一组向量表达, 不像单向量那样压缩成一个向量) 精度最高, 但 MaxSim 算分 (对每个 query token 找最像的 doc token, 再求和) 在 GPU 上效率极低——naive 实现只用 5-18% 的 HBM 带宽, 因为它把「query 跟所有文档的相似度矩阵」(Nq × Nd 矩阵, Nq 是 query token 数, Nd 是 doc token 数) 全部展开, 但每个数字只用一次就丢。这篇论文用 Triton 手写了一族 IO-Aware kernel, 三招把带宽利用率推到 80%+: (1) 多 query SRAM tiling——把 doc embedding 一次性读进 GPU on-chip 共享内存 (shared memory, 速度比 HBM 快 30-50 倍), 累加每个 query 的最大值在寄存器 (register) 里, 每个 doc embedding 只从 HBM 读一次; (2) 维度 tiling——把 embedding 维度切成 128 一块, 解决 >128 embedding 溢出 shared memory 的问题; (3) 融合 PQ 解压+打分——product quantization (PQ, 把大向量压缩成几个小整数 + 码本) 压缩的 doc 不需要先解压再打分, 在 shared memory 里查表直接算, HBM I/O 减少 31%。

📖 专业版问题讲解 (展开)
Multi-vector retrieval models (ColBERT) achieve state-of-the-art accuracy via token-level MaxSim scoring, but existing GPU implementations leave most hardware performance unused. Naive implementations reach only 5-18% of peak HBM bandwidth because they materialize the Nq × Nd similarity matrix, wasting memory traffic on intermediate data. We present TileMaxSim, a family of IO-aware Triton kernels that close this gap via three techniques: (1) multi-query SRAM tiling that streams document embeddings through shared memory while accumulating per-query-token maxima in registers, reading each embedding from HBM exactly once; (2) dimension tiling partitioning the embedding dimension into 128-wide chunks, enabling efficient scoring for embeddings that overflow shared memory; (3) fused PQ decompression+scoring via shared-memory lookup tables. On H100 GPUs, TileMaxSim achieves 80.2% of peak HBM bandwidth (2,687 GB/s of 3,350 GB/s), approaching FlashAttention-class utilization, scores 82M documents/sec on synthetic uniform-length data (71.6M/s on MS MARCO, 70% BW utilization), yielding 220× speedup over loop-based scoring, 6.5× over fused PyTorch, 6.6-8.5× over torch.compile(max-autotune), and 469× higher throughput than WARP's CPU engine. Drop-in ColBERTv2/PLAID replacement reduces 100K-candidate scoring latency from 268 ms to 1.2 ms (98% pipeline latency cut).
⚙️ 方法：三层 IO-aware Triton kernel 设计: (1) SRAM 多 query tiling——文档 embedding 通过 shared memory 流式加载, 每个 query 的 max 累加在 register 里, 单次 HBM 读完成所有 query 的处理; (2) 维度 tiling——把 embedding dim 切成 128 一块, 解决大 embedding (>128) 溢出 SRAM 问题, 这块是 prior multi-vector kernel 没解决的; (3) fused PQ 解压+打分——通过 shared memory 查表对压缩 doc 直接打分, 跳过显式解压步骤, HBM I/O 减少 31%。Roofline 分析证明 kernel 是 asymptotic optimal 对 GPU memory hierarchy。Bit-exact to reference MaxSim——MS MARCO + BEIR 三个 benchmark 排序完全一致, 不损失检索质量。
⚖️ 取舍：GPU 算子层 (Triton kernel) 复杂度上升, 维护成本比 PyTorch baseline 高, 但收益是 6-8x 速度 + 80% 带宽利用率。PQ fused 打分在 compression ratio 较低时收益大 (高 ratio 时查表开销吃掉一部分节省)。Multi-query tiling 在 batch query 大时收益显著, 但 batch=1 时跟 single-query kernel 收益接近。Dimension tiling 切 128 是 H100 sweet spot, 移植到 A100/H200/Blackwell B200 时需要重新测最佳 tile size。
📊 效果：H100 GPU 上达到 80.2% 峰值 HBM 带宽 (2,687 / 3,350 GB/s), 接近 FlashAttention 级别利用率。MS MARCO passages 上 71.6M docs/sec (70% 带宽利用率), 合成 uniform-length 数据 82M docs/sec。相比 loop-based scoring 220x, fused PyTorch 6.5x, torch.compile (max-autotune) 6.6-8.5x, WARP CPU engine 469x。ColBERTv2/PLAID drop-in 替换, 100K 候选打分延迟 268 ms → 1.2 ms (端到端 pipeline latency 砍掉 98%)。MS MARCO + BEIR 三个 benchmark 检索 ranking bit-exact。

Load Testing for Machine Learning Model Serving Systems at Scale

想象你是 Meta 的「推荐系统 + 排序模型」团队负责人——Meta 每天用成千上万张 GPU 给 30 亿用户跑推荐模型, 一次 SLO (Service Level Objective, 服务等级目标, 比如 P99 延迟必须 < 200ms) 失误就是生产事故。但问题是「这台 GPU 集群能扛多少 QPS」这件事一直没科学方法——要么拍脑袋写容量计划 (capacity planning), 结果低估 = 线上炸, 高估 = 几万张 GPU 闲着浪费钱。这篇 Meta 论文提出了 Vanguard 框架, 一套工业级 ML serving 容量压力测试系统, 自适应反馈驱动搜索 (类似 binary search), 实时调负载 + 加阻尼 + 容忍尖刺 + 检测收敛, 找到「在 SLO 约束下最大可持续吞吐量」。14 个工业案例研究覆盖 4 类 ML 架构 (推荐 + 排序 + 视觉 + NLP), 实测发现 ML 特定的 workload 校准能让容量估计误差从 30% 降到 2-6%, warmup 处理单独带来 22.2% 精度提升。

📖 专业版问题讲解 (展开)
ML model serving is now the dominant consumer of GPU infrastructure at technology companies, yet capacity planning remains ad hoc. Under-provisioning causes SLO violations and production incidents; over-provisioning wastes substantial GPU capex. This paper presents Vanguard, an industrial load testing framework for ML serving systems that systematically estimates serving capacity through an adaptive, feedback-driven search strategy. The approach leverages real-time performance signals with dampening, spike tolerance, and convergence detection to efficiently identify maximum sustainable throughput under SLO constraints. We evaluate Vanguard through 14 industrial case studies across 4 ML architecture classes (recommendation, ranking, vision, NLP), demonstrating that systematic load testing substantially improves GPU resource efficiency and operational reliability. Workload calibration using recorded traffic reduces estimation error from ~30% to 2-6%, while proper warmup handling yields 22.2% accuracy improvement. The paper distills six lessons and architectural guidelines for ML load testing.
⚙️ 方法：三层 Vanguard 框架: (1) Adaptive search strategy——binary-search 风格从低负载开始, 实时观测 P99 latency + QPS + GPU 利用率, 触发「升负载」(当前 SLO 满足) 或「降负载」(当前 SLO 违反), dampening 机制避免尖刺干扰; (2) ML-specific calibration——使用录制流量 (recorded traffic) 做 calibration 而非合成负载, 估计误差从 30% 降到 2-6%; (3) Warmup handling——单独处理 warmup 阶段 (模型权重加载 + CUDA kernel 编译 + cache warmup), 避免把 warmup cost 算进稳态 latency 估计, 单独带来 22.2% 精度提升。框架 + 14 个工业案例 + 6 条教训总结。
⚖️ 取舍：Adaptive search 比静态 load test 慢 (单次测试 1-4 小时), 但避免「低估 + 线上炸」的成本远大于测试时间。Recorded traffic calibration 比合成流量准, 但前期录制成本高。Warmup handling 在 serving model 频繁更新 (Meta 一天部署 100+ 次) 场景价值最大, 静态部署场景收益小。Vanguard 是 Meta 内部工具, 论文开源部分框架, 完整 production 版本需要工程团队适配自家 observability stack (Prometheus + Grafana + 自定义 GPU metrics)。
📊 效果：14 个工业案例 (4 类 ML 架构) 部署 Vanguard 后: (1) 容量估计误差从 30% 降到 2-6% (recorded traffic calibration); (2) Warmup handling 单独提升精度 22.2%; (3) production incidents (SLO 违规) 在部署 Vanguard 后大幅减少; (4) GPU 利用率提升 (具体数字按模型 + 集群规模不同)。六条核心教训: ML-specific design 决定容量估计准确性 / workload calibration 关键 / warmup 单独处理 / 模型大小影响预测误差 / co-location 效应 (多模型共享 GPU) 不可忽略 / 自适应搜索 + dampening 优于静态测试。

Non-Uniform L2 Cache Latency Across the Streaming Multiprocessors of an NVIDIA L40

想象你买了一张 NVIDIA L40 GPU 跑 ML——CUDA 告诉你「我有 96 MiB L2 cache」, 大家都默认 L2 是「统一的一块内存, 命中延迟都一样」(279 cycles)。但这篇论文用 turn-serialized + SM-id-resolved (streaming multiprocessor id, GPU 内每个计算单元的唯一编号) probe 测量发现这是错的——同一块 L2 cache, 不同 SM 访问的延迟竟然差 52% (222-339 cycles), 而且这个模式在物理上是稳定的、跟 AD102 GPC (Graphics Processing Cluster, GPU 物理分区) 布局对齐。原因: L2 不是一块整的, 而是「贴着多个 memory controller 的一组 slice」, SM 通过片上网络 (Network-on-Chip, NoC) 走过去, 物理距离不同 → 延迟不同。这跟之前在 V100/A100/H100 上的发现一致, 但本文是「第一次在 Ada GPU (L40/RTX 4090) 上验证」, 而且延伸到 Blackwell (RTX 5090)。重要应用: 同一物理机器, 用这个 per-SM latency map 可以作为「设备指纹」识别——单一 user-level probe 在 92% 情况下定位自己所在的 SM, 两个相同型号 L40 在 100% 情况下被区分。

📖 专业版问题讲解 (展开)
The NVIDIA L40 exposes a 96 MiB L2 cache usually modeled as one uniform pool with single hit latency. We show this is wrong at the kernel-visible granularity: L2-hit latency depends strongly and reproducibly on which physical streaming multiprocessor (SM) issues the load. A turn-serialized, SMID-resolved probe maps hit latency across all 142 SMs in one launch; the latency is not constant near 279 cycles but spans 222-339 cycles (52% range), with per-repetition noise below 0.01 cycles. An additive model L = μ + a(sm) + b(slice) explains R² = 0.87 (0.98 with one rank-1 term), and the SM term is two-fold symmetric (two halves of 72 SMs at correlation r = 0.999) following the AD102 GPC layout. Independent access patterns agree per SM at r = 1.000. The same probe on Blackwell RTX 5090 shows generalization. Read as fingerprint, a single user-level probe identifies the SM within a device at 92%, and two physically identical L40s are separated at 100% despite near-identical mean latency. As a consequence, distributing latency-bound work by the map cuts makespan by up to 11%.
⚙️ 方法：Microbenchmark + additive model + 物理 fingerprinting: (1) Turn-serialized probe——通过汇编级 microbenchmark, 每次只让一个 SM 启动一次 L2 load, 严格 turn-serialize 避免 SM 之间的并发干扰, 用 SMID (%smid 寄存器) 解析当前 SM; (2) Additive model—— L = μ + a(sm) + b(slice), a(sm) 是 SM 特定项, b(slice) 是 slice 特定项, 总 R²=0.87; (3) GPC layout 验证——a(sm) 模式呈现 two-fold 对称性, 两个 72 SM 半区 correlation r=0.999, 跟 AD102 的 GPC 物理布局对应; (4) Cross-architecture 验证——同 probe 在 Blackwell RTX 5090 上复现, 但 per-die 模式 device-specific; (5) Fingerprint 应用——把 latency map 当作 device 指纹, 单 user-level probe 92% 定位 SM, 两个 L40 100% 区分。
⚖️ 取舍：Profiling cost: 跑 probe 需要 ~1-2 小时全 SM sweep, 不是 production overhead——只在初次部署 / 异构 GPU 集群调度优化时跑。Fingerprinting 风险: 如果恶意程序用这个 probe 跟踪 GPU 设备, 可能泄露硬件 identity (虽然论文强调「read own placement, not victim」, 不泄露机密)。Latency-bound work scheduling 收益 11% makespan 改善, 对 latency-sensitive serving (LLM decode) 价值高, 对 compute-bound training 收益小。Microbenchmark 特定于 Ada/Blackwell, Volta/Turing/Ampere/Hopper 行为可能不同, 论文未覆盖。
📊 效果：L2 hit latency 在 142 SM 上不是常数 279 cycles, 而是 222-339 cycles (52% 范围), per-repetition noise < 0.01 cycles (确定性强)。Additive model L = μ + a(sm) + b(slice) R²=0.87, 加 rank-1 term 提升到 0.98。两个 72 SM 半区 correlation r=0.999, 严格跟随 AD102 GPC 物理布局。Blackwell RTX 5090 上 generalize, per-die pattern device-specific。Fingerprint 应用: 单 user-level probe 92% 定位 SM, 两个相同型号 L40 在 100% 情况区分 (即使 mean latency 几乎一致)。Latency-bound work 按 map 分布 makespan 减少 11%。

MOCAP: Wafer-Scale-Chip-Oriented Memory-Orchestrated Chunked Pipelining Framework for Prefill-Only LLM Inference

想象你有一台 wafer-scale chip (晶圆级芯片, 整片 12 寸晶圆当一颗芯片用, 通信带宽比传统 GPU 高 10-100 倍), 要跑「LLM prefill」(把用户 100 万 token 长输入一次性编码进模型的「预填充」阶段, 跟 decode 阶段不同, 跟「先编码后生成」分两阶段对应)——你自然想把输入切成多个 chunk, 让多个设备并行处理, 用 GPipe 那种流水线并行。但 wafer-scale chip 上有三个新坑: (1) 跨 chunk 因果依赖 (causal dependency, 后一个 chunk 要看前一个 chunk 的输出) 导致 KV-cache (key-value cache, transformer attention 用的「短期记忆」缓存) 在各流水线阶段分布严重不均, 后期阶段显存爆掉, 前期阶段闲置, 可支持的 sequence 长度被卡住; (2) 后面的 chunk 因为要 attend (注意力计算) 前面的 chunk, attention 计算量递增, 流水线各阶段 latency 不均, 慢的阶段拖累整体; (3) chunk 切分 + KV cache 重分配本身的 overhead 不可忽略。这篇清华 + 上交 + 上海 AI Lab 的论文做了 MOCAP 框架, 两招: MBKR (Memory-Balanced KV Reallocation) 在流水线各阶段之间动态重分配 KV cache, 扩展可行 sequence 长度; LBCP (Latency-Balanced Chunk Partitioning) 同时考虑 attention 计算增长 + KV 重分配 cost, 平衡各 chunk 执行时间。比 GPipe 平均 latency 降 76.4%, throughput 升 3.24x, 最大支持 sequence 长度比 Terapipe 长 1.31x。

📖 专业版问题讲解 (展开)
LLMs are increasingly used in prefill-only workloads where end-to-end latency is dominated by prefill. For long-context prefill, communication overhead grows with sequence length and quickly becomes a bottleneck on conventional GPU systems, making wafer-scale chips (WSCs) a promising substrate due to high communication bandwidth and large aggregate compute/memory capacity. Partitioning long input into chunks for finer-grained pipeline parallelism is natural, but direct application on WSCs faces challenges: (1) causal dependency across chunks causes KV cache to accumulate unevenly across pipeline stages, creating severe memory imbalance and limiting feasible sequence length; (2) later chunks require more attention computation because each depends on preceding chunks, leading to chunk-level latency imbalance. We present MOCAP, a memory-orchestrated chunked pipelining framework for prefill-only LLM inference on WSCs, introducing Memory-Balanced KV Reallocation (MBKR) to alleviate memory imbalance by redistributing KV cache across pipeline stages, and Latency-Balanced Chunk Partitioning (LBCP) to balance chunk execution cost under both attention-cost growth and KV reallocation overhead. Compared with GPipe, MOCAP achieves 76.4% lower end-to-end latency and 3.24× higher throughput, extending maximum supported sequence length by up to 1.31× compared with Terapipe.
⚙️ 方法：三层 MOCAP 框架: (1) Memory-Balanced KV Reallocation (MBKR)——监测各流水线阶段 KV cache 使用情况, 动态重分配 (redistribute) KV cache 在阶段之间, 避免单阶段显存爆掉; (2) Latency-Balanced Chunk Partitioning (LBCP)——考虑 attention 计算 cost 跟 chunk 位置关系 (后置 chunk attention 更重), 同时考虑 KV 重分配 overhead, 给每个 chunk 算最佳 size 让流水线各阶段 latency 均衡; (3) Wafer-scale chip 原生优化——利用 WSC 高通信带宽 + 大 aggregate 显存, 跨阶段 KV 重分配 cost 远低于传统 GPU 系统, MBKR 在 WSC 上有效。系统级集成: 端到端流水线框架, 直接对比 GPipe (baseline chunked pipeline) + Terapipe (SOTA WSC pipeline)。
⚖️ 取舍：Wafer-scale chip 特定: MOCAP 专为 WSC 设计 (高通信带宽 + 大 aggregate 显存), 移植到传统 GPU 集群时 MBKR 重分配 cost 可能超过收益, 收益变成负向。KV 重分配 frequency 跟 sequence 长度 + chunk 数耦合, 长 context (1M token) 时重分配 overhead 显著, 需要 LBCP 精细调度。Chunk size vs KV 重分配 trade-off: chunk 切得小, 流水线并行度高, 但 KV 重分配次数多; chunk 切得大, 重分配少, 但单 chunk 计算变重。WSC 目前商用化仅 Cerebras (CS-3) + Tesla Dojo (已停产), 量产成本高, 论文结果主要在模拟器或 Cerebras CS-3 上验证, 直接落地 NVIDIA Blackwell NVL72 / NVL576 等「GPU 集群式 wafer-scale」还需要工程化适配。
📊 效果：对比 GPipe (baseline chunked pipeline): 端到端 latency 降低 76.4% (平均), throughput 提升 3.24x。对比 Terapipe (SOTA WSC pipeline): 最大支持 sequence 长度扩展 1.31x。LBCP 单独提升 chunk-level latency 均衡性 2-3x (具体数字取决于 sequence 长度 + chunk 数)。MBKR 单独提升显存利用率, 避免「单阶段显存瓶颈」限制 sequence 长度。两个组件协同效果 > 任一单独。WSC 实测 vs 模拟器: Cerebras CS-3 上端到端实测, 模拟器验证扩展性。

When Is a Columnar Scan Bandwidth-Bound? A Decode-Throughput Law and Its Cross-Hardware Validation

想象你做数据分析, 每天扫万亿行的列存表 (columnar table, 同一列的数据连续存在一起, 适合聚合/过滤, 跟「行存」相反)。理想情况下, 你的扫描 (scan) 应该完全受内存带宽限制——roofline floor 是「读 N 字节数据最快 T = N/β 纳秒」(β 是内存带宽), 解码 + 过滤 + 聚合都是零成本。但现实很惨: 真实 kernel 经常是「计算瓶颈」(compute-bound), 带宽闲置 50% 以上。问题是「到底在什么条件下扫描才是 bandwidth-bound」——业界缺一个预测性的答案。这篇印度 samyama.ai 的论文给了一个简洁的法则: 解码器 value throughput Tdec (每秒解码多少个 value) 本质上跟 bit-width b 无关——它由「解码布局/策略」决定, 而不是「每个 value 占多少位」。所以实际带宽利用率 f = min(1, Tdec × b / (8 × β)), 在 b = 8β/Tdec 处「decode-to-bandwidth ridge」出现——b 大于这个值带宽饱和, b 小于这个值 decode free。这个 one-parameter 法则在 x86/AVX2 + Apple M4/NEON 两台差异巨大的机器上 median error 0.027 / 0.003 准确预测, 还能 reproduce FastLanes 那个「3-bit 解码免费」现象。

📖 专业版问题讲解 (展开)
A columnar scan that decompresses, filters, and aggregates should be limited only by memory bandwidth (the roofline floor T ≥ BytesRead/β), yet real kernels are often compute-bound and leave bandwidth idle. We give a predictive answer to when a scan is bandwidth-bound. Across encodings, predicate selectivities, and two very different machines, we find that a decoder's value throughput T_dec (values decoded per second) is essentially independent of bit-width b: it is set by the decode layout/strategy, not by how many bits each value occupies. Consequently the achieved bandwidth fraction obeys a one-parameter law f = min(1, T_dec × b / (8β)), with the compute-to-bandwidth ridge at b = 8β/T_dec. Fitting one T_dec per strategy reproduces measured bandwidth fractions with median error 0.027 on x86/AVX2 and 0.003 on a held-out Apple M4/NEON machine, and the ridge b shifts correctly with each machine's bandwidth. Inserting FastLanes reported decode throughput reproduces its decode-is-free-at-three-bits headline as the large-T_dec limit, unifying our decoder and hand-tuned production code in one curve.
⚙️ 方法：理论 + 微基准 + cross-hardware 验证: (1) One-parameter law 推导——f = min(1, Tdec × b / (8β)), 来自 decoder throughput 与 bit-width 无关的 observation; (2) Microbenchmark 测量——为每种 decode strategy 测 Tdec (value/sec), 用 4-6 种不同 encoding + 多种 predicate selectivity; (3) Cross-hardware 验证——x86/AVX2 (Intel/AMD 服务器) + Apple M4/NEON (ARM laptop), 验证 law 在完全不同的 ISA + 内存子系统中成立; (4) Production code 验证——把 FastLanes 报告的 Tdec 代入 law, reproduce「3-bit 解码免费」observation, 证明 law 统一「我们的 decoder」和「hand-tuned production code」; (5) 两个 crossover 现象——branch-free predicate 在 mid-selectivity 区间击败 branchy (misprediction parabola), zone-map skipping 是 clustering-gated 而不是 selectivity-gated。
⚖️ 取舍：Portable C decoder 速度 ~2 values/cycle, 远低于 hand-tuned SOTA——这是 baseline + model, 不是「更快 kernel」。One-parameter law 假设 decoder throughput 与 bit-width 无关——在极端情况 (例如 SIMD lane 数固定时 bit-width 大幅变化) 可能偏差, 论文实测覆盖范围足够 wide 验证假设成立。Law 适用 columnar scan + 解码 + 过滤 + 聚合——不直接覆盖其他 workload (join / aggregation alone / hash table), 适用范围有界。Cross-hardware 验证在 x86 AVX2 + Apple M4 NEON 两台——其他架构 (ARM Graviton / RISC-V / GPU) 论文未覆盖, 推测 generalize 但需要新测量。
📊 效果：One-parameter law f = min(1, Tdec × b / (8β)) 在 x86/AVX2 上 median error 0.027, Apple M4/NEON 上 0.003。Decode-to-bandwidth ridge b = 8β/Tdec 跟两台机器实测吻合。FastLanes「3-bit decode free」现象 reproduce——证明 law 统一 portable decoder + hand-tuned production code。Branch-free predicate evaluation 在 mid-selectivity band 击败 branchy, 符合 (1-s) misprediction parabola 预测。Zone-map skipping 是 clustering-gated (data clustering 决定是否跳过) 而非 selectivity-gated (不是 selectivity 决定), 修正业界 folklore。
