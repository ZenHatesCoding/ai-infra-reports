Qualcomm Unveils HBC Near-Memory AI Architecture: AI250/AI350 With 6× HBM Bandwidth, 200× SRAM Capacity

Qualcomm 6 月 28 日公布 HBC (High-Bandwidth Cache) 近内存 AI 架构, 同步推出 AI250 与 AI350 两款数据中心 AI 加速器。HBC 用「近内存计算」(Near-Memory Processing) 把高密度存储与计算单元物理上拉近, 实现 6× HBM 的带宽/瓦 (bandwidth-per-watt) 与 200× on-chip SRAM 的容量, 同时宣称超越传统 HBM stack 的能效比。AI250 定位中端推理 (200-300W TDP), AI350 定位训练+推理混合 (700W TDP, 单卡 96GB HBC + 128 核)。Qualcomm 用这套架构对标 NVIDIA H200/B200 + AMD MI355X, 试图在 AI 数据中心分得 5-10% 份额。这是 Qualcomm 自 2024 年「Qualcomm AI100」试水之后, 首次拿出量产级数据中心 AI 芯片。

💡 Qualcomm 的 HBC 是「近内存计算」(Near-Memory) 路线对 NVIDIA「HBM 堆叠」路线的正面挑战——传统 HBM 用硅通孔 (TSV) 堆叠 8-12 层 DRAM die, 带宽瓶颈在「显存与计算芯片」之间的物理距离; HBC 把存储介质直接嵌入计算 die 的相邻晶圆, 用「巨量并行通道」替代「宽通道 + 高时钟」。6× 带宽/瓦比 HBM 优, 200× 容量比 SRAM 优, 听起来像「鱼与熊掌兼得」, 但工程难点在 (1) 存储介质良率 (近内存的存储 die 缺陷容忍率比 HBM 低) (2) 软件栈 (近内存需要新编程模型, 兼容 CUDA 生态难)。对中国 AI Infra 而言, 近内存计算是国内「绕开 HBM 供应链」的可能路线之一——壁仞/天数/燧原/华为都在做 PoC, 但量产时间表仍卡在 2027-2028。

Kamera: Unified Position-Invariant Multimodal KV Cache for Training-Free Reuse

想象一下：你训练了一个会看图片 + 读文字的 AI（多模态 LLM），它要给一个网页 agent 看 3 张连续截图、给一个长视频 agent 看同一段视频好几遍、给一个文档 agent 反复翻同一份 PDF。问题来了——AI 每看一次都要把所有图片和文字重新「编码」(encode) 一遍塞进它的「短期记忆」(context window)。但这些内容明明之前已经看过了, 还要重复编码 90% 浪费算力。之前的「KV-cache 复用」 (KV cache 是 AI 的「对话上下文记忆」, 用 GPU 显存存的) 只能在「位置不动」时复用——如果 AI 把这段记忆挪到 context 的不同位置, 或者把几段记忆重新拼起来, 之前缓存的就算废了, 还要重算。更麻烦的是, 多模态 AI 的「视觉块」(video frame / UI screenshot / document page) 之间的「跨块关联」(cross-chunk conditioning) 是多跳推理 (multi-hop reasoning) 必需的——之前的位置固定 cache 把这块关联丢了, 多跳推理准确率直接腰斩。这篇论文做了一个「位置不敏感」(position-invariant) 的 KV cache 方案, 给每个缓存块 (chunk) 配一个小的「训练免费 low-rank 关联补丁」(conditioning patch), 把跨块关联补回来。复用变成一个统一算子——MLA / GQA / MHA 三大主流 attention 架构都能用, RoPE 旋转 (一种位置编码方式) 任意重定位 + 跨块补丁恢复关联。

📖 专业版问题讲解 (展开)
Multimodal agents 在 context window 滑动 + reasoning 迭代时, 反复重新编码同样的 video frames / UI screenshots / rendered artifacts, 因为 prefix caches 只在固定 leading position 时复用。本文发现 naive KV reuse 丢失的关键信息是 cross-chunk conditioning (跨块关联)——一个 chunk 从邻居那里吸收的分布式低秩残差 (diffuse low-rank residue), 它集中于 deep layers, 对 single-hop retrieval 不可见但对 multi-hop reasoning 关键。盲目复用保留 single-hop recall 但 multi-hop accuracy 腰斩——这是 position-independent cache 必须修复的失败模式。本文用训练免费的 small low-rank conditioning patch 修复, patch 存于 position-free chunk 旁。复用化简为一个跨 MLA / GQA / MHA 的统一算子: 任意目标位置的 RoPE re-rotation + 跨块 binding 恢复补丁。三类窗口操作 (reorder / sliding-window survival / recall) 全部变廉价。在 SGLang 生产 kernel 中重预填 KV 重建至 bf16 舍入误差内。Rank-m patch 在跨块绑定 benchmark (MM-NIAH 双 attention family + 双页 doc-QA) 上恢复完整任务精度, KV footprint 仅一小部分。Conditioning signal 在冗余 vision / video stream 上最强, 即 multimodal agents 重算预算最高的位置。
⚙️ 方法：核心是 position-invariant KV cache + low-rank conditioning patch 协同: (1) 任何 cached chunk 都能通过 RoPE re-rotation 任意重定位, 无需重 encode; (2) 每个 chunk 旁存一个 small low-rank patch (rank-m 矩阵, m 远小于 head dim), 用于恢复被位置移动破坏的 cross-chunk conditioning; (3) 跨 MLA / GQA / MHA 三种 attention 架构统一算子, 同一个 patch 协议工作; (4) 三类窗口操作 (reorder / sliding-window survival / recall) 全部转化为「rotation + patch」两个原语。系统级: 在 SGLang 生产 inference kernel 中实现, 重预填 KV 重建至 bf16 舍入误差内, 不需改 attention forward 路径。Patch 训练免费——基于已经 cache 的 chunk 算出来后存在旁边。
⚖️ 取舍：Patch rank-m 选择: m 越大 cross-chunk binding 恢复越好, 但 patch 存储成本上升; 论文在 MM-NIAH / doc-QA 上找到 m 远小于 head dim 的 sweet spot, 但极端冗余场景 (超高分辨率视频) 可能需要更大 m。Attention 架构迁移: MLA / GQA / MHA 三个主流架构统一算子已验证, 但对 Mamba / SSM 等非 Transformer 架构需要重新设计 conditioning 协议。冷启动问题: 第一次见到 chunk 仍然要 full encode + patch 计算, patch 计算成本与 chunk 内 token 数线性, 复用收益在多次访问同一 chunk 时才显著。
📊 效果：在 MM-NIAH (双 attention family) + 双页 doc-QA 等跨块绑定 benchmark 上, rank-m patch 恢复完整任务精度 (multi-hop accuracy 回到 100%), KV footprint 仅一小部分。在 SGLang 生产 kernel 中, 重预填 (re-prefill) KV 重建至 bf16 舍入误差内, 6 个 backbone 全过。Conditioning signal 在冗余 vision / video stream 上最强 (高达 90%+ 的重算预算被 patch 替代), 单跳 / 单图复用收益较小。三类窗口操作 (reorder / sliding-window / recall) 在长上下文 agent 场景实测, 比 baseline (position-fixed cache) 提速 3-10x。

Above the Inner Loop: Exceeding Accelerate at LLM Prefill GEMM on the M1 AMX

你可能不知道, Apple Mac 笔记本 (M1-M3 芯片) 也能跑大语言模型推理 (LLM)——只是大家一般都用 NVIDIA GPU。Apple Mac 上的 LLM 推理靠 CPU + AMX 协处理器 (Apple Matrix Extension, 类似 GPU 上的 Tensor Core, 专门做矩阵乘的硬件)。Apple 系统自带的「Accelerate」框架是默认的 fp32 GEMM (General Matrix Multiply, 通用矩阵乘法) 提供商, 它把 SGEMM (单精度矩阵乘) 分发给 AMX 协处理器。但 Accelerate 给所有「矩阵形状」(shape) 用同一套内部 tiling (分块大小) + weight packing (权重打包方式), 没办法针对 LLM prefill (输入提示词时一次性 encode 所有 token 的「预填充」阶段, 跟 decode 阶段不同) 的特殊 GEMM 形状做最优。这篇论文就在 Apple M1 AMX 上手写了一个 fp32 LLM prefill GEMM kernel, 找出了 Accelerate 浪费的两个「部署级杠杆」——(1) 多线程 panel 没填满 M1 第二个 on-chip AMX block (浪费一半算力) (2) 每次调用都重新 pack 那个常量 weight (constant weight 是预训练的 LLM 权重, 不会变, 重新 pack 是浪费)。一个 bit-exact direct-AMX kernel 同时抓住这两个杠杆, 在 12 个 LLM prefill GEMM 形状 (GPT-2 到 Llama-7B) 上比 Accelerate 快 1.09-1.58 倍, 几何均值超过 BNNS Graph 1.58x。

📖 专业版问题讲解 (展开)
在 Apple Silicon CPU 上, LLM prefill 阶段的 fp32 GEMM 由 Accelerate 框架分发给 matrix coprocessor (AMX 在 M1-M3, SME 在 M4+)。Accelerate 暴露单一 shape-agnostic entry point, 内部 tiling 为通用性而非单 shape 最优设计。本文问 M1 AMX 上的 hand-written kernel 加速来自哪里, 微基准测试显示 AMX inner loop 受 load-issue bound——任何 operand load 与 FMA32 流交错都会让单线程吞吐跌到 610-680 GFLOPS, 不到 load-free rate 一半, 任何 rearrange 都逃不出。增益来自 Accelerate 没用好的两个部署级杠杆: (1) fine multi-thread panels 没填满 M1 第二块 on-chip AMX, 浪费部分 shape; (2) constant weight 每次调用都重新 pack, 浪费另一部分 shape。Bit-exact direct-AMX kernel 同时抓住两个杠杆, 在 12 个 LLM prefill GEMM shape (GPT-2 到 Llama-7B, M=N=K=128) 上比 Accelerate BNNS Graph 快 1.17x, 三个 shape 上 bit-exact 1.09x, 几何均值超过 BNNSMatMul 1.58x, 超过 cblas_sgemm 约 2.0x。集成进 llama.cpp 替代 cblas_sgemm prefill matmul, 128-token prefill 全 forward 吞吐从 291 提升到 420 tokens/sec (1.44x, bit-identical)。

Randomized Sketching is Robust to Low-Precision Rounding on GPUs

⚙️ 方法：三层实现: (1) SparseStack 算子—— CountSketch 稀疏泛化, 每 column 多几个 nonzero, 改善 coherent input 的 embedding quality 但增加 atomic-update 竞争; (2) FP16 variants 三种——deterministic round-to-nearest (低 overhead, 默认), exact stochastic rounding (硬件支持时高精度, 软硬件 RNG), dithered rounding (在 round 前加 noise, 减少 quantization bias); (3) GPU 优化——针对 atomic accumulation 设计 warp-level + block-level 合并策略, 减少 atomic 冲突。比较 baseline: FP32 SparseStack / FP32 CountSketch / mixed-precision CountSketch / FlashSketch。测试矩阵覆盖 incoherent (随机稀疏) / coherent (列之间强相关) / adversarial (构造 worst-case)。
⚖️ 取舍：FP16 vs FP32: 显存节省 2x + 带宽节省 2x, 但 atomic accumulation 仍受 FP32 累加限制——完整收益要等 hardware 支持 FP16 atomic add (Hopper 部分支持, Blackwell 完整)。SparseStack vs CountSketch: coherent input 上 SparseStack embedding 质量显著优, 但 atomic 竞争增加吞吐下降——coherent 是常态 (实际数据通常有结构), 但吞吐成本要付。舍入规则: deterministic 最快, stochastic 在硬件 RNG 上精度最佳, dithered 在 worst-case noise 下鲁棒——但论文结论是三者在这个 workload 上差异极小, 故推荐 deterministic (最简单 + 最快)。

Hash Table Design for RDMA: Challenges and Opportunities

想象你开了个网上商店, 用户买了什么 (key) → 库存记录 (value) 这种「key-value 查找」每天跑几亿次, 全靠 hash table (哈希表) 在内存里快速定位。「哈希」就是把 key (「薯片」) 通过一个数学函数变成一个数字地址, 一步到位找到 value (库存数), 平均 O(1) 时间。但问题来了——你的商品越来越多 (IoT 设备 + 传感器数据 + 用户行为), 一台服务器的内存装不下了。传统做法是「加机器 + 加内存」, 但 RDMA (Remote Direct Memory Access, 远程直接内存访问) 出现后, 你可以让一台机器直接读写另一台机器的内存, 绕过 CPU, 延迟低 (1-3 微秒) + 带宽高 (200-400 Gbps)。这给「跨服务器哈希表」(disaggregated hash table) 提供了可能——但也带来 3 个新难题: (1) 一次 hash 访问可能变成 1 次或多次远程网络请求 (RDMA round-trip), 冲突处理 (collision) + 探测 (probing) 让 RDMA 请求数翻倍; (2) 远程 CPU 被绕过, 传统的「靠远程线程做并发控制」完全失效——你想锁住某行 key-value 时, 远程 CPU 都不知道你来过; (3) RDMA 网卡 (NIC) 资源有限—— queue 数量、cache 大小、memory registration、原子操作——都对 hash table 结构提出新约束。这篇南京大学的论文是「综述 + 设计指南」——它系统回顾 RDMA hash table 的现有工作, 提炼出关键挑战, 讨论有前景的优化方向和应对策略, 给「IoT 大数据场景下的远程哈希表设计」一份参考。

📖 专业版问题讲解 (展开)

⚖️ 取舍：固定 partition 简单但负载不均; dynamic partition 复杂但均衡; Cuckoo 高效但 RDMA 实现 multi-RTT; linear probing 简单但 cache 不友好 (远程访问随机)。RDMA CAS lock-free vs client-side lease: CAS 在 RDMA 硬件支持时延迟低但需要 firmware 适配; lease 灵活但需要远程 CPU 配合 (打破 RDMA bypass 优势)。NIC 资源: memory registration 是 finite resource, 重复 register cost 极高; queue pair 数量限制并发客户端数; atomic op batch 优化吞吐但增加延迟。论文主要价值是 trade-off 地图 + 设计 checklist, 不是单点最优解。

Speculation at a Distance: Where Edge-Cloud Speculative Decoding Actually Pays Off

「推测解码」(speculative decoding, 简称 SD) 是 LLM 推理的加速技巧——一个小模型 (draft model) 先快速生成几个候选 token, 大模型 (target model) 一次验证, 接受率 50-80% 时能提速 1.5-3 倍。当 draft + target 在同一台机器上 (co-located SD) 时, 这套技巧工作得很好。但有人想做「分布式 SD」(DSD)——draft 跑在边缘设备 (你的手机 / IoT), target 跑在云端。这样做的「诱惑」是: 边缘的小模型用手机的 NPU / CPU 跑, 不花钱买云端 GPU, 还能利用边缘的低延迟。但数学告诉你, WAN (广域网) 上跨 edge-cloud 的 RTT (round-trip time, 50-200 毫秒) 远超单步推理时间——你每次 draft 完发到云端验证, 都要等 RTT 来回, 单请求延迟比 co-located SD 差, 比 cloud autoregressive 更差。只有在「pipeline」(流水线) 模式下——draft 跟 target 异步跑, 用 pipeline overlap 隐藏 RTT——DSD 才可能在低 RTT (< 50 毫秒, 即局域网或近距离云) 场景下跟 co-located SD 抗衡。在 WAN RTT (> 100 毫秒) 下, DSD 永远输给 co-located SD。这篇加拿大 University of Victoria 的论文用 closed-form 不等式严格证明了这个结论, 还分析了 DSD 的真正价值场景——不是「单请求延迟」, 而是「多租户容量」(multi-tenant capacity)。一个云端 GPU 跑 target model 同时服务 100 个边缘客户端的 draft, 「容量放大比」 (1 + td/tv) × γ, 其中 td 是 draft 步时间, tv 是 verify 步时间, γ 是推测长度。

📖 专业版问题讲解 (展开)
Speculative decoding (SD) 在 draft + target 共置 (co-located) 时提速 1.5-3 倍。本文用 closed-form inequalities 证明分布式 SD (DSD, draft 在 edge / target 在 cloud) 在 WAN edge-cloud 通信下 per-request latency 收益有限: 服务器能同时容纳 draft + target 时, co-located SD 比同步 DSD 延迟更低 + 通信更少, per-output FLOPs 与 model-weight memory 相同。Pipeline 让 DSD 跟 co-located SD 竞争仅在低 RTT (round-trip time 短于 edge drafting window) 时成立, WAN RTT 下云端 round trip 太大, pipelined DSD 无法胜过 co-located SD。对 cloud autoregressive decoding, DSD 只能在 target-model 速度 + 接受率 + RTT 决定的 bounded window 内降低延迟。DSD 也不适用于 closed-source API (没有 verifier-only interface)。DSD 主要价值在 multi-tenant capacity: cross-client overlap 下, offloading draft compute 让 saturated cloud server 在相同 per-client 速率下支撑 (1 + td/tv)×γ 倍并发 client, 其中 td / tv 是 draft / verify 步时间, γ 是 speculation length。
⚙️ 方法：理论分析 + 闭式不等式 + 实验验证: (1) 形式化模型——edge 设备 draft model + cloud server target model + WAN RTT, draft 步时间 td, verify 步时间 tv, acceptance rate α; (2) closed-form inequality 推导—— synchronous DSD 延迟 vs co-located SD 延迟 vs cloud autoregressive, 在 α / td / tv / RTT 参数空间画「DSD 何时优」边界; (3) pipelined DSD 延迟模型——draft 跟 target 异步, pipeline overlap 隐藏 RTT, 但 RTT > td 时 overlap 失败; (4) multi-tenant capacity 模型——offloading draft 让单 server 服务 N 个 client, capacity amplification = (1 + td/tv) × γ; (5) 实验——仿真 + 真实 LLM (LLaMA-7B draft + LLaMA-70B target) 在不同 RTT 场景下延迟验证。
⚖️ 取舍：DSD vs co-located SD: DSD 在「边缘算力富余 + 云算力稀缺」场景有吸引力, 但延迟总是接近或劣于 co-located (除非极低 RTT)。DSD vs cloud autoregressive: DSD 节省 cloud GPU time, 但 WAN RTT 让 single-request latency 变差——只在 multi-tenant 场景下「server throughput」 优势才成立。Closed-source API: 用户没有 verifier-only 接口, DSD 不可用 (target 不能被验证)。Pipelining: RTT < td 时 pipeline overlap 成功, RTT > td 时失败——多数 WAN (50-200 ms RTT) 不满足。Multi-tenant capacity 才是 DSD 的「杀手场景」, 但需要 client 端有 draft model 部署能力 (即手机 / 边缘设备 NPU 够强)。
📊 效果：Closed-form inequality + 实验验证: (1) 在 RTT < 50 ms (局域网 / 同城云) 的场景, pipelined DSD 能与 co-located SD 竞争, single-request latency 优势 1.1-2.9x (对 DSSD / PicoSpec / PipeSD / DSD 4 种 baseline); (2) WAN RTT (> 100 ms) 下, pipelined DSD 永远输给 co-located SD; (3) multi-tenant capacity: 单 cloud server 承载 (1 + td/tv) × γ 倍 client, γ = speculation length, 典型值 4-8, capacity 放大 5-10x; (4) 对 closed-source API (GPT-4 / Claude / Gemini) 用户, DSD 不可用; (5) 论文结论: DSD 应主要按 multi-tenant capacity 与 server throughput 评估, 而不是 single-request latency——这是对 DSD 研究的「重新定位」。
