RolloutPipe: Overlapping Pipelined Rollout and Training in Disaggregated On-Policy LLM Reinforcement Learning

想象你开了一家奶茶店, 店里分两条线: 一边是「调饮师」做新口味的试喝 (rollout, 让 AI 模型尝试不同回答), 一边是「配方师」根据调饮师的反馈调整配方 (training, 更新模型权重)。问题来了——传统做法是调饮师把所有口味都试完, 配方师才开始改配方, 配方师常常在空等。RolloutPipe 这篇论文做的就是把这两条线流水线重叠起来: 调饮师做完一小批口味, 配方师立刻就开始改这一批的配方, 同时调饮师继续试下一批, 互不耽误。在 Qwen3-1.7B 上, rollout 到 train 端到端时间能省 30% 到 42%。

📖 专业版问题讲解 (展开)
LLM 推理型训练 (reasoning RL) 越来越依赖可验证奖励的强化学习 (RLVR)。现代 RLVR 系统采用 disaggregated 架构, rollout 与 policy training 解耦到独立 GPU 池。同步 on-policy GRPO 系统 rollout 完才训练, trainer GPU 空等; 异步流水线重叠两端但训练数据陈旧。RolloutPipe 提出 complete-group pipelining (CGP) + frontier-group dispatch (FGD), 让 trainable complete groups 边生成边送往 trainer, 同时 rollout 节点优先准入下一批 frontier group。设计保留 on-policy 正确性, 在 Qwen3-1.7B 四个推理 + 科学 benchmark + 12 个 rollout 配置下, rollout-to-train 时间缩短 30.7-42.3%, trainer waiting ratio 降低 37-76% 对比 Slime。
⚙️ 方法：CGP 把 rollout 输出切成 complete groups, 每个 group 生成完毕立刻 FIFO 送往 trainer 节点, 不必等整批 rollout 结束。FGD 是 rollout 节点的准入策略, 优先准入能组成下一批训练 batch 的 frontier groups, 让 trainer-ready groups 更早更稳到达。两项技术组合让 trainer 在 rollout 完成前就开始训练, 同时保持 on-policy 正确性。系统级设计: rollout pool (生成) + trainer pool (更新) + dispatcher (调度), 各 pool 异构 GPU 配置。
⚖️ 取舍：流水线重叠带来内存与同步复杂度上升: rollout 节点需要保留 frontier group 的中间状态, trainer 节点需要处理不完整 batch。对 stale data 容忍度低的 workload (如对超长 chain-of-thought 训练) 仍同步更安全。比起纯异步流水线 (Async RL), RolloutPipe 的 CGP 保留了 "完成一个 group 立即训练" 的同步性, 折中 stale 与空闲两端。
📊 效果：Qwen3-1.7B 模型 + 四个推理/科学 benchmark (数学/逻辑/科学任务) + 12 种 rollout 设置下, rollout-to-train 端到端时间缩短 30.7%–42.3%, trainer waiting ratio 降低 37%–76% 对比 SOTA 基线 Slime。设计保持 on-policy 正确性, 不需要牺牲数据新鲜度。

Moebius: Serving Mixture-of-Expert Models with Seamless Runtime Parallelism Switch

MoE 模型 (混合专家, 一个大问题拆给多个「专家」GPU 分头跑) 跑服务的时候有个老难题: 高并发时专家并行 (EP) 快, 低并发时张量并行 (TP) 快, 但你不能等流量变了重启服务。Moebius 这篇做的就是「不重启引擎、不丢正在服务的请求」的前提下, 让系统在 EP 和 TP 之间无缝切换。实现的核心洞察是: EP 和 TP 其实是同一模型的两种切法, 切的是「同一份专家权重 + 同一份对话缓存 KV-cache 的归属关系」, 移动这些归属关系就够了, 不必搬数据。8 卡 H200 跑 Qwen3-235B-A22B, 切换只要 215-434ms, 显存多占 2.4%。

📖 专业版问题讲解 (展开)
MoE 大模型推理需要在张量并行 (TP) 和专家并行 (EP) 间选择, 取决于并发度: 低并发 TP 优, 高并发 EP 优。生产 workload 持续穿越这一边界 (在线服务突发访问 / RL rollout 高并发峰值后回落长尾)。固定任一布局都跨边界时牺牲性能。Moebius 在运行时不重启引擎、不丢 in-flight 请求, 动态切换 EP 与 TP。洞察: EP/TP 是同一模型的两种布局, 不是两个模型, 权重 + KV-cache 字节相同, 切换仅改变 rank 归属。利用现代高带宽 GPU 互联 (NVLink/InfiniBand), 在 decode 步骤间完成切换。
⚙️ 方法：Moebius 在 EP/TP 间运行时不重启切换, 关键设计: (1) 保持两种并行布局的 runtime resident, expert weights + KV-cache 在固定地址; (2) fused GPU-to-GPU transfer kernel, 只移动 rank 归属变更的切片; (3) decode 步骤间切换, 不需 drain in-flight 请求; (4) 切换时间 215-434ms, 显存开销仅 2.4%。在 8×H200 上服务 Qwen3-235B-A22B 验证。
⚖️ 取舍：维持两种布局同时驻留显存增加 2.4% 显存开销, 对显存紧张的小模型不友好; fused transfer kernel 需要高带宽互联 (NVLink/InfiniBand), PCIe-only 部署会显著拖慢切换时间。对比固定单一布局: 不需要为不同 workload 维护多个服务实例, 节省运营复杂度; 对比纯静态 TP/EP: 在 RL rollout 跨并发边界场景下获益 1.16-1.25x。
📊 效果：8×H200 + Qwen3-235B-A22B 上, Moebius 在每个操作点匹配最优静态布局, RL rollout 跨步骤胜出 1.16-1.25x。每次切换耗时 215-434ms, 显存开销仅 2.4%。

DMuon: Efficient Distributed Muon Training with Near-Adam Overhead

Muon 是 2024 年开始火的矩阵优化器 (跟 Adam 同一量级但几何性更好, DeepSeek-Kimi 都在用), 但它有个老问题: 分布式训练时, 每一步都要把所有权重矩阵汇总回来做 Newton-Schulz 迭代 (一种正交化算法), 通信开销巨大, 比 AdamW 多花 2 倍以上的步时间。DMuon 这篇做的就是把这个通信开销压到接近 AdamW 水平——通过把矩阵分片规则与分布式分片规则对齐, 减少重构矩阵的通信量, 实现 drop-in 模块化集成。在具身基础模型 + LLM 训练 workload 上, 端到端步时间 1.48-3.01 倍加速, optimizer step 6.85-163 倍加速。

📖 专业版问题讲解 (展开)
矩阵正交化优化器 (Muon) 比 AdamW 在 compute-optimal 预训练下 compute efficiency 高约 2x (Moonlight), Kimi-K2/DeepSeek-V4 已在生产规模使用。但 Muon 需要 Newton-Schulz 迭代, 作用于完整权重矩阵, 与现代分布式训练栈 (按 parameter shard 切分, AdamW 是 element-wise) 假设不符。Vanilla Muon 分布式步时间 > 2x AdamW。DMuon 是开源分布式 Muon 实现, 集成进现有训练 pipeline 作 drop-in 模块, 端到端步时间加速 1.48-3.01x, optimizer step 加速 6.85-163x。
⚙️ 方法：DMuon 把 Muon 的矩阵正交化计算与分布式训练的 parameter sharding 规则对齐, 减少每步需要重构完整矩阵的通信次数; 通过矩阵分片重建顺序优化, 让 Newton-Schulz 迭代尽量在本地 shard 上完成; 提供 PyTorch drop-in 接口, 无需框架级修改; 同时支持 LLM 与具身基础模型训练 workload。
⚖️ 取舍：对 element-wise 优化器 (AdamW) 友好的分布式架构要重新调整以适配矩阵优化器, 工程改造面广; 矩阵正交化的额外计算开销仍在 (虽然通信压缩了), 对小模型不一定有收益; MoE 模型的 expert 矩阵切分需要单独处理。DMuon 的 drop-in 集成减少框架改动, 但 PyTorch distributed 内部仍要适配。
📊 效果：在具身基础模型 + LLM 训练 workload 上, DMuon 端到端 step time 加速 1.48-3.01x, optimizer step 加速 6.85-163x, per-step 延迟压到接近 AdamW 水平。Code 开源在 github.com/X-Square-Robot/dmuon。

SOLAR: AI-Powered Speed-of-Light Performance Analysis

你买了一块新 GPU, 想跑一个深度学习模型, 怎么提前知道「理论最快能跑多快, 现在的实现离这个极限还差多少」? SOL 分析 (Speed-of-Light) 就是回答这个问题的——算一个 workload 在给定硬件上的理论最短时间。问题是, 这个 SOL 上限现在还得人工算, 容易出错, 还跟不上模型迭代。SOLAR 这篇做的就是用 AI 自动算 SOL: 给你一个深度学习模型图 + 目标硬件, 它自动分析 workload 特性, 给出 SOL 理论上限 + 当前实现的优化空间, 不用手工拆 roofline 模型。

📖 专业版问题讲解 (展开)
深度学习 workload 在目标硬件上跑多快、当前实现离理论上限多远, 是软件/硬件/算法优化的核心问题。Speed-of-Light (SOL) 分析计算 workload 在给定架构上的理论最短执行时间。但推导 SOL 上限目前依赖手工, 容易出错, 与快速模型迭代脱节。SOLAR 提出自动化框架: 给定模型图 + 目标硬件, 自动分析 workload roofline, 给出 SOL bound + 优化空间。
⚙️ 方法：SOLAR 用 AI agent + 自动 roofline 分析: 输入模型图 (PyTorch/JAX/ONNX) + 目标硬件 spec; 自动识别 compute-bound vs memory-bound vs communication-bound 段; 计算每个算子的 SOL 上限 (考虑 HBM bandwidth + compute throughput + interconnect); 输出端到端 SOL bound + 每个算子的优化机会排序; 内置 NVIDIA GPU 性能模型 + 跨硬件验证。
⚖️ 取舍：SOLAR 的精度取决于性能模型对真实硬件的拟合, 对小众加速器 (TPU/AWS Trainium) 精度有限; 自动分析输出需要用户解读, 不能直接给出 "加速 5x" 的具体 patch; 模型图需要完整 (没有 dynamic shape 的特殊化优化); 不替代 nsys profiling, 而是在 profiling 之前给出 "理论天花板"。
📊 效果：框架集成在 NVIDIA 内部性能分析流水线, 用于指导新模型 (Megatron/Transformer Engine) 的算子实现优先级排序。论文在多个 workload 上展示了 SOL bound 与真实 profiling 的拟合度, 给出优化空间估算。

SegFold: Accelerating Sparse GEMM with a Fine-Grained Dynamic Dataflow

稀疏矩阵乘法 (SpGEMM, 两个「大部分是 0」的大矩阵相乘) 在很多场景都关键 (图神经网络、推荐系统、稀疏 attention), 但 GPU 和现有加速器都还在用「静态数据流」(inner product / outer product / Gustavson 等固定模式), 每种模式都会牺牲一部分数据复用机会 + 强加负载平衡限制。SegFold 这篇做的就是给 SpGEMM 加「动态性」: 细粒度动态调度优化复用, 动态重映射部分完成的工作改善负载平衡, 形式化为新的数据流 Segment, 并配套设计 SegFold 加速器 (含动态内存控制器 + 合并网络)。在多种密度/矩阵大小下, 比 SOTA SpGEMM 加速器几何平均 1.95 倍加速, 比最优静态配置 5.3 倍。

📖 专业版问题讲解 (展开)
SpGEMM (稀疏矩阵-矩阵乘) 在图神经网络 / 推荐系统 / 稀疏 attention 等领域关键。现有 CPU/GPU/专用加速器依赖静态数据流 (inner product, outer product, Gustavson 等), 每种静态数据流都牺牲部分数据复用 + 强加负载平衡约束。SegFold 提出 fine-grained 动态数据流 Segment: 细粒度动态调度优化 reuse, 动态 remap 部分完成工作改善负载平衡; 配套设计 SegFold 加速器 (内存控制器识别局部 window 复用 + 合并网络动态路由到 PE)。
⚙️ 方法：Segment 数据流扩展传统 SpGEMM 数据流, 引入 dynamism: (1) 细粒度动态调度, 根据 stationary input 的局部 window 识别复用机会, 动态分配给处理单元 (PE); (2) 动态 remap 部分完成工作, 改善负载平衡; (3) 内存控制器识别 fine-grained reuse opportunities, 通过动态 work assignment 利用; (4) 合并网络路由 inputs 到合适 PE, 动态重映射 PE 工作。代码 / RTL 开源 (论文中给出)。
⚖️ 取舍：Segment 数据流的动态性需要运行时调度支持, 比静态数据流硬件复杂度更高, area overhead 与功耗上升; 动态 remap 的开销对极小矩阵 (低利用率) 不明显; 配套的 SegFold 加速器需要定制 RTL, 不直接兼容 GPU 软件栈。优势在 SpGEMM workload 的常见密度区间 (10%-50% sparsity) 收益最大。
📊 效果：多种 density + 矩阵大小下, SegFold 比 SOTA SpGEMM 加速器几何平均加速 1.95x, 比最优静态数据流配置加速 5.3x。证明动态数据流设计空间解锁了静态调度单独无法实现的复用 + 负载平衡收益。
