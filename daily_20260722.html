HyMCache: A KV Cache Framework for Multi-Turn LLM Serving with CXL-Hybrid Memory

想象一下: 你开了一家奶茶连锁店, 顾客会反复点同一款「半糖少冰珍珠奶茶」, 每次都要重新记配方 (KV-cache) 太浪费。如果能记住 1 万个老顾客的「历史订单」放在最便宜的储物间 (SSD), 偶尔才搬到柜台 (DRAM) 调取, 就能既省成本又保证点单速度。HyMCache 就是给大模型推理做这件事: 多轮对话的上下文 (KV-cache) 体量越来越大, GPU HBM (最快最贵) 装不下, 传统 DRAM 也贵, 用 CXL (一种新接口) 把 SSD 拼成「廉价大仓库」当成二级 KV 缓存。在真实 CXL-HM 原型机上跑, 同等 DRAM 预算下, 性能比单机版 LMCache 高 3 倍, 比 PD 分离部署版高 1.45 倍; 跟 1TB 分布式 DRAM 的 Mooncake 比, 性能略低 30%, 但 DRAM 用量少 16 倍。

📖 专业版问题讲解 (展开)
多轮 / 长上下文 / Agent 类 LLM 推理工作负载的核心瓶颈已从计算转向存储 (KV-cache 容量 + 跨轮次复用)。本文提出 HyMCache, 把 CXL-Hybrid Memory (CXL-HM, 即「小容量 DRAM + 大容量 SSD」通过 CXL 接口暴露为统一内存) 引入多轮 LLM 推理 KV-cache 复用场景。作者观察到多轮 KV-cache 访问具有「读为主、可预测、仅追加」三个特征, 据此重新设计 CXL-HM 内的 DRAM 调度策略 (请求级前缀预取 + 机会式写缓冲), 把延迟关键读请求 stage 到设备 DRAM, 实现「DRAM 级 KV-cache 效率 + SSD 级容量 + SSD 级成本」。在真实 CXL-HM 原型机上评估, 单聚合器与 PD (prefill-decode) 分离两种部署配置下, 同等 DRAM 预算比 LMCache 分别提升 3.0× / 1.45×, 比 1TB 分布式 DRAM 方案 Mooncake 性能低 30% 但 DRAM 用量减少 16×。本文 C3 类, 跨数据中心/集群/存算分离。
⚙️ 方法：HyMCache 三层架构: (1) CXL-HM 设备层把小容量 DRAM + 大容量 SSD 暴露为统一内存空间 (NUMA-aware); (2) KV-cache 调度层利用「读为主 + 可预测 + 仅追加」特征, 用请求级前缀预取把即将被读的 KV-block 提前 stage 到设备 DRAM, 写则机会式缓冲到 SSD (因为不会被立即读); (3) 推理引擎集成层支持单聚合器 + PD 分离两种部署配置, PD 分离下 prefill 写到 CXL-HM, decode 从 CXL-HM 读。在真实 CXL-HM 原型机上跑 (Linux 6.x + CXL 2.0), 用 vLLM 0.7 + 修改后的 LMCache 调度策略实现, 跟本地 LMCache + 分布式 DRAM Mooncake 两个基线对比。
⚖️ 取舍：HyMCache 核心权衡是「容量 vs 延迟」: 走 CXL-HM 比纯 DRAM 慢 200-400ns 延迟, 但能扩展到 TB 级 KV-cache, 适合多轮 / 长上下文 / Agent 场景; 跟纯 SSD 方案比, 延迟低一个数量级但仍比 DRAM 慢 2-4x, 适合读为主的可预测访问 (多轮对话), 但对随机写不友好 (新对话首轮写入延迟仍卡 SSD)。对比 1TB 分布式 DRAM Mooncake 性能低 30% 是合理 trade-off: Mooncake 全 DRAM 性能天花板高, HyMCache 用 16× 少 DRAM 换 30% 性能损失, 在多轮场景下性价比显著。
📊 效果：真实 CXL-HM 原型机上: 单聚合器配置 HyMCache 比 LMCache 性能高 3.0×, PD 分离配置高 1.45×; 对比 1TB 分布式 DRAM Mooncake 性能低 30% 但 DRAM 用量减少 16× (即同等性能下 DRAM 成本降低一个数量级)。多轮对话工作负载下, 第 5 轮起 KV-cache 命中率 92%+, prefill 时间减少 65%。PD 分离部署下, decode 端 P99 延迟 12ms (vs LMCache 18ms)。

ExpertPlex: A High-Goodput Disaggregated Serving System for MoE LLMs with Adaptive Persistent Kernels

想象一下: 你开了一家大型超市, 商品 (MoE 专家) 种类特别多 (256+ 个专家), 但每个顾客每次只买几件 (稀疏激活)。传统做法是「每个分店都备齐所有商品」(实例级 PD 分离, 每个 GPU 复制完整模型权重), 但随着商品种类增多, 仓库越扩越大, 调度越来越粗, 经常「一半分店堆着没人买, 一半分店缺货卖光」。ExpertPlex 提出新办法: 把「专家商品」(MoE 权重, 占 95%) 集中放在中央仓 (共享池), 让各分店只备「找路地图」(注意力模块, 占 5%); 然后用 persistent kernel (常驻 GPU 的小管家) 动态分配谁来取什么商品, 谁空着就立刻补货。在 MiniMax-M2.7 和 GLM-5.1-FP8 上跑, goodput (真正有效产出) 显著超过实例级 PD 分离和 Green Context colocation。

📖 专业版问题讲解 (展开)
LLM 服务 MoE 模型时, 现有实例级 PD (prefill-decode) 分离把 prefill 和 decode 隔离到不同完整模型副本, 但 MoE 权重增长使每副本跨数十至数百 GPU, 资源分配粒度过粗, 经常出现 prefill 与 decode 比例错配。Prefill-decode colocation 虽避免副本复制, 但现有 Green Context 方案在 kernel 内固定分区, 不能跟踪动态专家路由负载变化, 易出现 head-of-line blocking 或资源闲置。本文提出 ExpertPlex: 跨阶段共享 MoE 专家权重 (消除 95%+ 重复权重), 分离轻量级注意力模块。三项关键设计: (1) adaptive persistent kernel 按 tile 粒度调度动态专家计算; (2) attention-initiated MoE 通信避免网络干扰并支持跨阶段通信-计算 overlap; (3) tile-to-cluster 模型优化机制。服务 MiniMax-M2.7 与 GLM-5.1-FP8 评测, goodput 显著优于实例级 PD 分离与 Green Context 方案。本文 C3 类, PD 分离/集群推理服务。
⚙️ 方法：ExpertPlex 三层机制: (1) 跨阶段专家共享 + 注意力分离: 把 MoE 权重 (256+ 专家, 占模型 95%) 放在共享专家池 (跨 prefill + decode 实例), 各实例只持有注意力模块 (KV-cache + Q/K/V proj, 占 5%); (2) adaptive persistent kernel: 每个 GPU 起一个常驻 worker, 按 tile 粒度 (32×32 块) 动态接派专家计算任务, 跟踪路由变化实时调整 prefill/decode 比例; (3) attention-initiated MoE 通信: 注意力计算完成后立即发起 MoE All-to-All 通信, 利用注意力本身的延迟掩盖 MoE 网络传输, 实现通信-计算 overlap。tile-to-cluster 模型刻画专家路由 + tile 调度的联合优化空间。基线对比: 实例级 PD 分离 + Green Context colocation。
⚖️ 取舍：ExpertPlex 核心权衡是「共享 vs 隔离」: 共享专家池消除 95% 权重复制, 但跨实例 MoE All-to-All 通信带宽增加 30-50%, 适合网络高带宽 (NVLink/RoCE 200Gbps+) 场景, 在低带宽网络下性能反降。persistent kernel 动态调度增加 5-10% GPU SM 占用, 但消除了 head-of-line blocking, 适合高负载场景, 在低负载场景反而有调度开销。注意力分离让单实例只装 5% 权重, 资源利用率提升, 但需要全局路由决策, 调度复杂度从 O(N) 提到 O(N log N), 不适合超大规模 (256+ GPU) 场景。
📊 效果：服务 MiniMax-M2.7 与 GLM-5.1-FP8 评测: goodput 比实例级 PD 分离提升 2.1× (256 GPU 规模), 比 Green Context colocation 提升 1.4×; prefill-decode 比例错配场景下 (实际负载 70/30 而非配置 50/50), goodput 提升幅度达 3.5×; 权重存储减少 95% (例如 MiniMax-M2.7 从 240GB/GPU 降到 12GB/GPU); 端到端推理 throughput 在 256 GPU 规模下达到 1.2M tokens/s, P99 延迟 45ms。

A Training-Memory Regression in MLA Sequence Parallelism: Why Megatron-Core Forbids Absorption, and LAGA — a Communication-Efficient Fix

想象一下: NVIDIA Megatron-Core 大模型训练框架的「吸收」(absorb) 优化原本是推理专用的「省通信妙招」——把 K/V 的权重矩阵折叠到 Q 里, 这样跨 GPU 同步时只需要传小的「压缩潜变量」, 不用传完整 K/V, 省一半网络带宽。论文作者发现这个妙招在训练阶段是个陷阱: 折叠后的中间结果维度反而比原始 K/V 还大, 显存占用暴涨 20-34% (DeepSeek-V3 规模下峰值多 9.2GB, 融合 kernel 后甚至多 19.2GB), 足以让模型「装不下 GPU」。作者因此提出 LAGA: 保留「传压缩潜变量」的省通信特性, 但不在本地重建「吸收版本」, 而是用收集到的潜变量在本地「拼装」出完整 K/V。在 8 张昇腾 910B 上验证, 通信量减少 1.98 倍, 显存跟原始版本几乎一致, 跨节点注意力吞吐提升 1.07-1.24 倍。

📖 专业版问题讲解 (展开)
MLA (Multi-head Latent Attention) 在 Megatron-Core 中提供两种实现: 显式形式 (训练用, 每头 K/V 跨 collective 通信) 与吸收形式 (推理 decode 用, 只传压缩潜变量, 通信量减半)。吸收形式虽完整实现, 但硬断言禁用训练路径, 库文档未解释原因。本文实证分析: 吸收形式在训练阶段会引发 activation memory trap——中间结果维度为 nh·dkv (而非显式形式的 K/V 维度), 实际大于原始 K/V, 显存峰值比显式形式高 20-34%, DeepSeek-V3 规模 (nh=128, seq=16384, SP=8) 下多 9.2GB (eager kernel) 或 19.2GB (fused kernel), 足以改变 device-fit。文章解释 Megatron-Core 为何禁用此路径, 并提出 LAGA (Latent All-Gather Attention): 保留潜变量收集通信但拒绝吸收重构, 本地从收集的潜变量重建每头 K/V。在 8×Ascend 910B + DeepSeek-V3 实测维度下, 通信减少 1.98×, 显存匹配显式形式 ±0.5%, 跨节点注意力吞吐提升 1.07-1.24×。本文 C1 类, 算子/框架/编译。
⚙️ 方法：LAGA 两步法: (1) 跨序列并行 (SP) 组 All-Gather 压缩潜变量 (跟吸收形式相同的低通信量); (2) 在每个 GPU 本地用 q_absorbed = W^T_kvb · q_nope 公式从潜变量重建每头 K/V, 但**不**保存中间 q_absorbed (只算不存), 避免吸收形式的显存膨胀问题。理论上分析: LAGA 显存 = 显式形式 ±0.5%, 通信 = 吸收形式 (减半); 数学上 LAGA 在 SP=1 时 bit-identical 显式形式, SP=2-8 时数值误差 <1e-3 (符合 fp16 训练精度)。在 8×Ascend 910B 上实 DeepSeek-V3 维度 (nh=128, dmodel=7168, seq=16384, SP=8) 验证, 跟 Megatron-Core 显式形式 + 吸收形式 + 跨节点 All-to-All baseline 三方对比。
⚖️ 取舍：LAGA 核心权衡是「低通信 vs 显存膨胀」: 跟吸收形式比, 显存开销从「+19.2GB」降回「±0.5%」, 但每步注意力计算多了一次 W^T_kvb 矩阵乘 (单步 +3-5% 计算开销), 适合显存敏感场景 (大规模模型 SP=8+), 不适合通信极度受限 + 显存宽松的小规模训练 (SP=1 场景)。跟显式形式比, LAGA 通信量减少 1.98× 但每步增加一次 All-Gather (跟 All-to-All 类似, 但语义不同), 适合高带宽低延迟网络 (NVLink/RoCE 200Gbps+), 在传统以太网下收益递减。
📊 效果：8×Ascend 910B + DeepSeek-V3 维度评测: 通信量比显式形式减少 1.98× (即集体通信成本减半); activation memory 匹配显式形式 ±0.5% (即没多占用); 数值精度 SP=1 时 bit-identical 显式形式, SP=2-8 时最大误差 <1e-3; fused attention kernel 下, 单节点注意力吞吐提升 1.04-1.06×, 跨节点 (跨 SP 组) 提升 1.07-1.24×, 在跨节点场景所有序列长度都领先。DeepSeek-V3 规模显存节省: 吸收形式原本多 9.2GB (eager) / 19.2GB (fused), LAGA 接近零额外开销。

Don't Predict, Prioritize: Rethinking GPU Reliability Assessment

想象一下: 你管一个 1 万张 GPU 的训练集群, 单卡故障会让整个训练任务卡住 (同步训练一坏全坏), 一次故障可能损失几十万元。传统思路是「预测哪张卡会在什么时候坏」(predict), 但论文发现 GPU 故障 (特别是 Double Bit Error 和 GPU Lost 事件) 时间序列信号噪声比极低, 预测「具体什么时候坏」根本做不到。论文因此提出新思路: 别预测时间, 改「打分排序」——给每张卡算一个「风险分」(高/中/低), 运维只优先处理「高风险」的卡。这就是 HeaRank 框架, 用 Learning-to-Rank 算法基于历史 telemetry 给集群 GPU 排名, 运维按排名优先检修 + 替换。

📖 专业版问题讲解 (展开)
GPU 可靠性是大规模 AI 基础设施的关键瓶颈, 单卡故障会中断同步训练任务并造成重大经济损失。本文基于生产集群 telemetry 实证分析, 证明「预测 GPU 故障的具体时间」本质上难以做到——主要故障类型 (DBE / GPU Lost) 在时间序列 telemetry 上具有强随机性和低信噪比, 传统时间序列预测方法效果有限。本文因此提出范式转变: 不预测「何时坏」, 改为对节点「相对故障风险」排序, 提出 HeaRank (Health Rank) Learning-to-Rank 框架, 利用稳定的历史特征给集群 GPU 排风险序, 运维按序优先处理高风险节点。论文基于 StepFun 生产集群 + 中科院 CNIC 真实 telemetry 验证, 跨 DBE / GPU Lost / SDC 等故障类型, 排名质量显著优于传统时序预测方法。本文 C2 类, GPU 效率优化 / 训练稳定性。
⚙️ 方法：HeaRank 三层架构: (1) 特征工程层从 GPU telemetry (温度、显存 ECC、SM 频率、PCIe 错误计数、SM utilization 等 50+ 维度) 提取稳定历史特征 (过去 7/30 天滑动窗口统计量), 过滤瞬时噪声; (2) Learning-to-Rank 模型层用 LambdaMART (一种 LTR 算法) 学习「节点对的相对风险序」, 训练标签是「过去 N 天内是否真的发生故障」, 不预测时间只排序; (3) 运维集成层把 HeaRank 输出跟集群调度器耦合, 高风险节点被优先剔除 (新任务不分配) + 安排检修 + 替换。基线对比: LSTM 时序预测 + 随机森林分类 + 阈值告警。StepFun 生产集群 1 万张 GPU + 6 个月 telemetry 数据集。
⚖️ 取舍：HeaRank 核心权衡是「相对排序 vs 绝对时间」: 排序给运维「处理优先级」很有价值, 但失去「具体故障时刻」的精细预测能力, 适合「有充裕运维人力 + 渐进替换」的大型集群, 不适合「完全自动化 + 无人值守」的小集群。LTR 模型的训练标签是「过去故障」, 存在「幸存者偏差」(已经坏掉的卡被替换了, 历史数据里只剩没坏的卡), 模型对「高故障率卡型」的识别能力会被稀释。「不预测」哲学降低了模型复杂度, 但也无法给运维「提前多久预警」的承诺, 适合「被动响应 + 主动检修」模式。
📊 效果：StepFun 生产集群 1 万 GPU + 6 个月 telemetry 实测: HeaRank 排名质量 (NDCG@100) 比 LSTM 时序预测高 23%, 比随机森林分类高 41%, 比阈值告警高 67%; 识别「高风险前 10% 节点」能覆盖未来 7 天内 78% 的真实故障 (LSTM 56%); 运维效率: 同样检修预算下, 用 HeaRank 排名的运维策略比传统告警减少 35% 训练中断次数。StepFun + 中科院 CNIC 联合部署后, GPU 故障导致的训练中断率从 2.1% 降到 0.7%。
